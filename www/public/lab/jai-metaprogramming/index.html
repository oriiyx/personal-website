jai metaprogramming

compile-time code execution · #run · #insert · why it matters for game engines

What problem does metaprogramming solve?

You're writing a game engine. You have a struct Entity with 30 fields. Now you need: a serializer that writes each field to disk, a debug inspector that prints each field's name and value, a network replicator that diffs fields between frames, a script binding layer that exposes each field to Lua. All of these need to walk the struct's members and do something per-field.

In C, you have roughly three options: write all four by hand (tedious, error-prone, falls out of sync when you add a field), use preprocessor macros to define fields in an X-macro table (ugly, limited, no type safety), or use an external code generator that parses your headers and emits C (now you have a build step, a parser, and two languages to maintain).

The core tension: the information about your struct's layout already exists inside the compiler. You just can't access it. The compiler knows every field name, every type, every offset. But that knowledge evaporates after compilation. Your runtime code starts from zero.

What Jai does differently

Jai's answer is radical in its simplicity: let you run arbitrary code at compile time, using the same language, with access to type information from the compiler. Not a special macro sublanguage. Not a template system. Not a separate code generator. The same Jai you write at runtime — loops, allocations, string operations, file I/O — all available during compilation.

This single decision eliminates the entire category of problems above. Your serializer, inspector, replicator, and script bindings can all be generated by a compile-time function that iterates over a struct's members and emits the appropriate code. Add a field to the struct and everything updates automatically on the next compile.

Blow's framing: Most languages treat "compile time" and "run time" as fundamentally different execution environments with different capabilities. Jai treats them as the same environment, just at different points in the build process. The compiler is a Jai interpreter.

Why game engines specifically?

Game engines are uniquely punished by the lack of metaprogramming. They have large struct hierarchies (components, entities, resources), performance-critical data layout requirements (SOA transforms, cache-line alignment), extensive tooling needs (editors, inspectors, serialization), and scripting integration. Every one of these involves the same pattern: "for each member of this type, do X." C++ attempts to address this with templates, but templates operate on types — they can't iterate over struct members, generate string names, or make decisions based on field attributes.

This is why every major C++ engine (Unreal, Unity's native layer) ends up building a custom code generator or reflection system. Jai's position is that this should be a language feature, not an external tool.

Compile-time execution as an interpreter

When the Jai compiler encounters a #run directive, it doesn't shell out to a subprocess or invoke a template engine. It interprets the Jai code directly, inside the compiler process, using a bytecode interpreter. The code being run has access to the same type system, the same standard library functions, and can even do file I/O and memory allocation.

This is the key architectural decision. The compiler contains a full interpreter for the language it's compiling. The implications:

Same language

No macro syntax to learn. No template meta-language. No build-script DSL. If you can write it as a runtime function, you can run it at compile time.

Full capabilities

Compile-time code can allocate memory, read files, do string manipulation, call into modules. It's not a restricted subset — it's the full language.

Type introspection

Compile-time code can query the compiler for type info: struct members, their types, sizes, offsets, names. This is what makes struct-walking metaprogramming possible.

Code injection

Compile-time code can generate new source code (as strings) and insert it back into the program via #insert. The compiler then compiles the generated code as if you'd written it by hand.

#run executes after types are resolved but before final code generation. This means it can inspect types and generate code that will be compiled in subsequent passes. The compiler may iterate — #inserted code can itself contain #run directives.
Not a preprocessor. C's #define operates on tokens before the compiler even sees them — it's text substitution with no type awareness. Jai's #run operates inside the compiler, after parsing and type resolution. It knows what your types look like. This is the fundamental difference.

Two primitives, one model

The entire metaprogramming system rests on just two directives:

Directive What it does Analogy
#run Execute this expression/block at compile time. If it returns a value, that value is baked into the compiled program as a constant. Like constexpr in C++, but without the restrictions — you can call any function, allocate, do I/O.
#insert Take a string (usually produced by #run) and compile it as source code at this location. Like C's #include but for dynamically generated code, not files.

#run lets you compute. #insert lets you inject the results of that computation back into the program as compilable code. Together, they form a complete metaprogramming system.

Basic #run: compile-time constants

The simplest use of #run is computing a value at compile time and baking it into the binary. No runtime cost — the compiler evaluates it and embeds the result.

// A lookup table computed at compile time SINE_TABLE :: #run generate_sine_table(); generate_sine_table :: () -> [256] float { table: [256] float; for i: 0..255 { table[i] = sin(cast(float) i / 256.0 * TAU); } return table; }

At compile time, the compiler calls generate_sine_table(), runs the loop, computes all 256 sine values, and stores the resulting array directly in the binary's data section. At runtime, SINE_TABLE is just a static array — no initialization code runs.

Compare with C: In C, you'd either hard-code the table (error-prone, unreadable), write a separate program that generates a .c file containing the array (extra build step), or compute it at startup in main() (runtime cost, initialization ordering issues). Jai makes it one declaration.

#run with side effects

#run isn't limited to pure computation. It can do I/O, which means it can read files, write logs, even generate assets during compilation:

// Read a config file at compile time and bake it in DEFAULT_CONFIG :: #run read_entire_file("default_config.json"); // Print a message during compilation #run print("Compiling player module... entity count = %\n", MAX_ENTITIES);

The first example reads a file from disk during compilation and stores its contents as a compile-time string constant. The second just prints to the compiler's stdout — useful for build-time diagnostics.

#run at statement level

#run doesn't have to produce a value. You can use it at the top level to execute arbitrary setup code during compilation — registering types, running tests, validating constraints:

// Run tests at compile time — if they fail, compilation fails #run { assert(size_of(Entity) <= 64, "Entity exceeds cache line!"); assert(MAX_ENTITIES % 16 == 0, "MAX_ENTITIES must be SIMD-aligned"); }

If the assertion fails, the compiler reports an error at the #run site. Your invalid assumptions are caught before a binary is ever produced. This is compile-time validation with zero runtime overhead.

What #run can't do

Despite being "the full language," there are practical limits. #run code executes in the compiler's interpreter, which is slower than native execution — it's fine for generating tables and code strings, but you wouldn't use it to train a neural network. It also can't call into external C libraries via FFI during compilation (the linker hasn't run yet). And any side effects (file writes, network calls) happen at compile time, not runtime — which is the point, but can surprise you if you're not thinking about it.

From values to code

#run alone can produce compile-time constants — numbers, arrays, strings. But the real power comes when you combine it with #insert to generate code. The pattern: a #run block builds a string of valid Jai source code, and #insert tells the compiler to compile that string as if it were part of the source file.

// Generate a print_fields function for any struct print_fields :: (val: $T) { info := type_info(T); #insert #run () -> string { builder: String_Builder; ti := type_info(T); for member: ti.members { print_to_builder(&builder, "print(\"%: %\\n\", val.%);\n", member.name, "%", member.name); } return builder_to_string(&builder); }(); }

Let's trace what happens if you call print_fields(my_entity) where Entity has fields x, y, and health:

The lambda iterates over Entity's type info and builds a string:

The function body is just three print calls. No reflection, no hash maps, no runtime overhead.

The key insight: it's just string concatenation

There's no AST manipulation API, no quote/unquote operators, no hygiene system. You build a string of valid source code using print_to_builder or plain string concatenation, and the compiler parses it. This is intentionally low-ceremony — if you can printf it, you can generate it.

The tradeoff is that your generated code is opaque to the IDE and harder to debug than hand-written code. If you build a malformed string, you get a compiler error pointing at the #insert site, not at the specific character in your generated string. In practice, this means you want to keep generated code simple and test it by printing the string during development.

Debugging tip: During development, add a #run print(generated_string) before the #insert so you can see the exact code being generated. Remove it once it's working.

#insert in declarations

#insert isn't limited to function bodies. It can appear anywhere the compiler expects declarations — inside struct definitions, at module scope, wherever you need to generate members or top-level definitions:

// Generate struct members from an external schema Network_Packet :: struct { header: Packet_Header; #insert #run generate_payload_fields("packet_schema.def"); }

The #run reads a schema file at compile time, generates field declarations as a string (like "x: float;\ny: float;\nz: float;\n"), and #insert splices them into the struct. The resulting struct is indistinguishable from one written by hand — same layout, same performance, same type safety.

The landscape

Every systems language has some answer to "how do I do things at compile time." The approaches differ radically in power, safety, and ergonomics. Here's how Jai fits in.

C: preprocessor macros

C — X-macro pattern for struct iteration
#define ENTITY_FIELDS \ X(float, x) \ X(float, y) \ X(int, health) typedef struct { #define X(type, name) type name; ENTITY_FIELDS #undef X } Entity; void print_entity(Entity* e) { #define X(type, name) printf(#name ": ...\n"); ENTITY_FIELDS #undef X }

This works, but: the macro operates on tokens, not types — it can't inspect a struct that was defined normally. You have to define your struct through the macro, which couples your data definition to your metaprogramming trick. No type safety inside the macro body. No conditional logic beyond #ifdef. Error messages are incomprehensible. And the #define X / #undef X dance is brittle and ugly.

Jai's advantage: You define your struct normally and introspect it at compile time. The two concerns are decoupled.

C++: templates and constexpr

C++ templates are Turing-complete, but they operate on types and values, not on struct members. You can write template<typename T> to be generic over a type, but you cannot write "for each field in T, emit a print statement." There's no for_each_member in the language.

constexpr (C++11/14/17/20) lets you evaluate functions at compile time, but with restrictions that have only gradually loosened across standards — no allocations (until C++20), no I/O, no virtual calls, limited container support. And even constexpr can't generate new declarations or struct members.

Jai's advantage: No restrictions on compile-time code. Full struct introspection. Can generate arbitrary declarations.

Zig: comptime

Zig — compile-time struct iteration
fn printFields(value: anytype) void { const info = @typeInfo(@TypeOf(value)).@"struct"; inline for (info.fields) |field| { std.debug.print("{s}: {}\n", .{ field.name, @field(value, field.name) }); } }

Zig's comptime is the closest relative to Jai's model. It also lets you run code at compile time with access to type information, and inline for over struct fields. The key differences:

Aspect Jai Zig
Mechanism String-based code generation (#insert) Semantic comptime evaluation + inline for
Code generation Generate arbitrary source strings No string-based codegen; uses comptime type construction instead
Side effects File I/O, allocation, printing during compile @compileLog for output; no general I/O
Struct introspection Via type_info() Via @typeInfo()
Safety String-based — errors caught when #inserted code is compiled Semantic — errors caught immediately with clear messages

Zig's approach is more principled (no string-based code generation means fewer "magic string" bugs). Jai's approach is more flexible (you can generate literally anything that's valid source code). The tradeoff is safety vs. power.

Rust: procedural macros

Rust's proc macros operate on token streams — they receive the AST of their input and produce a new token stream. This is more structured than string concatenation but requires a separate crate, compiles as a separate binary, and involves the syn/quote ecosystem that has a steep learning curve.

Crucially, Rust proc macros can't access type information. A derive macro sees the struct's tokens (field names as text, type names as text), but can't query whether a type implements a trait, what its size is, or what its members are. This is a fundamental limitation — proc macros operate at the syntactic level, not the semantic level.

Jai's advantage: Operates at the semantic level (types are resolved). Same language, no separate crate. Rust's advantage: Macro hygiene prevents accidental name collisions. Token-stream API is more structured than string concatenation.

Pattern 1: Compile-time lookup tables

The simplest and most common use. Any table that can be computed from a formula or a data file should be a #run constant — zero runtime initialization, no "did we call init() yet?" bugs.

CRC32_TABLE :: #run () -> [256] u32 { table: [256] u32; for i: 0..255 { crc := cast(u32) i; for j: 0..7 { if crc & 1 { crc = (crc >> 1) ^ 0xEDB88320; } else { crc = crc >> 1; } } table[i] = crc; } return table; }();

This pattern replaces the common C idiom of a static array initialized in a function that you have to remember to call early in main().

Pattern 2: Auto-serialization via type info

This is the flagship use case. Instead of writing a serialize function per struct, you write one generic function that uses type info to handle any struct:

serialize :: (buf: *Buffer, val: $T) { info := type_info(T); for member: info.members { // At compile time, this loop is unrolled. // Each iteration knows the field's type and offset. write_field(buf, member.name, val + member.offset, member.type); } }

Because Jai bakes type information into the binary (names, sizes, offsets, member lists), this kind of reflection is available at both compile time and runtime. The compile-time version is faster (the compiler can unroll and inline), but the runtime version works too — useful for editor tools that need to inspect types dynamically.

Pattern 3: Compile-time asserts as documentation

Use #run to encode your assumptions as checked constraints. These serve double duty as documentation and validation:

// Document and enforce layout assumptions #run assert(size_of(Vec3) == 12); #run assert(offset_of(Entity, "transform") == 0, "Transform must be first for SIMD alignment"); #run assert(MAX_ENTITIES * size_of(Entity) <= L1_CACHE_SIZE * 4, "Entity array should fit in ~4x L1 for streaming access");

When someone changes Entity and breaks a layout assumption, the build fails with a message that explains why the constraint exists. This is vastly better than a runtime crash three weeks later.

Pattern 4: Build-time asset processing

Since #run can read files, you can process assets during compilation — parsing config files, validating data, even converting formats:

// Parse a level file at compile time, embed as typed data LEVEL_1 :: #run parse_level_file("data/levels/level_01.ldtk"); // The result is a typed struct, not a raw string. // Invalid level files cause compile errors, not runtime crashes.

This moves errors left — a malformed level file is caught at compile time, not when a player reaches that level. And the parsed data is available as a typed constant, so the runtime code doesn't need a parser at all.

Pattern 5: SOA transforms

Jai has built-in SOA (Structure of Arrays) support, but the mechanism underneath is metaprogramming. Given a struct with N fields, an SOA container stores N separate arrays instead of one array of structs. This transform is mechanical — it's a perfect fit for compile-time code generation.

AOS vs SOA — what the metaprogramming generates
AOS (what you write)
Entity :: struct { x: float; y: float; health: int; } entities: [1024] Entity; // 1024 × {x, y, health} interleaved
SOA (what the compiler generates)
// Conceptually, the metaprogramming transforms it to: entities_x: [1024] float; // all x values contiguous entities_y: [1024] float; // all y values contiguous entities_health: [1024] int; // all health values contiguous
SOA layout means a loop that only touches x and y (e.g., position update) never loads health into cache. This is the data-oriented design principle — organize data by access pattern, not by object identity.

In C or C++, you'd implement this transform manually or with a code generator. In Jai, the compiler can do it for you because it has the struct's type information and the ability to generate new layout code at compile time.

When not to use #run

Not everything needs metaprogramming. Some guidelines:

Use #run when Don't use #run when
The computation is deterministic and its inputs are known at compile time The inputs depend on runtime state (user input, network data)
You're generating repetitive code that mirrors a data structure The code is simple enough to write by hand and unlikely to change
You want compile-time validation of invariants The generated code is so complex that debugging the generator is harder than writing the code directly
You need the same struct-walking logic in 3+ places You need it in exactly one place
Blow's rule of thumb: If you find yourself copy-pasting code that differs only in field names, that's a signal for metaprogramming. If you're writing a function that's unique and won't be repeated, just write the function.
