base64url from first principles

RFC 4648 §5 · url-safe alphabet · no padding · for developers who want the full picture

Why does base64 exist?

Many transport protocols — email (SMTP), HTTP headers, URLs, JSON — were designed for text. They use certain bytes as control characters: newlines terminate headers, NUL terminates C strings, + means "space" in a URL query, / separates path segments, " delimits JSON strings.

Now imagine you need to send a cryptographic signature, a compressed blob, or a JPEG through one of these channels. Raw binary will inevitably contain bytes that collide with the protocol's special characters. You'd get truncated data, misparses, or silent corruption.

Base64 solves this by encoding arbitrary binary data using only 64 characters that are safe in virtually all text contexts. The tradeoff is size: 3 input bytes become 4 output characters — a ~33% expansion. That's the cost of universality.

Why "url-safe"?

Standard base64 (RFC 4648 §4) uses + and / as its 62nd and 63rd characters. Both are problematic in URLs: + is interpreted as a space in query strings, and / is a path separator. So you'd have to percent-encode them (%2B, %2F), which defeats the purpose of a compact encoding.

Base64url (RFC 4648 §5) replaces them with - and _, which are unreserved in URLs. It also typically omits the = padding characters, since the decoder can infer the original length from the encoded string's length.

This makes base64url the standard encoding for JWTs, JWKs, and most modern web token formats — which is exactly the context in our credential issuance system.

The core contract

Encoding ratio
3 bytes → 24 bits → 4 × 6-bit values → 4 ASCII characters
Every group of 3 input bytes (24 bits) maps to exactly 4 output characters.
The key insight: 24 is the least common multiple of 8 and 6, so 24 bits is where the byte boundary and the base64-character boundary align cleanly.

Why 6 bits = 64 characters

A single bit gives you 2 values (0 or 1). Each additional bit doubles the range. With 6 bits you get 26 = 64 possible values (0–63). That's enough to index into an alphabet of 64 printable ASCII characters, and it divides evenly into 24 — which is the key property that makes the 3-byte → 4-character mapping work.

Why not 7 bits (128 chars)? You'd need 128 printable characters that are safe everywhere — there aren't enough. Why not 5 bits (32 chars)? That works (it's base32), but the expansion ratio is worse: 5 bytes → 8 characters (60% overhead vs. 33%).

6 bits is the sweet spot: large enough alphabet to keep expansion low, small enough to stay within safe printable ASCII.

The 64-character table

The position in this table is the 6-bit value. Character at index 0 is A, at index 25 is Z, at index 62 is -, at index 63 is _.

In code, this is just a string literal — the index operator gives you the character:

static const char ENC_TABLE[] = "ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789-_"; // ENC_TABLE[0] → 'A' // ENC_TABLE[25] → 'Z' // ENC_TABLE[52] → '0' // ENC_TABLE[63] → '_'

Standard vs. url-safe

Index Standard (RFC 4648 §4) URL-safe (RFC 4648 §5) Why the change
62 + - + means space in URL query strings
63 / _ / is a URL path separator
padding = omitted = is the assignment operator in query params

Characters 0–61 are identical in both variants.

The operators, explained

Operator Name What it does here
<< n Left shift Moves bits n positions to the left, filling vacated positions with zeros. Used to place each byte into its lane within the 24-bit integer.
>> n Right shift Moves bits n positions to the right. Used to bring a 6-bit group down to the bottom so it can be masked.
| Bitwise OR "If either bit is 1, result is 1." Since our shifted values occupy different lanes (no overlapping 1-bits), OR merges them into one value. Think of it as "paste into these bit positions."
& Bitwise AND "Keep only bits where both sides are 1." & 0x3F keeps the lowest 6 bits and zeros everything above — a mask.

Why 0x3F?

0x3F in binary is 00111111 — six 1-bits. In decimal: 26 − 1 = 63. When you AND any value with 0x3F, you keep only the lowest 6 bits and discard everything above. This is the standard "extract N bits" pattern in C: shift the bits you want to the bottom, then AND with (2N − 1).

When input length isn't a multiple of 3

The 3→4 encoding only works cleanly when you have complete triples. But input can be any length. After processing all full triples, you're left with 0, 1, or 2 remaining bytes. These are the remainder (or "tail") cases.

Remainder = 1 byte (8 bits → 2 characters)

One byte gives us 8 bits. We need to represent them as 6-bit base64 characters. One character encodes 6 bits, two characters encode 12. So 2 characters are needed (1 would only give us 6 bits — not enough for 8).

The trick: we still shift the byte to the top of a 24-bit integer (<< 16), then extract from the top as usual — but only output 2 characters instead of 4. The bottom 4 bits of the second character are trailing zeros that the decoder discards.

No padding in base64url. Standard base64 appends = characters to fill out the final group to 4 chars. Base64url skips this — the decoder calculates the tail length from encoded_len % 4. A tail of 2 chars → 1 byte, 3 chars → 2 bytes, 1 char → invalid (6 bits can't represent a whole byte).

Output length formula

Without padding: output_len = ceil(input_len × 4 / 3)

Since C doesn't have a ceiling division operator, we use the standard integer trick: ceil(a / b) = (a + b − 1) / b. So:

(input_len * 4 + 2) / 3 // +2 because b-1 = 3-1 = 2

Verify: 3 bytes → (12+2)/3 = 4 ✓ · 1 byte → (4+2)/3 = 2 ✓ · 2 bytes → (8+2)/3 = 3 ✓ · 5 bytes → (20+2)/3 = 7 ✓

Decoding is the exact reverse

Encoding takes 3 bytes, combines them, and slices into 4 characters. Decoding takes 4 characters, combines their 6-bit values, and slices into 3 bytes. The shift amounts are mirrored:

Pack (b0 << 16) | (b1 << 8) | b2 (v0 << 18) | (v1 << 12) | (v2 << 6) | v3
Extract (n >> 18) & 0x3F etc. (n >> 16) & 0xFF etc.
Group size 3 in, 4 out 4 in, 3 out
Mask 0x3F (6 bits) 0xFF (8 bits, implicit via uint8_t cast)

Decode tail handling

Trailing chars Bits available Bytes produced Rationale
0 0 0 Clean boundary, nothing left.
1 6 INVALID 6 bits < 8 bits. Can't produce a whole byte. This can't have come from a valid encode.
2 12 1 12 ≥ 8. Top 8 bits are real, bottom 4 are trailing zeros from encode.
3 18 2 18 ≥ 16. Top 16 bits are real, bottom 2 are trailing zeros.

Two-phase loop structure

Both encode and decode process in two phases: first the "full" groups (triples for encode, quads for decode) in a tight loop, then handle the remainder separately. This avoids any branching inside the hot path.

// Phase 1: full groups (no branching inside) for (size_t i = 0; i < full_triples; i++) { // pack → extract → store (always 3 in, 4 out) } // Phase 2: handle 0, 1, or 2 remaining bytes if (remainder == 1) { /* 1 byte → 2 chars */ } else if (remainder == 2) { /* 2 bytes → 3 chars */ }

The DEC_TABLE ambiguity problem

The decode table is a 256-entry array where DEC_TABLE['B'] returns 1, DEC_TABLE['z'] returns 51, and so on. The problem: DEC_TABLE['A'] returns 0, but so does DEC_TABLE['!'] or DEC_TABLE['@'] — because all uninitialized entries are zero by default in C.

You cannot distinguish "this is the letter A, which legitimately maps to 0" from "this is an invalid character that happens to be 0." So the code validates all characters before decoding, using a simple range-check function:

static int is_valid_b64char(const char c) { if (c >= 'A' && c <= 'Z') return 1; if (c >= 'a' && c <= 'z') return 1; if (c >= '0' && c <= '9') return 1; if (c == '-' || c == '_') return 1; return 0; }
An alternative approach used by some libraries: fill invalid entries with a sentinel like 0xFF (255) instead of leaving them at 0. Then the table itself can detect invalid characters and you don't need a separate validation pass. Our implementation chose the explicit validation approach — simpler to read, same result.

The (unsigned char) cast on decode

In C, char may be signed (typically −128 to 127) on some platforms. If you index an array with a negative value, you're reading outside the array bounds — undefined behavior. The cast (unsigned char) in[si] ensures the index is always 0–255:

DEC_TABLE[(unsigned char) in[si]] // always 0–255, safe DEC_TABLE[in[si]] // could be negative if char is signed!

The (uint32_t) cast on encode

in[si] is a uint8_t — an 8-bit value. When you write in[si] << 16, C promotes it to int (typically 32-bit signed). For values ≤ 127 this works fine, but for larger byte values the result might not be what you expect with signed arithmetic. The explicit (uint32_t) cast makes the promotion to unsigned 32-bit explicit and safe for all byte values 0–255.
