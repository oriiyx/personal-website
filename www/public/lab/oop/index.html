why OOP failed — and what actually works

inheritance is not a feature · encapsulation is not a goal · your objects are not "real things"

The promise OOP sold you

Every introductory OOP course teaches the same thing: objects model real-world entities. A Dog extends Animal. A Car extends Vehicle. You organize your code the way you organize reality — as a taxonomy of things with shared properties. Inheritance gives you free code reuse. Encapsulation gives you safety. Polymorphism gives you flexibility.

This is seductive, intuitive, and almost entirely wrong. Real-world domains don't form clean trees. The moment you try to model anything non-trivial, the hierarchy fights you. You spend more time refactoring your taxonomy than solving the actual problem.

As Casey Muratori put it: OOP was supposed to make code easier to reason about by mapping it to the real world. Instead it created an artificial world — the class hierarchy — that you now have to maintain in addition to the real problem.

The diamond problem isn't an edge case — it's the normal case

Consider a game engine. You start with a reasonable-looking hierarchy:

The "clean" hierarchy (day 1)
Entity
├── Creature ├── Vehicle ├── Item
├── Fish ├── Bird ├── Boat ├── Car ├── Weapon
Everything has exactly one parent. The world makes sense. Ship it.

Then requirements arrive. A FlyingFish needs to swim (Creature→Fish) and fly (Creature→Bird). An AmphibiousVehicle drives on land (Vehicle→Car) and floats on water (Vehicle→Boat). A MountedTurret is both a Vehicle and a Weapon. A DragonMount is a Creature, a Vehicle, and it has an attack — so is it also a Weapon?

The hierarchy after real requirements (day 30)
Entity
├── Creature ├── Vehicle ├── Item
├── FlyingFish ??? ├── AmphibiousVehicle ??? ├── DragonMount ???
??? = needs multiple parents. Single inheritance can't express this. Multiple inheritance creates the diamond problem. Neither works.

The OOP response to this is always the same: "you designed the hierarchy wrong." But that's the point — there is no correct hierarchy. The real world doesn't have a single canonical taxonomy. A flying fish is simultaneously a fish (by biology), a flying thing (by locomotion), and a prey animal (by ecology). Which axis you choose as "the" inheritance axis is arbitrary, and every choice makes the other axes painful.

is-a vs. has-a: the question OOP can't answer

Is a Stack a List? If you inherit from List, you get push and pop for free — but you also expose insertAt(), removeAt(), and sort(), which violate stack semantics. So a Stack is-not-a List. But it has-a List internally.

Is an Employee a Person? Seems obvious: yes. But a Person can become an Employee, then become a Contractor, then go back to being just a Person. Inheritance is static; the real world is not. You can't change an object's class at runtime (in most languages). So now you need the Strategy pattern, or the State pattern, or a Role object — all of which are workarounds for inheritance being the wrong abstraction.

Brian Will's core observation: OOP forces a taxonomic question ("what is this thing?") that has no stable answer. The useful question is "what does this thing need to do right now?" — which is a question about capabilities, not identity. Capabilities compose; identities don't.

The pattern explosion

The Gang of Four design patterns book contains 23 patterns. Most of them exist to work around limitations of class-based OOP: Visitor compensates for the inability to add operations to a closed class hierarchy. Strategy compensates for the inability to swap behavior without subclassing. Observer compensates for the tight coupling that encapsulation creates. Abstract Factory compensates for the fact that new ClassName() is a hard dependency.

In languages without class-based OOP — where you have first-class functions, interfaces, and composition — most of these patterns dissolve into one-liners. A "Strategy" is just a function argument. A "Factory" is just a function that returns a value. An "Observer" is just a callback.

OOP Pattern What it compensates for Without OOP
Strategy Can't pass behavior as a value Function argument
Observer Can't subscribe without coupling Callback / channel
Factory new is a hard dependency Constructor function
Visitor Can't add methods to existing types Type switch / pattern match
Decorator Inheritance is the only extension axis Function wrapping / middleware
Command Actions must be objects to be stored Closures
Template Method Can't parameterize a step in an algorithm Higher-order function

Design patterns are not a sign of a language's power. They're a sign of its limitations. As Peter Norvig showed: 16 of the 23 GoF patterns are invisible or trivial in languages with first-class functions.

Encapsulation hides the wrong thing

The sales pitch: encapsulation protects you from implementation details. The reality: it hides the data you need to reason about while exposing the behavior that creates coupling.

When you debug a system, you're almost always asking: "what is the state right now?" Encapsulation makes this harder. The data is scattered across private fields of dozens of objects, accessible only through getter chains. You can't just inspect a data structure — you have to trace method calls through layers of indirection to reconstruct the state in your head.

Meanwhile, the public methods — the "encapsulated" interface — are the primary coupling surface. Every consumer depends on those method signatures. Change a public method, and every caller breaks. Encapsulation didn't isolate anything; it just added a layer of indirection between you and the data you need.

Encapsulated OOP — where's the state?
class OrderService { private repo: OrderRepository; private validator: OrderValidator; private pricing: PricingEngine; private notifier: NotificationService; process(id: string) { const order = this.repo.find(id); this.validator.validate(order); this.pricing.apply(order); this.notifier.send(order); // What did pricing do to order? // Did validator mutate it? // What state is notifier in? // Nobody knows. } }
Procedural — state is visible
func processOrder(db *DB, id string) error { order, err := db.FindOrder(id) if err != nil { return err } if err := validateOrder(order); err != nil { return err } order.Total = calculatePrice(order.Items) // ^ I can see exactly what changed return sendNotification(order.Email, order.Total) // ^ pure function of its arguments }

In the Go version, you can see every piece of state, every mutation, and every data dependency by reading top to bottom. In the OOP version, you need to open 4 classes to understand what process() does.

Every polymorphic object carries a hidden pointer (vptr) to its class's vtable. Every virtual call is: load vptr → load function pointer → indirect call. That's 2 dependent pointer chases before any work happens.

This has three concrete costs:

The hidden cost: heap allocation per object

OOP languages allocate every object on the heap (Java, C#, Python, Ruby, JavaScript). Even a simple Point(x, y) with two integers triggers a heap allocation, a GC header, and a vtable pointer. In Java, a Point object uses ~32 bytes of memory for 8 bytes of data. In Go or C, two integers in a struct: 8 bytes, stack-allocated, zero overhead.

OOP inverts the correct mental model

Software is fundamentally about data transformations: data comes in, you transform it, data goes out. A web server reads bytes from a socket, parses them into a request struct, routes to a handler, builds a response struct, serializes it back to bytes. That's it. Every program is a pipeline of data transformations.

OOP tells you to think about this differently: instead of "what transformations happen to the data," you ask "what objects exist and what messages do they send each other?" This is backwards. You're now designing a social network of objects instead of a data pipeline. The objects become the point; the data becomes an afterthought — something that's hidden inside objects and only accessible through method calls.

Data flows left to right. Functions transform it. No hidden state. Each step is testable in isolation.

In the data-flow model, the data structures are the design. Request and Response are plain structs. The functions between them are stateless transformations. You can test each one independently by constructing the input struct and asserting on the output struct. No mocks, no dependency injection, no service locators.

ORMs: the worst manifestation

Object-Relational Mappers are the purest example of abstraction inversion. A relational database is already a data-transformation engine — it's set-theoretic, declarative, and operates on flat rows. An ORM wraps this in an object graph, hydrating full objects with all their fields when you need two columns, lazy-loading related objects behind property accessors that silently fire N+1 queries, and mapping a well-designed relational schema into an awkward object hierarchy that fights the database's actual strengths.

ORM — hydrate everything, query hidden
// C# Entity Framework var users = context.Users .Include(u => u.Orders) .ThenInclude(o => o.Items) .Where(u => u.Active) .ToList(); // What SQL did this generate? // How many joins? // Did it select all columns? // What if Orders has 10,000 rows? // You don't know. The ORM "handles it."
Plain SQL — you see exactly what runs
// Go + sqlc or plain database/sql rows, err := db.Query(` SELECT u.id, u.name, COUNT(o.id) as order_count FROM users u LEFT JOIN orders o ON o.user_id = u.id WHERE u.active = true GROUP BY u.id, u.name `) // Exact SQL. Exact columns. Exact joins. // Returns only the data you need. // No hidden queries. No N+1.

The ORM was supposed to save you from writing SQL. Instead, you spend your time learning the ORM's query DSL (which is a worse version of SQL), debugging generated queries, fighting lazy-loading, and adding .Include() chains to avoid N+1 problems that wouldn't exist if you'd written the query yourself.

The cruel irony: ORMs were created because OOP developers didn't want to deal with "raw data" (SQL result sets). But a SQL result set is already a perfect data structure — it's a typed, ordered list of named fields. The ORM converts this into a worse structure (an object graph with hidden lazy-loading) and calls it an improvement.

The Service-Repository-Controller-DTO hall of mirrors

Enterprise OOP codebases follow a predictable pattern: every piece of data passes through 5+ layers of abstraction before anything happens. A single "create user" operation touches a UserController, a UserDTO, a UserService, a UserValidator, a UserRepository, a UserEntity, a UserMapper, and a UserResponse. Eight classes. For one database insert.

Go equivalent — one file, one function
func handleCreateUser(w http.ResponseWriter, r *http.Request) { var req CreateUserRequest if err := json.NewDecoder(r.Body).Decode(&req); err != nil { http.Error(w, "bad request", 400); return } if req.Email == "" || req.Name == "" { http.Error(w, "email and name required", 422); return } user, err := db.Exec(`INSERT INTO users (name, email) VALUES ($1, $2) RETURNING id`, req.Name, req.Email) if err != nil { http.Error(w, "internal error", 500); return } json.NewEncoder(w).Encode(user) }

Twelve lines. No layers. No mapping. Read top to bottom and you know exactly what happens. The "architecture" is the function.

What "no inheritance" looks like in practice

Go, Zig, and C have no class inheritance. This is by design, not by oversight. They force you to compose behavior explicitly. The result is code where you can always see exactly what's happening — no hidden parent methods, no fragile base class problem, no override surprises.

The core insight: you almost never want "is-a." You want "has-a" or "can-do." Inheritance conflates code reuse (I want to share this logic) with subtyping (I want this to be substitutable). These are different goals and should be expressed differently.

Go interfaces: implicit, structural, small

Go interfaces are satisfied implicitly — you never write implements IFoo. If your struct has the right methods, it satisfies the interface. This means the interface is defined by the consumer, not the provider. You don't have to predict in advance which interfaces your type might need to satisfy.

Java — interface declared at definition site
// The library author must predict // every interface you'll need public class FileStore implements Readable, Writable, Closeable, Flushable, Serializable { // What if you want Cacheable? // Can't add it without modifying // the source file. You're stuck. }
Go — interface declared at call site
// The consumer defines what it needs type Reader interface { Read(p []byte) (int, error) } // FileStore doesn't know about Reader. // It just has a Read method. // Any type with Read([]byte)(int,error) // satisfies Reader. No declaration needed. func process(r Reader) { /* works */ }

This inverts the dependency: the consumer specifies the minimal interface it requires. The provider doesn't need to know about it. This is real decoupling — not the fake decoupling that Java's implements provides (which is actually a compile-time coupling to the interface definition).

The Go proverb: "The bigger the interface, the weaker the abstraction." Go's standard library interfaces are tiny: io.Reader (1 method), io.Writer (1 method), fmt.Stringer (1 method), error (1 method). This is not an accident — it's the entire design philosophy.

Embedding vs. inheritance: same result, different semantics

Go's embedding gives you code reuse without subtyping. When you embed a struct, its methods get promoted to the outer struct — but there's no "is-a" relationship. The compiler literally copies method references. No vtable, no override chain, no super() calls.

Java — inheritance hierarchy for a logger
abstract class BaseHandler { protected Logger logger; protected Metrics metrics; protected void logRequest(Request r) { logger.info("handling: " + r.path); } } class UserHandler extends BaseHandler { void handle(Request r) { logRequest(r); // inherited // but now UserHandler IS-A BaseHandler // can be passed where BaseHandler expected // fragile base class: if BaseHandler // changes, all children might break } }
Go — embedding (has-a with promoted methods)
type RequestLogger struct { Logger *slog.Logger } func (rl RequestLogger) LogRequest(path string) { rl.Logger.Info("handling", "path", path) } type UserHandler struct { RequestLogger // embedded, not inherited DB *sql.DB } // UserHandler.LogRequest() works. // But UserHandler is NOT a RequestLogger. // No subtype relationship. No fragility. // Just method promotion — syntactic sugar // for UserHandler having a RequestLogger field.

Zig and C: no methods at all, and that's fine

Zig and C take the most radical position: there are no methods. There are structs (data) and functions (behavior). A function that operates on a User takes a User* as its first argument. There is zero ambiguity about what it can access or modify.

// Zig: functions are namespaced, not owned const User = struct { name: []const u8, email: []const u8, active: bool, pub fn isValid(self: User) bool { return self.name.len > 0 and self.email.len > 0; } }; // This is just syntax for: fn isValid(self: User) bool // The struct namespaces the function. That's it. // No vtable, no virtual, no override, no inheritance. // user.isValid() compiles to a direct function call.

In Zig, what looks like a method is just a function namespaced inside a struct. The self parameter is explicit. The compiler generates a direct call, not an indirect one. There's nothing to prevent inlining, nothing to confuse the branch predictor, nothing hidden.

The principle: behavior is orthogonal to data layout. A User struct defines how data is arranged in memory. Functions that operate on User define transformations. These are two separate concerns, and combining them into a "class" helps neither.

Side-by-side: the same feature in Java vs. Go

Problem: multiple types of notification (email, SMS, push) that share a common "send" contract.

Java — abstract class + subclasses
abstract class Notifier { protected String recipient; abstract void send(String msg); void sendWithRetry(String msg, int n) { for (int i = 0; i < n; i++) { try { send(msg); return; } catch (Exception e) { /* retry */ } } } } class EmailNotifier extends Notifier { void send(String msg) { /* SMTP */ } } class SMSNotifier extends Notifier { void send(String msg) { /* Twilio */ } } // 3 files. Inheritance coupling. // Can't add retry to a third-party notifier // without wrapping or inheriting from it.
Go — interface + plain functions
type Notifier interface { Send(msg string) error } func SendWithRetry(n Notifier, msg string, attempts int) error { var err error for i := 0; i < attempts; i++ { if err = n.Send(msg); err == nil { return nil } } return err } // EmailNotifier and SMSNotifier are plain structs. // They don't know about Notifier interface. // SendWithRetry works with ANY type that has Send. // Third-party types work automatically.

The Go version has no inheritance, no abstract classes, no coupling between the retry logic and the concrete implementations. SendWithRetry is a free function that works with anything satisfying a one-method interface. The Java version requires every notifier to extend Notifier — a decision that ripples through the entire codebase if you ever want to change it.

The problem isn't "classes" — it's the paradigm

Classes as a namespacing mechanism for data + related functions are fine. Go structs with methods, Zig namespaced functions, Rust impl blocks — these are all "classes" in the useful sense. The problem is the paradigm: the belief that inheritance hierarchies, encapsulation, and polymorphism via virtual dispatch should be the primary organizing principle of your code.

Most production code doesn't need polymorphism at all. It needs to read data, transform it, and write it somewhere. The question is which paradigm makes those transformations clearest.

Why Go hits the sweet spot for server-side work

Most server-side code is: receive request → validate → query database → transform → respond. This is inherently procedural. Go matches this mental model perfectly because it gives you structs (data), functions (transformations), interfaces (polymorphism where you actually need it), and goroutines (concurrency). Nothing else. No inheritance, no generics-heavy abstractions (generics exist but are deliberately constrained), no framework-level magic.

What server code actually needs vs. what OOP provides
Need OOP solution Go solution
Group related data Class with private fields + getters Struct with exported fields
Share a contract Interface + implements Interface (implicit)
Reuse logic Inheritance or abstract class Plain function or embedding
Handle errors Exceptions (hidden control flow) Return values (explicit)
Concurrency Thread pools, synchronized blocks Goroutines + channels
Dependency management DI container (Spring, etc.) Pass it as a function argument

Go's "dependency injection" is: pass the database connection to the function that needs it. No container, no annotations, no XML configuration, no reflection. The call graph is visible in the code.

Why Zig / C hit the sweet spot for systems work

Systems code (game engines, databases, embedded, OS kernels) has a different constraint: you need to control memory layout precisely because performance depends on cache behavior. OOP's per-object heap allocation is fundamentally incompatible with this requirement.

Zig and C give you:

Feature Why it matters for systems
Value-type structs Data lives where you put it — stack, array, embedded in another struct. No hidden pointers.
No hidden allocation Every allocation is explicit. You choose the allocator. You know the lifetime.
SoA layout You can arrange data in Structure-of-Arrays for cache-optimal iteration. OOP forces AoS (Array-of-Structures).
Comptime (Zig) Generate specialized code at compile time without runtime polymorphism.
No runtime No GC, no vtables, no RTTI. What you write is what runs.

When OOP is actually appropriate

OOP isn't wrong everywhere. It's wrong as a default. There are domains where object identity and message-passing are the natural model:

Domain Why OOP fits Caveat
GUI frameworks Widgets genuinely are objects with identity, state, and event handling Even here, React's functional model is winning over class components
Simulation / agent systems Independent agents with internal state and behavior ECS (data-oriented) outperforms OOP here in practice
Protocol implementations State machines with well-defined transitions Sum types / tagged unions are usually cleaner
The root issue: Java, C#, and PHP taught an entire generation that everything is a class. A file is a class. A utility function must live in a class. A namespace is a class. This "everything is an object" dogma — not classes themselves — is what broke software engineering for 20 years. The fix isn't "better OOP." It's using the right tool for the job: structs for data, functions for transformations, interfaces for contracts, composition for reuse.

The decision framework

Choosing a paradigm for your next project
Web server / API / CRUD? → Procedural (Go, plain C#, plain Python)
Game engine / real-time simulation? → Data-Oriented (C, Zig, Odin + ECS)
Compiler / parser / data pipeline? → Functional (OCaml, Haskell, Elixir)
GUI with stateful widgets? → OOP is defensible here (but consider React/SwiftUI)
Embedded / kernel / driver? → C or Zig (no runtime, full control)
Enterprise CRUD in a Java/C# shop? → Use the language, but minimize class hierarchies. Prefer composition. Keep it boring.

The worst outcome is "our entire codebase is a class hierarchy because that's what we learned in school." The best outcome is choosing the paradigm that matches the problem, and being willing to use different paradigms in different parts of the same system.
