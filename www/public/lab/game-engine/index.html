game engine architecture

the full map · every subsystem from frame loop to netcode · for developers building their first engine

What is a game engine?

A game engine is the runtime infrastructure that sits between your game logic and the hardware. It handles the things that every game needs — rendering frames, processing input, simulating physics, playing audio, managing memory — so that game code can focus on what makes a particular game that game.

The engine's fundamental job is to run a loop: read the world state, advance it by one tick, present the result on screen, repeat. Every subsystem in this reference hangs off this loop. Understanding its structure is understanding the spine of the entire architecture.

The frame lifecycle

Every frame follows the same high-level sequence. The order matters — each phase depends on the output of the previous one.

One frame, simplified
Poll Input
→
Physics
→
Game Logic
→
Render
→
Present
Poll Input captures OS events. Physics steps the simulation. Game Logic runs gameplay code (AI, state machines, scripted behavior). Render builds the command list for the GPU. Present swaps the back buffer to the display.

In practice, many of these overlap or run in parallel on different threads (physics on one, rendering on another, audio on a third). But the logical dependency order is always: know what the player did → update the world → show the result.

The timestep problem

Frames don't take constant time. A simple scene might render in 4ms; a complex one in 20ms. If you advance your simulation by "one frame" each loop, your game runs faster on fast machines and slower on slow ones. Physics breaks. Animations stutter. Multiplayer desyncs.

There are three common approaches, each with a fundamental tradeoff:

Approach How it works Upside Downside
Variable dt Pass elapsed time since last frame as dt to all systems Simple, smooth visuals Physics non-deterministic; large dt spikes cause tunneling and instability
Fixed timestep Accumulate real time; step simulation in fixed-size chunks (e.g. 1/60s) Deterministic physics; stable; reproducible Rendering between steps shows stale state — need interpolation
Semi-fixed Fixed physics step + interpolated render + capped accumulator Best of both — smooth visuals, stable physics More complex; must store previous + current state for interpolation
The industry-standard pattern is semi-fixed (Glenn Fiedler's "Fix Your Timestep!"). Physics runs at a fixed rate (e.g. 60 Hz), the renderer interpolates between the previous and current physics state based on the leftover accumulator time. This gives buttery-smooth visuals with deterministic simulation.

The canonical semi-fixed loop

const double STEP = 1.0 / 60.0; // 16.67ms fixed physics tick double accumulator = 0.0; double prev_time = get_time(); while (running) { double now = get_time(); double frame_dt = now - prev_time; prev_time = now; accumulator += frame_dt; accumulator = min(accumulator, 0.25); // clamp: avoid spiral of death // Phase 1: Fixed-rate simulation (may run 0, 1, or multiple times) while (accumulator >= STEP) { save_previous_state(); // for render interpolation poll_input(); step_physics(STEP); step_game_logic(STEP); accumulator -= STEP; } // Phase 2: Render with interpolation double alpha = accumulator / STEP; // 0.0–1.0: how far between ticks render(alpha); // lerp(prev_state, curr_state, alpha) present(); }

The spiral of death is what happens when simulation is slower than real-time: the accumulator grows every frame, causing more simulation steps per frame, which makes the frame even slower, which grows the accumulator further. The 0.25 clamp prevents this — it caps to at most ~15 physics steps per frame, dropping simulation time if the machine truly can't keep up.

Subsystem execution order

Within the loop, ordering isn't arbitrary. Here's a more complete picture showing dependencies:

Execution order within one simulation tick
1. Input — capture OS events, update action map state
2. Gameplay scripts — AI decisions, state machines, spawn/destroy entities
3. Physics — integrate forces, detect & resolve collisions
4. Post-physics logic — collision response callbacks, trigger events
5. Animation — blend trees, IK, skeleton update
6. Scene graph update — propagate transforms, update bounding volumes
7. Audio — update listener position, mix sources
Gameplay before physics: so player movement is applied before collision resolves it. Animation after physics: so ragdoll and physics-driven bones reflect the current frame. Audio last: so spatial positioning matches the final resolved transforms.

Why memory matters more than algorithms

Modern CPUs can execute billions of operations per second — but they stall for hundreds of cycles waiting for data from main memory. A cache miss on an L1 lookup costs ~4ns; a main memory fetch costs ~100ns. That's a 25× penalty. In a game engine running at 60fps (16.6ms per frame), thousands of cache misses can eat your entire budget.

This is why engine programmers obsess over data layout — how objects live in memory determines how fast you can iterate over them, which determines your frame time. The wrong data structure with O(1) access can be slower than a linear scan of a cache-friendly array.

Custom allocators

General-purpose malloc/free (or new/delete) are flexible but slow for game workloads. They maintain free-lists, deal with fragmentation, and take unpredictable time. Engine allocators trade generality for speed by constraining how memory is used.

Allocator Strategy Alloc cost Free cost Use case
Linear / Bump Increment a pointer. Reset the whole block at once. O(1) — pointer bump O(1) — reset all Per-frame scratch data, temporary strings, command buffers
Stack Linear, but you can also pop the most recent allocation (LIFO). O(1) O(1) — LIFO only Scoped temp allocations, level loading stages
Pool / Slab Pre-allocate N fixed-size blocks. Free-list of available slots. O(1) O(1) Entities, particles, bullets — many identical objects created/destroyed rapidly
Arena Allocate from a large block; free everything when the arena is destroyed. O(1) Bulk only Level lifetime data, asset groups that load/unload together
The unifying principle: restrict the freeing pattern and you eliminate fragmentation and overhead. A bump allocator is literally just ptr += size; return ptr - size; — the fastest possible allocation.

AoS vs SoA: the layout decision

When you have thousands of entities, the way you arrange their data in memory determines your cache behavior. This is the single most impactful data layout decision in engine design.

Array of Structures (AoS) — the OOP default
Entity[] entities; // each entity is a struct with all fields
pos₀ vel₀ hp₀ name₀ │ pos₁ vel₁ hp₁ name₁ │ …
Moving all entities? You load vel, hp, name into cache even though you only need pos and vel. Wasted bandwidth.
Structure of Arrays (SoA) — the DOD approach
Separate array per field — iterate only what you need
pos₀ pos₁ pos₂ pos₃ pos₄ …
vel₀ vel₁ vel₂ vel₃ vel₄ …
Moving all entities? Load only positions and velocities. Every byte in the cache line is useful. This is why SoA can be 4–10× faster for batch operations.

ECS: the engine-scale pattern

Entity Component System is SoA thinking applied to the entire engine. It separates identity, data, and behavior into three orthogonal concepts:

Concept What it is Analogy
Entity A unique ID (often just an integer/index). No data, no behavior. Just an identifier that components attach to. A row number in a spreadsheet
Component A plain data struct. Position { x, y, z }, Health { current, max }, Sprite { texture_id, uv }. No methods. A column in the spreadsheet
System A function that iterates over all entities that have a specific set of components. The behavior. A formula applied to matching rows

A "MovementSystem" queries all entities with Position + Velocity and does pos += vel * dt. It doesn't know or care if those entities also have Health or Sprite. Components compose freely — you build entity archetypes by mixing and matching data, not by inheritance hierarchies.

Why it wins: No inheritance diamonds. No virtual dispatch overhead. Data lives in cache-friendly arrays. Systems can be parallelized trivially (they operate on disjoint component sets). Adding new behavior = adding a new system, not modifying a base class.

Asset lifetime management

Textures, meshes, audio clips, and shaders are loaded from disk and live in memory until unloaded. The engine needs to track: who references this asset? and when can it be freed?

Strategy Mechanism Tradeoff
Reference counting Each asset has a count. Load increments, unload decrements. Free at zero. Simple; can leak with cycles (rare for assets). Atomic refcounts expensive in hot paths.
Handle table Assets accessed via opaque handles (ID + generation). Table maps handle → pointer. Handles can be validated. Indirection cost; safe against dangling. Generation catches use-after-free.
Arena/scope All assets for a level go in one arena. Free the arena on level transition. Bulk free is fast; no per-asset tracking. Can't share assets across levels without a global tier.

Most engines use a tiered approach: a global pool for shared assets (fonts, UI textures, player model), a level arena for level-specific data, and a frame allocator for transient per-frame data.

What the renderer actually does

The renderer's job: take a scene description (meshes, materials, lights, camera) and produce a 2D image — 60+ times per second. This involves geometry transformation, visibility determination, lighting computation, and communicating all of it to the GPU in a way that doesn't bottleneck on the CPU.

The GPU is a massively parallel processor optimized for one thing: doing the same operation on millions of data points (vertices, then pixels). The renderer's entire design revolves around keeping the GPU fed — minimizing the time it sits idle waiting for the CPU to tell it what to draw next.

The rendering pipeline

High-level stages — CPU side then GPU side
CPU — prepare work for the GPU
1. Frustum culling — discard objects outside the camera's view pyramid
2. Sorting — group by material/shader to minimize GPU state changes
3. Batching — merge compatible draws into fewer GPU calls
4. Command submission — record draw commands into a command buffer
GPU — execute the pipeline
5. Vertex shader — transform vertices from model space → clip space
6. Rasterization — convert triangles to fragments (candidate pixels)
7. Fragment shader — compute final color per pixel (lighting, texturing)
8. Output merge — depth test, blending, write to framebuffer

Draw calls: the CPU-GPU bottleneck

A draw call is a single instruction from the CPU to the GPU: "draw this mesh with this material." Each one has overhead — validating state, transferring uniform data, synchronizing. A desktop GPU can handle millions of triangles but may choke on 10,000 individual draw calls.

This is the central tension of rendering architecture: the GPU wants volume, the CPU wants simplicity. Every rendering optimization technique is, at its core, a strategy for submitting fewer, larger draw calls.

Technique Strategy When to use
Static batching Merge non-moving meshes sharing a material into one big mesh at load time Environment geometry, props, decorations
Dynamic batching Merge small meshes per-frame into a shared vertex buffer Particles, small dynamic objects
Instancing Draw one mesh N times with per-instance data (position, color, scale) Forests, crowds, grass — many copies of the same mesh
Indirect drawing GPU itself decides what and how many to draw (GPU-driven rendering) Modern engines; Nanite-style virtual geometry

Forward vs Deferred rendering

The two dominant approaches to handling multiple lights differ in when lighting is computed.

Forward rendering
Compute lighting in the fragment shader, per object, per light
Cost: objects × lights
Simple to implement
Handles transparency naturally
MSAA works out of the box
Struggles with many dynamic lights
Deferred rendering
First pass: write geometry data to G-buffer (position, normal, albedo)
Second pass: compute lighting per pixel from G-buffer
Cost: pixels × lights (objects don't matter)
Handles hundreds of lights efficiently
High memory bandwidth; MSAA is expensive
Transparency requires separate forward pass

Most modern engines use deferred for opaque geometry (to handle many lights efficiently) and a forward pass for transparent objects (which can't write to a G-buffer meaningfully, since multiple surfaces overlap per pixel).

The material/shader model

A material is the combination of a shader program (vertex + fragment code) and its parameter bindings (textures, colors, roughness values). The engine typically organizes materials into a hierarchy to minimize state changes:

Sorting key structure (most expensive changes first)
Render pass > Shader program > Material (textures) > Depth (front-to-back for opaque)
Objects are sorted by this key before submission. Changing render pass or shader is expensive (pipeline state change). Changing textures is moderate. Depth sorting maximizes early-Z rejection (fragments behind already-drawn geometry are discarded before the fragment shader runs).

Graphics API landscape

API Level Platform Notes
OpenGL / OpenGL ES High-level Cross-platform Legacy but simple. Driver does most work. Still used in learning / older engines.
Vulkan Low-level Cross-platform (excl. Apple) Explicit control over memory, sync, command buffers. Verbose but performant.
DirectX 12 Low-level Windows / Xbox Similar philosophy to Vulkan. Dominant on Windows for AAA.
Metal Mid-level Apple only Modern, clean API. Required for iOS/macOS.
WebGPU Mid-level Browsers / native (via wgpu) Cross-platform abstraction over Vulkan/Metal/DX12. Rising fast.

Most custom engines abstract over multiple APIs via a Render Hardware Interface (RHI) — a thin translation layer that maps engine-level concepts (buffers, textures, pipelines) to API-specific calls.

What the physics engine solves

The physics engine answers two questions every tick: "Is anything overlapping?" (collision detection) and "What happens when things overlap?" (collision response). On top of this, it integrates forces to move rigid bodies — gravity, impulses, friction, constraints.

The core challenge is performance: checking every pair of N objects for collision is O(N²). With thousands of objects, that's millions of checks per frame. The entire architecture of a physics engine is designed to avoid doing unnecessary work.

The two-phase collision pipeline

Broad phase → Narrow phase

Broad phase uses simple bounding volumes (axis-aligned bounding boxes) and spatial structures to quickly eliminate pairs that can't possibly be colliding. This reduces the candidate set from O(N²) to something manageable.
Narrow phase performs exact geometric tests (GJK, SAT) only on candidate pairs. Expensive, but only runs on a small subset.

Spatial partitioning strategies

Broad phase needs a data structure that answers "which objects are near each other?" efficiently. The choice depends on your game's characteristics:

Structure How it works Best for Weakness
Uniform grid Divide world into fixed-size cells. Objects register in cells they overlap. Uniformly distributed objects of similar size (particles, bullets) Wastes memory on empty cells; struggles with varied object sizes
Quadtree / Octree Recursively subdivide space. Deeper nodes = smaller regions. 2D/3D games with clustered objects Objects spanning cell boundaries need multi-registration
BVH (Bounding Volume Hierarchy) Binary tree of bounding volumes wrapping groups of objects. Top-down or incremental rebuild. General-purpose; dominant in modern engines Rebuild cost when many objects move; incremental updates help
Sort-and-sweep Sort AABBs along one axis; overlaps on sorted axis = candidate pairs. Scenes where objects mostly move slowly (overlap lists stay coherent) Degrades with clustered objects along the sweep axis

Rigid body simulation

A rigid body has mass, position, orientation, and velocity (linear + angular). Each physics tick follows the same loop:

One physics tick
1. Apply forces — gravity, user impulses, springs, drag
2. Integrate — velocity += force/mass * dt; position += velocity * dt
3. Detect collisions — broad → narrow → generate contacts
4. Resolve constraints — push overlapping bodies apart, apply impulses
5. Apply velocity corrections — friction, restitution (bounciness)
Steps 3–5 often iterate multiple times (solver iterations) because resolving one contact can create or worsen another. More iterations = more stable stacking, but more CPU cost.
Tunneling: a fast-moving object can pass entirely through a thin wall between two ticks (bullet-through-paper problem). Solutions: continuous collision detection (CCD) — sweep the object's path between ticks and find the first time of impact. Expensive, so typically only enabled for fast/small objects.

Collision shapes

Narrow phase performance depends entirely on the geometric primitives used. Engines support a hierarchy from cheap to expensive:

Shape Test cost Use case
Sphere Trivial — distance check Projectiles, particles, simple characters
AABB (Axis-Aligned Box) Very cheap — 3 axis interval overlaps Broad phase, triggers, boxy objects
OBB (Oriented Box) Moderate — SAT with 15 axes Rotated crates, vehicles
Capsule Cheap — line segment distance Characters (common for player controllers)
Convex hull Moderate — GJK algorithm Complex rigid bodies
Triangle mesh Expensive — per-triangle tests via BVH Static environment only

Why these three share a tab

Input, audio, and UI are the engine's interface layer — the systems that bridge the player and the simulation. They share common challenges: platform-specific APIs, latency sensitivity, and the need for an abstraction layer that keeps game code portable.

Input abstraction

Raw input comes from the OS as hardware events: key presses, mouse deltas, gamepad axis values, touch coordinates. The engine needs to transform these into game-meaningful actions that game code queries without knowing which device produced them.

Input abstraction pipeline
OS events
key, mouse, pad
→
Device layer
normalize, deadzone
→
Action map
bind → action
→
Game code
is_action("jump")?
The action map is the key abstraction: it decouples "spacebar was pressed" from "the player wants to jump." This allows rebinding, multiple input devices, and input recording/replay.
Input concept What it means
Action A named game intent: "jump," "fire," "move_x." Can be bound to any hardware input.
Binding A mapping from hardware input to action. "Space → jump," "Gamepad A → jump."
Context A set of bindings active at a given time. "Gameplay" context vs "Menu" context vs "Vehicle" context.
Deadzone Threshold below which analog stick input is treated as zero — prevents drift from hardware imprecision.
Input buffering Storing recent inputs for a short window so the game can accept a "jump" pressed slightly before landing. Critical for responsive feel in action games.

Audio engine

Audio runs on its own thread with its own fixed-rate tick (typically 48 kHz for mixing). The engine's audio system manages three concerns:

Concern What it involves Key concepts
Playback & mixing Playing multiple sound sources simultaneously, blending them into a final stereo/surround output Voice limits, priority system, bus routing (SFX bus, music bus, voice bus), volume ducking
Spatialization Making sounds feel positioned in 3D space relative to the listener Panning, distance attenuation, HRTF (head-related transfer function), reverb zones
Streaming Large audio (music, voice lines) loaded in chunks rather than entirely in memory Ring buffers, async I/O, decode-on-demand (Vorbis, Opus)
Voice limits: hardware and mixing budget limit how many sounds play simultaneously (64–256 is typical). When the limit is hit, a priority system decides which to cull. Gunfire near the player wins over ambient wind across the map.

UI: immediate mode vs retained mode

Game UI is fundamentally different from desktop application UI. Games rebuild their entire screen every frame, which enables a simpler programming model:

Immediate Mode (IMGUI)
No persistent UI objects — you "draw" the UI every frame
if (button("Fire")) { shoot(); }
State lives in game code, not in UI widgets
Easy to debug, easy to add/remove UI dynamically
Great for debug tools, HUDs, in-game menus
Scales poorly for complex layouts (no built-in layout engine)
Retained Mode
Persistent UI tree (like DOM or scene graph for UI)
Create widgets once, update properties when state changes
Built-in layout, styling, animation, accessibility
Better for complex UIs: inventories, skill trees, store pages
More overhead, harder to synchronize with game state
Examples: Scaleform, Coherent, Flutter (experimental)

Many engines use both: immediate mode for gameplay HUD and debug overlays, retained mode (often HTML/CSS-based via embedded browser) for menus, stores, and complex layouts.

The editor-runtime boundary

A game engine serves two audiences: the runtime (what players run — fast, lean, shipping code) and the editor (what developers use — inspection tools, scene manipulation, hot-reload, undo). The architecture must support both without letting editor features pollute runtime performance.

This split influences everything: how entities are created (editor-spawned with undo vs. runtime-spawned from code), how data is serialized (human-readable for the editor vs. binary for shipping), and how code is loaded (hot-reloadable during development vs. statically linked for release).

Entity lifecycle

Entities are created, live, and are destroyed — but the timing of these events is surprisingly tricky. You can't destroy an entity mid-iteration over a system (it invalidates the iterator). You can't create one and immediately reference it in the same tick (components may not be allocated yet). Most engines use a deferred command pattern:

Deferred entity operations
During tick — systems queue commands: spawn(archetype), destroy(entity), add_component(entity, data)
End of tick — engine processes the command queue atomically: all creates, then all destroys
Next tick — new entities are visible to all systems; destroyed entities are gone
This guarantees that no system ever sees a half-constructed or half-destroyed entity. The command queue also makes undo trivial in the editor — just replay the inverse commands.

Scene serialization

A "scene" or "level" is a collection of entities with their component data, plus references to assets (meshes, textures, sounds). The engine needs to save this to disk and load it back — faithfully, quickly, and across engine versions.

Format Use case Tradeoff
Text (JSON, YAML, TOML) Editor saves, human-readable diffs, version control Slow to parse, large files, easy to merge in Git
Binary (custom / FlatBuffers) Shipping builds, fast loading Fast, compact, but opaque and version-sensitive
Hybrid Text in editor, "cook" to binary for builds Best of both — extra build step. This is the standard approach.
Asset cooking: the process of converting source assets (PNG, FBX, WAV) into runtime-optimized formats (compressed GPU textures, optimized mesh buffers, encoded audio). This happens at build time, not runtime. The cooked formats are platform-specific — a GPU texture for PC differs from one for mobile.

Scripting & hot-reload

Iteration speed is everything during development. Recompiling a C++ engine takes seconds to minutes; designers and gameplay programmers need faster feedback loops. This drives the scripting layer:

Approach Languages Characteristics
Embedded scripting Lua, Wren, AngelScript Lightweight VM embedded in engine. Hot-reload by re-executing scripts. Low barrier. Sandboxed. Performance cost for heavy logic.
Managed runtime C# (Unity), GDScript (Godot) Richer language features, good tooling. Hot-reload via assembly reloading. Moderate overhead.
Native hot-reload C/C++ (via DLL/SO reload) Compile gameplay to a shared library; engine unloads/reloads it. Full native speed. Fragile — state migration across reloads is hard.
Visual scripting Blueprints (Unreal), Bolt (Unity) Node graphs. Accessible to non-programmers. Limited expressiveness. Can generate native code.

Most engines use a two-tier approach: performance-critical systems (rendering, physics, audio) in native code; gameplay logic in a scripting layer. The boundary between tiers is defined by the component interface — scripts read and write component data, native systems process it.

Prefabs and instancing

A prefab (or archetype template) is a reusable entity blueprint — "enemy_goblin" defines a set of components with default values. Instances in the scene can override specific values (this goblin has 50 HP instead of the default 100). When you change the prefab, all instances update — unless they've overridden that field.

This is essentially prototype-based inheritance for game objects. The technical challenge: tracking which fields are overridden per instance, and propagating prefab changes efficiently to thousands of instances without reserializing the entire scene.

The fundamental constraint

Network programming in games is dominated by one fact: light takes time to travel. Even at the speed of light, a round trip between New York and London takes ~60ms. Add routing, processing, and congestion, and real-world latency is 50–200ms. Your game loop ticks every ~16ms. By the time a player's input reaches the server, the game has advanced 3–12 frames.

Every networking architecture in games is a strategy for hiding, compensating for, or tolerating latency while keeping all players' views of the world acceptably consistent.

Network topologies

Client-Server
One authoritative server; clients send input, receive state
Server validates everything — anti-cheat built in
Scales to many players
Client sees delayed state; needs prediction
Server costs money to run
Used by: FPS, MMOs, most competitive games
Peer-to-Peer
All peers run the same simulation
No server cost
Latency = worst connection in the group
Trust problem — every peer can cheat
Hard to scale beyond ~8 players
Used by: fighting games, RTS, co-op

State synchronization strategies

The core question: how do multiple machines agree on the game state when messages take time to arrive?

Strategy How it works Latency feel Typical genre
Lockstep All peers exchange inputs for tick N; nobody advances until all inputs arrive. Same inputs → same result (deterministic simulation). Input delay = round-trip latency. Consistent but sluggish. RTS (StarCraft), some fighting games
Rollback Run simulation locally with predicted inputs. When real inputs arrive, if they differ, rewind to that tick and resimulate forward. Requires determinism + fast re-execution. Zero perceived input lag locally. Occasional "rollback jitter" when predictions are wrong. Fighting games (GGPO), some shooters
Server authority + client prediction Client simulates its own movement immediately (prediction). Server runs the authoritative sim. If server disagrees, client corrects (reconciliation). Other players are interpolated between known server snapshots. Own character feels responsive; others are slightly behind (interpolation delay). FPS (Quake, Overwatch, Valorant), action games
State broadcast Server periodically sends full or delta state snapshots to all clients. Clients interpolate/extrapolate between snapshots. Simple but high bandwidth. Smooth for slow-moving games. MMOs, simulation games, casual multiplayer

Client prediction & reconciliation

This is the most important pattern in real-time multiplayer and worth understanding at a conceptual level:

The prediction/reconciliation loop
1. Client processes input locally, advances own state immediately (feels responsive)
2. Client sends input + tick number to server
3. Server applies input to authoritative state, sends back confirmed state for that tick
4. Client receives server state for tick N. Compares with its own predicted state at tick N.
Match? → Prediction was correct. Discard saved state, continue.
Mismatch? → Snap to server state, re-apply all inputs from tick N+1 to current tick. This is reconciliation.
The client keeps a buffer of recent unconfirmed inputs. On mismatch, it replays them on top of the corrected server state. The result: the player's own character moves immediately and stays consistent with the server.
Entity interpolation: other players' positions arrive as server snapshots with network jitter. Rather than applying them immediately (jerky), the client renders them slightly in the past and smoothly interpolates between the two most recent known positions. This adds ~half a tick of visual delay but eliminates jitter.

Transport: TCP vs UDP

Protocol Guarantees Game use
TCP Ordered, reliable delivery. Retransmits lost packets. Chat, matchmaking, inventory, login. Not real-time state — a lost packet stalls everything behind it (head-of-line blocking).
UDP None. Packets may arrive out of order, duplicated, or not at all. Real-time state sync. The game builds its own reliability layer on top — selective acks, redundancy, sequence numbers.

Most game networking stacks use UDP with a custom reliability layer: unreliable-unordered for position updates (stale data is useless — just use the latest), reliable-ordered for critical events (kills, item pickups, chat). Libraries like ENet, GameNetworkingSockets, and Laminar implement this pattern.
