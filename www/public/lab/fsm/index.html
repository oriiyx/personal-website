finite state machines for Unity

from theory to MonoBehaviour · enemy AI · player control · game managers · C#

Why does every game need state machines?

Every entity in a game — enemies, the player, the UI, the game session itself — has modes of behavior that change over time. An enemy patrols until it sees you, then chases, then attacks. A player is idle until they press jump, then they're airborne, then landing. A game is on a menu screen, then playing, then paused.

The naive approach is to track each behavior with booleans and branch on them every frame:

The boolean swamp
bool isPatrolling;
bool isChasing;
bool isAttacking;
bool isDead;
bool isStunned;

void Update() {
  if (isDead) return;
  if (isStunned && !isDead) { ... }
  if (isAttacking && !isStunned) { ... }
  if (isChasing && !isAttacking
      && !isStunned) { ... }
  if (isPatrolling && !isChasing
      && !isAttacking) { ... }
}
The FSM version
enum State {
  Idle, Patrol, Chase,
  Attack, Dead
}

State current = State.Idle;

void Update() {
  switch (current) {
    case State.Patrol:  DoPatrol();  break;
    case State.Chase:   DoChase();   break;
    case State.Attack:  DoAttack();  break;
    case State.Dead:    break;
  }
}

The boolean version has 25 = 32 possible flag combinations, most of which are invalid (you can't be patrolling AND dead). You have to defensively negate every other flag in every branch. Add one more boolean and every existing branch needs another && !newFlag check.

The FSM version has exactly 5 possible states, all of them valid by construction. The entity is in exactly one state at any given time. Adding a new state means adding one enum value and one case — existing states don't change at all.

The three things that break without an FSM

Failure mode 1 — Illegal state combinations
With booleans, nothing prevents isAttacking = true and isDead = true simultaneously. You get a dead enemy that's still swinging its sword. An FSM makes this structurally impossible — the entity is in Attack OR Dead, never both.
Failure mode 2 — Missing enter/exit logic
When an enemy starts chasing, you need to play a "spotted" bark, set the nav agent speed, start a chase animation. When it stops chasing, you need to reset speed and clear the target. With if/else, this enter/exit logic has nowhere clean to live — it fires every frame or you add wasChasing flags. An FSM gives you explicit OnEnter and OnExit hooks per state.
Failure mode 3 — Uncontrolled transitions
An enemy in mid-attack shouldn't be able to suddenly start patrolling because the player walked away. With booleans, any code anywhere can flip any flag. An FSM makes transitions explicit and guarded — you define exactly which states can transition to which others, and under what conditions.

The tradeoff

An FSM adds a small amount of structure upfront — defining states, transitions, enter/exit hooks — in exchange for predictable behavior at any scale. For a solo developer, this matters enormously: you can come back to an enemy's behavior six months later and immediately read its state diagram to understand what it does. The boolean swamp is write-only code.

The rule of thumb: if an entity has 3+ distinct behaviors that shouldn't overlap, use an FSM. Two behaviors (alive/dead, on/off) can be a simple boolean. Three or more — you want a state machine.

The five formal components

A finite state machine is formally a 5-tuple (S, s₀, Σ, δ, F). In game development terms:

Formal Game term What it is Example
S States The finite set of modes the entity can be in Idle, Patrol, Chase, Attack, Dead
s₀ Initial state Which state the entity starts in Idle
Σ Events / inputs Things that can trigger a transition PlayerSpotted, ReachedWaypoint, TookDamage
δ Transitions Rules: (state + event) → new state Chase + PlayerLost → Patrol
F Terminal states States with no outgoing transitions Dead

In practice, game FSMs extend this formal model with lifecycle hooks — the most important practical addition.

The three lifecycle hooks

Every state gets three hooks. This is where the actual behavior lives — it's the reason you use an FSM instead of raw switch statements.

State lifecycle
OnEnter
Runs once when entering this state
Set animation, play sound, initialize timers, configure nav agent, reset counters
OnUpdate
Runs every frame while in this state
Check transition conditions, do per-frame work (move toward waypoint, track player, etc.)
OnExit
Runs once when leaving this state
Clean up: stop coroutines, reset nav agent, clear target references, stop particles
OnEnter/OnExit are the key advantage over raw if/else — they give you guaranteed setup and teardown. Without them, you're tracking "did I already start the chase animation?" with yet another boolean.

Transitions, guards, and actions

A transition is a directed edge from one state to another. It has up to three components:

Anatomy of a transition
Patrol
→
guard: dist < 10
condition that must be true
→
action: PlayBark()
side effect during transition
→
Chase
Guards are boolean conditions checked every frame (or on events). If the guard is true, the transition fires. Actions are one-shot effects that happen during the transition — after the old state's OnExit but before the new state's OnEnter. Most transitions only need a guard.
Transition priority matters. If multiple outgoing transitions have guards that are simultaneously true, the FSM picks the first one checked. Design your guards so this is either impossible (mutually exclusive conditions) or intentional (higher-priority transitions checked first).

Polled vs. event-driven transitions

Style How it works Good for
Polled Check guard conditions every frame in OnUpdate Distance checks, health thresholds, timers — things that change continuously
Event-driven External systems call fsm.HandleEvent(type) Player input, collision triggers, animation events — discrete happenings

Most game FSMs use both. Enemy AI tends to be polled (checking distance to player every frame). Player controllers tend to be event-driven (reacting to input). Game managers are almost entirely event-driven.

The full transition table

# From To Guard (condition) Priority

T8 is a "global" transition — it fires from any state. This is the standard pattern for death, stun, or interrupt effects. Check global transitions before state-specific ones so they always take priority.

Each transition fires exactly one OnExit (old state) then one OnEnter (new state). At no point is the entity in two states simultaneously.

What each OnUpdate actually does

State OnUpdate (every frame) Transitions checked (in order)
Idle Increment idleTimer. Face random direction slowly. T8 → T2 → T1
Patrol NavMeshAgent moves toward current waypoint. T8 → T4 → T3
Chase Set agent destination = player pos. Track lostTimer. T8 → T5 → T6
Attack Face player. Wait for animation. T8 → T7
Dead Nothing. None (terminal)
Every state checks T8 (death) first. This is the "global transition" pattern — rather than duplicating logic in a wrapper, each state simply checks the global condition at the top of its OnUpdate before anything else.

Three implementation patterns

There are three common ways to implement an FSM in Unity C#. Choosing the wrong one wastes either your time (over-engineering) or your future time (under-engineering).

Pattern 1: Enum + Switch

Everything lives in one MonoBehaviour. States are enum values. Behavior is a switch in Update(). Transitions are direct enum assignments.

public class GameManager : MonoBehaviour { enum GameState { Menu, Playing, Paused, GameOver } GameState _state = GameState.Menu; void Update() { switch (_state) { case GameState.Menu: if (startPressed) ChangeState(GameState.Playing); break; case GameState.Playing: if (Input.GetKeyDown(KeyCode.Escape)) ChangeState(GameState.Paused); if (playerHealth <= 0) ChangeState(GameState.GameOver); break; } } void ChangeState(GameState next) { // Exit logic switch (_state) { case GameState.Paused: Time.timeScale = 1; break; } _state = next; // Enter logic switch (_state) { case GameState.Paused: Time.timeScale = 0; break; case GameState.GameOver: ShowGameOverUI(); break; } } }

When this breaks down: any single state's logic exceeds ~30 lines. You want to share states between entity types. Enter/exit logic bloats ChangeState. Time to upgrade to Pattern 2.

Pattern 2: State objects

Each state is its own class implementing a common interface. The FSM holds a reference to the current state and delegates to it. This is the workhorse pattern — it handles 90% of game FSM needs.

public interface IState { void OnEnter(); void OnUpdate(); void OnFixedUpdate(); void OnExit(); } public class StateMachine { public IState Current { get; private set; } public void ChangeState(IState next) { if (next == null) return; Current?.OnExit(); Current = next; Current.OnEnter(); } public void Update() => Current?.OnUpdate(); public void FixedUpdate() => Current?.OnFixedUpdate(); }

Each state class receives the entity (MonoBehaviour) in its constructor and calls entity.ChangeState(nextState) when transition guards are met. States own their own transition logic.

Pattern 3: Hierarchical FSM

A state can itself contain a sub-FSM. The classic example: a boss with combat phases.

Hierarchical nesting
BossAI (root FSM)
├─ Phase1 (sub-FSM)
├─ Shoot · ├─ Dodge · └─ Summon
├─ Phase2 (sub-FSM)
├─ Charge · ├─ Slam · └─ Enrage
└─ Dead
The root FSM transitions between phases (Phase1 → Phase2 at 50% HP). Each phase has its own internal states. When the root exits Phase1, the sub-FSM's active state gets OnExit first, then Phase1 itself.

When you need this: when you're naming states like CombatMelee, CombatRanged, CombatDefend. Those "Combat" prefixes mean there's a parent state wanting to be a group. Start flat and refactor if needed.

Decision guide

Entity Recommended Why
Game manager Enum + Switch Few states, minimal per-state logic, one instance
Simple enemy (slime, turret) Enum + Switch 2–4 states, straightforward behavior
Player controller State Objects Many states, complex enter/exit, grows over time
Main enemy types State Objects Reusable states, 4+ states with nontrivial logic
Boss with phases HFSM Natural two-level structure
UI flow (menus, dialogs) Enum + Switch Simple sequencing, rarely needs reuse

Where the FSM lives in MonoBehaviour

The FSM is a plain C# object (not a MonoBehaviour) owned by the entity. The entity delegates its Unity lifecycle callbacks to the FSM:

public class EnemyController : MonoBehaviour { readonly StateMachine _fsm = new(); // Expose references that states will need public NavMeshAgent Agent { get; private set; } public Animator Anim { get; private set; } public Transform Player { get; private set; } public Transform[] Waypoints; public float SightRange = 12f; public float AttackRange = 2f; public float Health = 100f; void Awake() { Agent = GetComponent<NavMeshAgent>(); Anim = GetComponent<Animator>(); Player = GameObject.FindWithTag("Player").transform; } void Start() => _fsm.ChangeState(new IdleState(this)); void Update() => _fsm.Update(); void FixedUpdate() => _fsm.FixedUpdate(); // Public so states can trigger transitions public void ChangeState(IState next) => _fsm.ChangeState(next); public bool CanSeePlayer() { Vector3 dir = Player.position - transform.position; if (dir.magnitude > SightRange) return false; return !Physics.Raycast(transform.position + Vector3.up, dir.normalized, dir.magnitude, obstacleMask); } }
Why not make each state a MonoBehaviour? Adding/removing components on transition is expensive (GC allocation, GetComponent overhead), breaks serialization, and means states can't exist before they're entered. Plain C# objects are created once and essentially free to swap.

Update vs FixedUpdate in states

Put in OnUpdate (per frame) Put in OnFixedUpdate (physics step)
Input reading Rigidbody forces / velocity
Animation parameter updates Physics raycasts for ground checks
Timer increments (Time.deltaTime) Character controller movement (if physics-based)
NavMeshAgent destination setting Overlap checks for attack hitboxes
Transition guard checks —

For NavMeshAgent-based enemies (most enemies), you almost never need OnFixedUpdate — the agent handles its own physics. For the player controller with a Rigidbody, you'll use both.

Coroutines inside states

Some behaviors are sequential: play attack animation, wait, deal damage at a specific frame, check transitions. Coroutines handle this — but they're tied to the MonoBehaviour, not the state.

public class AttackState : IState { readonly EnemyController _e; Coroutine _routine; public AttackState(EnemyController e) => _e = e; public void OnEnter() { _e.Agent.isStopped = true; _e.Anim.SetTrigger("Attack"); _routine = _e.StartCoroutine(Sequence()); } IEnumerator Sequence() { yield return new WaitForSeconds(0.4f); // wind-up DealDamage(); yield return new WaitForSeconds(0.3f); // follow-through // Attack finished — check if player retreated float dist = Vector3.Distance( _e.transform.position, _e.Player.position); if (dist > _e.AttackRange) _e.ChangeState(new ChaseState(_e)); else _e.ChangeState(new AttackState(_e)); // self-transition: re-attack } public void OnExit() { // CRITICAL: stop coroutine if we exit early (death) if (_routine != null) _e.StopCoroutine(_routine); _e.Agent.isStopped = false; } public void OnUpdate() { // Still check death even during attack if (_e.Health <= 0) _e.ChangeState(new DeadState(_e)); } public void OnFixedUpdate() { } }
Always stop coroutines in OnExit. If the enemy dies mid-attack (T8), the coroutine is still running on the MonoBehaviour. Without cleanup, it will try to deal damage after death or call ChangeState on a dead entity.

Animation integration

Approach How When to use
FSM drives Animator OnEnter sets parameters/triggers Simple enemies, 1:1 state-to-clip mapping
Animator events drive FSM Animation Events call methods that trigger transitions When timing depends on clip length (attacks, casts)

The cleanest pattern: FSM drives Animator for entry (set animation in OnEnter), Animation Events drive FSM for exit (event at end of attack clip triggers transition check). The FSM controls what happens, the animation controls when.

Complete PatrolState walkthrough

This is the guard's Patrol state from the worked example, fully implemented. It demonstrates every concept: constructor injection, enter/exit hooks, prioritized transition checking, and early return after state change.

public class PatrolState : IState { readonly EnemyController _e; int _waypointIndex; public PatrolState(EnemyController enemy) => _e = enemy; public void OnEnter() { _e.Agent.isStopped = false; _e.Agent.speed = 2f; _e.Anim.SetBool("Walking", true); GoToNextWaypoint(); } public void OnUpdate() { // T8: global — always first if (_e.Health <= 0) { _e.ChangeState(new DeadState(_e)); return; // ← CRITICAL: stop executing old state } // T4: player spotted — higher priority than arrival if (_e.CanSeePlayer()) { _e.ChangeState(new ChaseState(_e)); return; } // T3: reached waypoint — go idle if (!_e.Agent.pathPending && _e.Agent.remainingDistance < 0.5f) { _e.ChangeState(new IdleState(_e)); return; } } public void OnFixedUpdate() { } public void OnExit() { _e.Anim.SetBool("Walking", false); } void GoToNextWaypoint() { if (_e.Waypoints.Length == 0) return; _e.Agent.SetDestination( _e.Waypoints[_waypointIndex].position); _waypointIndex = (_waypointIndex + 1) % _e.Waypoints.Length; } }

Avoiding GC allocations

The code above creates new ChaseState(_e) on every transition. For most games this is fine. If you have 100+ enemies transitioning frequently and see GC spikes in the profiler, pre-allocate:

public class EnemyController : MonoBehaviour { // Pre-allocate all states once in Awake public IdleState IdleState { get; private set; } public PatrolState PatrolState { get; private set; } public ChaseState ChaseState { get; private set; } public AttackState AttackState { get; private set; } public DeadState DeadState { get; private set; } void Awake() { IdleState = new IdleState(this); PatrolState = new PatrolState(this); ChaseState = new ChaseState(this); AttackState = new AttackState(this); DeadState = new DeadState(this); } // Now transitions are zero-alloc: // _e.ChangeState(_e.ChaseState); }
Don't pre-optimize. Start with new on every transition. Profile first. State object allocations won't show up until you have hundreds of entities transitioning per frame. Premature optimization here costs readability for zero measurable gain in most games.

Debugging: state labels in the scene

// Add to StateMachine public string CurrentName => Current?.GetType().Name ?? "None"; // Add to EnemyController void OnDrawGizmos() { #if UNITY_EDITOR UnityEditor.Handles.Label( transform.position + Vector3.up * 2f, _fsm.CurrentName); #endif }

When an enemy misbehaves, the floating label immediately tells you what state it's in. From there you read that one state's OnUpdate. Compare this to debugging the boolean swamp, where you'd check every flag and mentally simulate every branch.

Setup checklist for new entities

For every new entity type:

1. Define your states as an enum (even with state objects — useful for serialization and debug logging).

2. Draw the state diagram on paper — circles for states, arrows for transitions, labels for guard conditions.

3. Create one C# class per state implementing IState.

4. Write transition guards in each state's OnUpdate, in priority order (global first).

5. Add OnDrawGizmos to the controller for scene-view debugging.

6. Test by forcing the entity into each state and verifying every outgoing transition fires correctly.
