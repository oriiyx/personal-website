the observer pattern from first principles

GoF behavioral · publish-subscribe core · Go idioms · bridge to event systems

What coupling problem does this solve?

You have a thing that changes — a price feed, a player's health, a config reload — and multiple other things that need to react to that change. The naïve approach: the thing that changes calls each reactor directly.

PriceFeed must import Dashboard, AlertEngine, and Logger. It knows about all of them. Adding a fourth reactor means editing PriceFeed's source.

This creates three problems. First, the source depends on every reactor — it must import their packages, know their method signatures, and be recompiled when any of them changes. Second, new reactors require source modification — you can't add a "write to Kafka" reactor without touching PriceFeed. Third, the reactors can't be swapped at runtime — the call graph is baked in at compile time.

In a game engine, this is the exact same problem: a HealthComponent changes and the UI, sound system, achievement tracker, and death logic all need to know — but HealthComponent shouldn't import any of them.

The inversion

The Observer pattern inverts the dependency. Instead of the source calling each reactor, reactors register themselves with the source, and the source calls them through an agreed-upon interface. The source never learns their concrete types.

PriceFeed depends only on the Observer interface, not on Dashboard or AlertEngine. The arrows of dependency now point toward an abstraction.

The tradeoff

You get loose coupling and runtime flexibility. You pay with indirection — when reading PriceFeed's code, you can't immediately see who receives its notifications. You also pay with lifecycle complexity — observers must unsubscribe or you get goroutine leaks / stale references. And notification order is typically undefined, so observers must not depend on being called before or after each other.

The core contract: "I don't know who you are, and I don't care how many of you there are. If you implement this interface, I'll call you when something changes. You're responsible for subscribing and unsubscribing."

Three roles, two interfaces

Every Observer implementation has three participants. The Subject (also called Publisher or Observable) is the thing that changes. The Observer (also called Subscriber or Listener) is the thing that reacts. The Event (also called Notification or Message) is the data that flows from subject to observer.

In Go: interfaces and a struct

Go's implicit interface satisfaction is a natural fit. An observer doesn't "extend" a base class — it just has the right method signature. Here's the minimal contract:

// Event is the payload carried from subject to observers. type Event struct { Type string // e.g. "price_updated", "health_changed" Data any // the payload — observers type-assert as needed } // Observer is anything that can receive a notification. type Observer interface { OnNotify(event Event) } // Subject is anything that tracks observers and notifies them. type Subject interface { Subscribe(obs Observer) Unsubscribe(obs Observer) Notify(event Event) }

Why any for the event data?

The subject doesn't know what its observers need to do with the data. A dashboard needs a price float, an alert engine needs a threshold struct, a logger needs a string representation. Using any (Go 1.18+ alias for interface{}) lets the event carry domain-specific data without the contract itself importing domain types.

The cost: observers must type-assert. The benefit: the Observer interface stays stable as the system grows. In a typed event system (the end-goal), you'd replace this with generics — but at the pattern level, any is the right starting point.

Why not generics here? A Subject[T] can only notify observers of type Observer[T]. If a subject emits multiple event types (common in game engines), you'd need a subject per type or fall back to any anyway. Generics shine in the typed event bus layer — we'll get there in the last tab.

The subscribe/unsubscribe contract

Subscribe appends to the observer slice. Unsubscribe removes from it. Two design decisions to lock down early:

Decision Choice Rationale
Identity Pointer equality Go interfaces are comparable if their underlying types are. For unsubscribe, we compare the interface value directly — this works because observers are typically pointer receivers (*Dashboard), and two pointers to the same struct are equal.
Duplicates Prevent Subscribing the same observer twice means it gets notified twice. Almost never what you want. Check-before-append is cheap for the typical observer count (single digits to low hundreds).
Thread safety sync.RWMutex Notifications (read-path) are frequent; subscribe/unsubscribe (write-path) are rare. RWMutex lets multiple goroutines notify concurrently while serializing subscription changes.
Notify during unsubscribe Snapshot the slice Copy the observer slice under lock before iterating. Observers can safely unsubscribe from within OnNotify without corrupting the iteration. Covered in detail in the Edge Cases tab.

Domain: stock price feed

A PriceFeed receives new prices and notifies subscribers. Two observers: a Display that prints the price, and an AlertEngine that fires when the price exceeds a threshold. Real values, real code.

The subject: BaseSubject

We implement the subscription machinery once in a reusable struct that any domain subject can embed:

type BaseSubject struct { mu sync.RWMutex observers []Observer } func (s *BaseSubject) Subscribe(obs Observer) { s.mu.Lock() defer s.mu.Unlock() // Prevent duplicate subscriptions for _, existing := range s.observers { if existing == obs { return } } s.observers = append(s.observers, obs) } func (s *BaseSubject) Unsubscribe(obs Observer) { s.mu.Lock() defer s.mu.Unlock() for i, existing := range s.observers { if existing == obs { // Swap with last, truncate — O(1), no allocation s.observers[i] = s.observers[len(s.observers)-1] s.observers[len(s.observers)-1] = nil // avoid memory leak s.observers = s.observers[:len(s.observers)-1] return } } } func (s *BaseSubject) Notify(event Event) { s.mu.RLock() // Snapshot: copy the slice header so iteration is safe // even if an observer calls Unsubscribe during OnNotify snapshot := make([]Observer, len(s.observers)) copy(snapshot, s.observers) s.mu.RUnlock() for _, obs := range snapshot { obs.OnNotify(event) } }
Why swap-remove in Unsubscribe? The standard "shift left" removal (append(s[:i], s[i+1:]...)) is O(n) and allocates. Swap-with-last is O(1). Observer notification order is undefined anyway, so preserving insertion order has no semantic value.

The domain subject: PriceFeed

PriceFeed embeds BaseSubject and adds domain logic. Embedding promotes all three methods (Subscribe, Unsubscribe, Notify) to PriceFeed — it satisfies the Subject interface with zero boilerplate:

Two concrete observers

type Display struct{} func (d *Display) OnNotify(event Event) { if event.Type != "price_updated" { return } update := event.Data.(PriceUpdate) fmt.Printf("[DISPLAY] %s: $%.2f\n", update.Symbol, update.Price) } type AlertEngine struct { Threshold float64 } func (a *AlertEngine) OnNotify(event Event) { if event.Type != "price_updated" { return } update := event.Data.(PriceUpdate) if update.Price > a.Threshold { fmt.Printf("[ALERT] %s exceeded $%.2f! Current: $%.2f\n", update.Symbol, a.Threshold, update.Price) } }

Key observation: PriceFeed never imports Display or AlertEngine. It only knows the Observer interface. You can add a KafkaWriter observer, a MetricsCollector observer, or a test mock — without touching PriceFeed.

Three notification strategies

The walked example used push — the subject sends the full event data to every observer. This is one of three common strategies, each with different coupling and performance characteristics.

Strategy What's passed Observer action Coupling Best when
Push Full event data Use the data directly Observer coupled to event struct All observers need the same data; events are cheap to construct
Pull Signal only (or minimal hint) Query the subject for what it needs Observer coupled to subject's query API Observers need different slices of the data; data is expensive to compute
Channel Event sent to a chan Reads from its own channel Decoupled by the channel buffer Async notification; observers run on their own goroutines

Pull variant in Go

The observer receives a reference to the subject instead of event data. It then queries exactly what it needs:

// Pull-style: observer gets the subject, not the data. type PullObserver interface { OnChange(source any) // the subject itself } // The observer queries what it needs: func (d *Display) OnChange(source any) { feed := source.(*PriceFeed) price := feed.LatestPrice("AAPL") // pull only what we need fmt.Printf("[DISPLAY] AAPL: $%.2f\n", price) }

Trade-off: the observer now depends on the subject's query API (LatestPrice), which is tighter coupling than depending on an event struct. But it avoids sending data that most observers don't need — useful when the subject holds a large, complex state and different observers need different slices of it.

Channel variant in Go

This is the most Go-idiomatic variant. Instead of implementing an interface, each "observer" is a goroutine reading from a channel. The subject writes to all registered channels:

type ChanSubject struct { mu sync.RWMutex subs map[string]chan Event // keyed by subscriber ID } func (s *ChanSubject) Subscribe(id string, bufSize int) chan Event { s.mu.Lock() defer s.mu.Unlock() ch := make(chan Event, bufSize) s.subs[id] = ch return ch } func (s *ChanSubject) Unsubscribe(id string) { s.mu.Lock() defer s.mu.Unlock() if ch, ok := s.subs[id]; ok { close(ch) // signals the reader goroutine to exit delete(s.subs, id) } } func (s *ChanSubject) Notify(event Event) { s.mu.RLock() defer s.mu.RUnlock() for _, ch := range s.subs { select { case ch <- event: default: // Buffer full — drop or log. Never block the publisher. } } }
Channel observer — goroutine per subscriber
Notify()
event → ch_display (buffered)
event → ch_alert (buffered)
goroutine A
←
for event := range ch_display { ... }
goroutine B
←

Notify returns immediately after writing to all channel buffers. Each observer processes events on its own goroutine at its own pace. close(ch) in Unsubscribe causes range ch to exit cleanly — no goroutine leak.

When to use which

Use case Best fit Why
Game engine ECS events Push (sync) Tight frame budget; observers must complete before the frame advances. No allocation per event.
Config reload notification Pull Each service needs different config keys. Push the full config struct to everyone is wasteful.
Microservice event bus Channel (async) Publishers must not block. Subscribers process at their own rate. Backpressure via buffer size.
UI state binding Push (sync) State change → immediate re-render. Must be synchronous or you get flicker.
The channel variant is really pub/sub in disguise. Once you key subscribers by topic string instead of just ID, and route events to matching channels, you've built the core of an event bus. That's the bridge to your event system.

Unsubscribe during notification

An observer's OnNotify calls Unsubscribe(self). If you're iterating over the same slice you're modifying, you get either a skipped observer or a panic. This is the single most common Observer pattern bug.

The corrupted iteration problem
observers = [A, B, C]
i=0: call A.OnNotify()
→ A calls Unsubscribe(A)
→ swap-remove: observers becomes [C, B]
i=1: call observers[1] = B.OnNotify() ✓
i=2: out of bounds — len is now 2 ✗ panic or skip C

The fix is what our BaseSubject already does: snapshot the observer slice before iterating. The snapshot is a separate slice header pointing to the same underlying array at the moment of copy. Even if the original slice is modified during iteration, the snapshot remains consistent.

// Safe: iterate over a snapshot snapshot := make([]Observer, len(s.observers)) copy(snapshot, s.observers) for _, obs := range snapshot { obs.OnNotify(event) // can call Unsubscribe — doesn't affect snapshot }

Cost: one allocation per Notify call (make allocates a new slice). For hot paths (game loops), you can pre-allocate a reusable snapshot buffer — but for most systems, this allocation is negligible.

Memory leaks: the dangling observer

If an observer is subscribed to a subject and the observer goes out of scope without unsubscribing, the subject's slice still holds a reference to it. The garbage collector can't reclaim it. This is Go's version of the classic C++ dangling observer problem.

Leaked observer
feed.observers = [display, alert]
alert = nil // "drops" the reference
But feed.observers[1] still points to the AlertEngine.
GC cannot free it. OnNotify keeps getting called.

Prevention patterns:

Pattern Mechanism Tradeoff
Explicit unsubscribe Caller is responsible: defer feed.Unsubscribe(alert) Simple. Easy to forget.
Return a cancel func cancel := feed.Subscribe(alert) — call cancel() to remove Familiar from context.WithCancel. Hard to accidentally hold a stale subscription.
Weak references Not natively supported in Go. Possible with runtime.SetFinalizer but fragile. Avoid. Go's GC doesn't guarantee finalizer timing.
Channel + close Subject closes the channel → goroutine exits → GC cleans up Natural in the channel variant. Doesn't apply to interface-based Observer.
The cancel func pattern is the strongest fit for Go because it mirrors how the standard library handles cleanup: ctx, cancel := context.WithCancel(parent). Your event system can use the same idiom.

The cancel func variant

func (s *BaseSubject) Subscribe(obs Observer) func() { s.mu.Lock() defer s.mu.Unlock() for _, existing := range s.observers { if existing == obs { return func() {} // already subscribed — no-op cancel } } s.observers = append(s.observers, obs) // Return a cancel function that captures 'obs' in a closure var once sync.Once return func() { once.Do(func() { s.Unsubscribe(obs) }) } } // Usage: cancel := feed.Subscribe(display) defer cancel() // guaranteed cleanup

sync.Once makes the cancel function idempotent — calling it multiple times is safe.

Observer panics

If one observer's OnNotify panics, the remaining observers in the iteration never get called. In the synchronous variant, the panic propagates up to the caller of Notify.

// Resilient notify: recover per observer for _, obs := range snapshot { func() { defer func() { if r := recover(); r != nil { log.Printf("observer panic: %v", r) } }() obs.OnNotify(event) }() }

This wraps each observer call in a deferred recover. One bad observer can't take down the whole notification chain. The cost is a closure allocation per observer per notify — acceptable in most systems, too expensive for game-loop hot paths.

Ordering guarantees

The Observer pattern intentionally provides no ordering guarantees. Our swap-remove in Unsubscribe further breaks insertion order. If your system requires ordered notification, you have two clean options:

Option Mechanism When to use
Priority field Store observers as []struct{ obs Observer; priority int }, sort on subscribe Small observer count, rare subscription changes
Ordered phases Multiple subjects for different phases: PreUpdate, Update, PostUpdate Game engine frame lifecycle — subscribe to the phase you need

Observer → Event Bus: the generalization

The Observer pattern as implemented above has a structural limitation: each subject manages its own subscriber list. If you have 20 types of events, you need 20 subjects — or one subject where every observer receives every event and filters internally.

An event bus (or event system) centralizes this. Instead of subscribing to a specific subject, observers subscribe to event types on a shared bus. The bus routes events to the right subscribers.

Minimal event bus skeleton

This is the bridge between the Observer pattern above and a full event system. It uses the same mechanics — just centralized and keyed by event type:

type Handler func(event Event) type EventBus struct { mu sync.RWMutex handlers map[string][]Handler // event type → handlers } func NewEventBus() *EventBus { return &EventBus{handlers: make(map[string][]Handler)} } func (bus *EventBus) On(eventType string, h Handler) func() { bus.mu.Lock() defer bus.mu.Unlock() bus.handlers[eventType] = append(bus.handlers[eventType], h) // Return cancel func (same pattern as Observer) idx := len(bus.handlers[eventType]) - 1 var once sync.Once return func() { once.Do(func() { bus.off(eventType, idx) }) } } func (bus *EventBus) Emit(event Event) { bus.mu.RLock() handlers := make([]Handler, len(bus.handlers[event.Type])) copy(handlers, bus.handlers[event.Type]) bus.mu.RUnlock() for _, h := range handlers { h(event) } }
Notice what changed: subscribers no longer implement an interface — they register plain functions (Handler). The bus uses On/Emit vocabulary instead of Subscribe/Notify. And publishers call bus.Emit() instead of self.Notify() — the publisher doesn't even hold the subscriber list anymore.

Type-safe events with generics

The any payload and string-keyed routing work but push type checking to runtime. With Go generics, you can make the bus type-safe per event:

// A typed subscription: the handler receives the concrete type, not any. func On[T any](bus *EventBus, eventType string, handler func(T)) func() { return bus.On(eventType, func(e Event) { if data, ok := e.Data.(T); ok { handler(data) } }) } // Usage: handler receives PriceUpdate directly, no type assertion needed. cancel := On[PriceUpdate](bus, "price_updated", func(p PriceUpdate) { fmt.Printf("Price: $%.2f\n", p.Price) })

The type assertion is pushed into the On wrapper — each handler only sees its expected type. A mismatched event silently skips (you could log or panic instead, depending on your error philosophy).

Each layer adds one more degree of decoupling. Your event system can start at any layer and evolve upward as the architecture demands it.
