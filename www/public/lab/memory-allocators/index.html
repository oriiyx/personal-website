memory allocators from first principles

arena · stack · pool · game engine allocation strategies · for developers who want the full picture

Why not just use malloc?

malloc is a general-purpose allocator. It's designed to handle any allocation pattern: any size, any order, any lifetime. To do that, it maintains free lists, coalesces freed blocks, tracks metadata per allocation, and may acquire pages from the OS. Every call carries overhead you can't see: locking a mutex (thread safety), walking a free list (finding a fit), splitting or coalescing blocks (maintaining the heap), and sometimes calling mmap/VirtualAlloc (kernel transition).

For a web server that allocates a buffer every few milliseconds, this is fine. For a game running at 60 fps — allocating and freeing thousands of objects every 16.6ms — it's a slow death. Not because any single malloc is catastrophically slow, but because the aggregate cost compounds per frame, the heap fragments over time, and your data ends up scattered across memory in cache-hostile patterns.

The fundamental issue: malloc knows nothing about your program's allocation patterns. It has to assume the worst case — arbitrary sizes, arbitrary lifetimes, arbitrary free order. If you do know your patterns (and in a game engine, you almost always do), you can build allocators that are orders of magnitude faster by exploiting that knowledge.

What a game frame actually looks like

A single frame in a game engine produces several distinct allocation patterns. Understanding these is the key to choosing the right allocator.

The insight

If you know the lifetime, you know the allocator. Frame-scoped data → arena (reset every frame). Persistent data → arena (never freed until level unload). Fixed-size churn → pool. This is the core principle: you trade generality for speed by encoding your knowledge of allocation patterns directly into the allocation strategy.

Lifetime-based thinking

General-purpose allocators track each allocation individually — every malloc must be paired with a free, and the allocator must handle them in any order. Custom allocators work differently: you group allocations by lifetime, then free the entire group at once.

This is a categorical shift. You stop thinking "I need to free this object" and start thinking "everything in this arena was allocated for this frame, so I reset the arena at frame end." No per-object tracking. No free lists. No fragmentation. The lifetime is the deallocation strategy.

Alignment

CPUs don't read memory byte-by-byte — they read in chunks (4 bytes, 8 bytes, 16 bytes depending on the architecture and the data type). If a 4-byte integer starts at address 0x1003 instead of 0x1004, the CPU might need two reads instead of one (or on some architectures, fault entirely). This is alignment: the requirement that an N-byte value's address is a multiple of N (or some power of two).

Fragmentation: external vs internal

External fragmentation

Free memory exists, but it's split into chunks too small to satisfy a request. Caused by variable-size allocations freed in unpredictable order. This is the heap diagram from the previous tab.

Arena allocators eliminate this entirely — they never free individual objects, so holes can't form.

Internal fragmentation

An allocation uses more memory than the object needs. Caused by alignment padding, block-size rounding, or fixed-size pool slots bigger than the actual object.

Pool allocators trade this intentionally — you waste a few bytes per slot to gain O(1) alloc/free with zero external fragmentation.

The four allocator types

Allocator Alloc Free Best for Tradeoff
Arena (bump) O(1) — pointer bump O(1) — reset entire arena Frame-scoped, phase-scoped, load-time Can't free individual allocations
Stack O(1) — pointer bump O(1) — pop to saved position LIFO scopes: function scratch, recursive traversals Must free in reverse order
Pool O(1) — pop free list O(1) — push free list Fixed-size objects: entities, particles, nodes All slots same size; wastes space for smaller objects
General (malloc) O(n) — search free list O(n) — coalesce + merge Unpredictable sizes and lifetimes Slow, fragments, per-alloc metadata overhead
Notice the pattern: every custom allocator restricts what you can do (no individual free, LIFO-only free, fixed-size only) in exchange for making what you can do extremely fast and fragmentation-free. The engineering task is matching each restriction to a subsystem where it's naturally satisfied.

The simplest allocator that's actually useful

An arena allocator owns a large block of memory and maintains a single pointer: where the next allocation starts. To allocate, you bump that pointer forward by the requested size (plus alignment). To free, you reset the pointer to the beginning. That's it — no headers, no free lists, no bookkeeping per allocation.

typedef struct { uint8_t *buf; // backing memory (start of the block) size_t cap; // total capacity in bytes size_t offset; // current allocation point (the "bump pointer") } Arena;

Three fields. That's the entire allocator state. Contrast with malloc's implementation, which maintains thread-local caches, size-class bins, free trees, and per-allocation headers.

The alloc function

void *arena_alloc(Arena *a, size_t size, size_t align) { // Round offset up to required alignment size_t aligned = (a->offset + align - 1) & ~(align - 1); // Check if there's room if (aligned + size > a->cap) return NULL; // out of memory void *ptr = a->buf + aligned; a->offset = aligned + size; return ptr; }

The alignment formula (offset + align - 1) & ~(align - 1) rounds up to the next multiple of align. We'll break this down in the Implementation tab. The key point here: the entire function is a round-up, a comparison, an addition, and a pointer return. No branching except the bounds check.

Frame arena pattern

The most common use in game engines: allocate a large arena at startup (e.g. 8 MB), and use it for all per-frame scratch data. At the end of each frame, reset it. Every frame starts with a clean, unfragmented arena at zero cost.

// Game loop while (running) { arena_reset(&frame_arena); // one assignment — frees everything CollisionList *hits = arena_alloc(&frame_arena, ...); DrawCmd *cmds = arena_alloc(&frame_arena, ...); char *msg = arena_alloc(&frame_arena, ...); update(hits, cmds, msg); render(cmds); // no individual frees needed — arena_reset handles everything next frame }
This is the single most impactful pattern in game engine memory management. It eliminates per-frame allocation overhead, prevents fragmentation, and makes memory leaks structurally impossible for frame-scoped data — if you forgot to free something, the arena reset catches it anyway.

Stack allocator: arena with scoped rollback

A stack allocator is an arena that supports partial rollback. You save the current offset (a "marker"), allocate a batch of memory, use it, then restore the offset to the saved position. The constraint: you must free in reverse order of allocation — last in, first out.

typedef size_t StackMark; // just a saved offset StackMark stack_save(Arena *a) { return a->offset; } void stack_restore(Arena *a, StackMark m) { a->offset = m; }

That's the entire API. A stack allocator is literally an arena plus two trivial functions. The "mark" is just a saved copy of the offset.

The LIFO constraint is the key. You can't free the open_set while keeping the closed_set — you restore to a point, and everything after that point is gone. This matches naturally to function call scopes, recursive algorithms, and temporary work buffers where the allocation lifetime mirrors the call stack.

Pool allocator: O(1) alloc/free for fixed-size objects

Arenas and stacks can't free individual objects in arbitrary order. Pool allocators can — but only if every object is the same size. The trick: maintain a free list embedded directly in the unused slots.

When a slot is free, we store a pointer to the next free slot inside the slot itself (the slot's memory is unused anyway — we might as well use it for bookkeeping). This means the free list requires zero additional memory.

Alloc: pop the head of the free list (free_head = free_head→next). Free: push the returned slot onto the head (slot→next = free_head; free_head = slot). Both are O(1), both are one or two pointer operations. No searching. No coalescing. No fragmentation.

The embedded free list trick

typedef struct { uint8_t *buf; // backing memory size_t slot_size; // bytes per slot (must be ≥ sizeof(void*)) size_t count; // total number of slots void *free_head; // head of the embedded free list } Pool; void *pool_alloc(Pool *p) { if (!p->free_head) return NULL; // pool exhausted void *slot = p->free_head; p->free_head = *(void **)slot; // follow the embedded next pointer return slot; } void pool_free(Pool *p, void *slot) { *(void **)slot = p->free_head; // write current head into the slot p->free_head = slot; // slot becomes the new head }

The critical requirement: slot_size must be at least sizeof(void*) (8 bytes on 64-bit). If your objects are smaller than a pointer, pad the slot size up. This is a small price — a particle with 6 bytes of data wastes 2 bytes per slot to hold the free-list pointer when that slot is free.

The allocator hierarchy

A real engine doesn't use just one allocator — it uses a tree of allocators, each carved from a parent. At the root: one large allocation from the OS (via mmap / VirtualAlloc). Everything else is sub-allocated from that block.

Typical engine allocator layout
OS allocation (e.g. 256 MB via VirtualAlloc)
├── Frame Arena (8 MB) — reset every frame
├── collision scratch
├── debug draw commands
├── temp strings / formatting
└── visibility query results
├── Level Arena (64 MB) — reset on level change
├── mesh data
├── textures
├── audio buffers
└── navigation meshes
├── Entity Pool (16 MB) — fixed-size slots, individual alloc/free
├── Particle Pool (4 MB) — same pattern, smaller slots
├── Scratch Stack (2 MB) — LIFO scopes for algorithms
└── General Heap (remaining) — fallback for rare unpredictable allocs
Each subsystem gets its own allocator. The engine knows the total memory budget upfront. There's no fragmentation between subsystems because they don't share heap space.

Why pointers break with custom allocators

When an arena resets, all pointers into it become dangling. When a pool slot is freed and reallocated, a stale pointer now points to a different entity. This is the "use-after-free by design" problem: custom allocators make pointer invalidation a normal event, not a bug.

The solution: handles instead of pointers. A handle is an index (which slot) plus a generation counter (which version of that slot).

typedef struct { uint32_t index; // which slot in the pool uint32_t generation; // incremented each time the slot is reused } Handle;

Every pool slot stores its current generation. When you free and reallocate a slot, you increment the generation. When someone dereferences a handle, you compare the handle's generation to the slot's current generation — if they don't match, the handle is stale.

Double-buffer arenas

Some data needs to survive two frames — the physics system reads last frame's collision pairs while this frame computes new ones. Instead of one frame arena, use two and alternate:

Arena frame_arenas[2]; int current = 0; while (running) { current ^= 1; // toggle 0 ↔ 1 arena_reset(&frame_arenas[current]); // reset THIS frame's arena // frame_arenas[current ^ 1] still holds last frame's data // allocate this frame's data from frame_arenas[current] // read last frame's data from frame_arenas[current ^ 1] }

Each arena is reset every other frame. At any given time, one holds current-frame data and the other holds last-frame data. Zero additional complexity — just two arenas and a toggle.

The alignment formula

This is the most common bit trick in allocator code. Given an address (or offset) and a power-of-two alignment, round up to the next aligned address:

size_t aligned = (offset + align - 1) & ~(align - 1);

Let's break it down with offset = 12, align = 8:

Alignment round-up: offset=12, align=8
align - 1 = 8 - 1 = 7 = 00000111
~(align - 1) = ~7 = 11111000 ← the alignment mask
offset = 12 = 00001100
offset + 7 = 19 = 00010011 ← push past next boundary
00010011 & 11111000 = 00010000 = 16 ← snap down
Adding align - 1 ensures that if the offset is already aligned, it stays put (12 + 7 = 19, which rounds down to 16 — wait, 12 isn't aligned to 8 so this is correct). If offset were 16: 16 + 7 = 23, AND with mask = 16. Already aligned → no change. ✓

The mask ~(align - 1) clears the low bits, effectively rounding down to the nearest multiple of align. The add before the mask makes it round up.
This only works for power-of-two alignments. Alignment values like 3, 5, or 7 will produce wrong results. In practice, all natural CPU alignments (1, 2, 4, 8, 16, 32) are powers of two, so this isn't a real constraint — but if you're writing a debug build, assert (align & (align - 1)) == 0.

Backing memory: virtual vs physical

Where does the arena's backing buffer come from? Two approaches:

Physical allocation (malloc a big block)

Simple: arena.buf = malloc(8 * 1024 * 1024). The OS commits physical pages immediately (or on first touch). You pay for the full 8 MB even if you only use 200 KB this frame.

Good enough for small arenas. Not ideal for large ones where actual usage varies wildly between frames.

Virtual memory reservation (mmap / VirtualAlloc)

Reserve a huge range (e.g. 1 GB) but commit pages only as the bump pointer reaches them. The OS maps virtual addresses but doesn't allocate physical RAM until you touch each page.

You get a contiguous address range (no realloc ever needed) and only pay for pages you actually use. This is the production approach in serious engines.

// Linux: reserve 1 GB, no physical pages yet void *base = mmap(NULL, 1ULL << 30, PROT_READ | PROT_WRITE, MAP_PRIVATE | MAP_ANONYMOUS | MAP_NORESERVE, // MAP_NORESERVE = don't commit -1, 0); // Windows equivalent: void *base = VirtualAlloc(NULL, 1ULL << 30, MEM_RESERVE, // reserve only, no physical pages PAGE_READWRITE); // Later, commit pages as needed (e.g. when bump pointer crosses a page boundary): VirtualAlloc(page_addr, page_size, MEM_COMMIT, PAGE_READWRITE);

Pool initialization

When you create a pool, you need to chain all slots into the initial free list. This is a one-time O(n) setup cost:

void pool_init(Pool *p, void *buf, size_t slot_size, size_t count) { p->buf = buf; p->slot_size = slot_size; p->count = count; // Chain all slots into the free list for (size_t i = 0; i < count - 1; i++) { void *slot = (uint8_t *)buf + i * slot_size; void *next = (uint8_t *)buf + (i + 1) * slot_size; *(void **)slot = next; // embed next pointer in slot } // Last slot points to NULL *(void **)((uint8_t *)buf + (count - 1) * slot_size) = NULL; p->free_head = buf; // first slot is the head }

After this, the pool's backing memory contains a linked list threaded through the slots themselves. No separate allocation for the free list.

Debug features worth building in

Feature What it does Cost
High-water mark Track the maximum offset the arena ever reached. Tells you if your arena is oversized or nearly overflowing. One max() per alloc
Allocation count Count how many allocs happened per frame. Spike detection for unexpected allocation storms. One increment per alloc
Fill on reset Fill freed memory with 0xCD on reset. Makes use-after-free immediately visible in the debugger (values look obviously wrong). One memset per reset — debug builds only
Guard bytes Write a magic value (e.g. 0xDEADBEEF) after each allocation. On reset, check that the guard is intact — if not, something overwrote its bounds. 4–8 bytes per alloc + check on reset

Ship with the debug features behind a compile flag. #ifdef DEBUG_ALLOCATORS around the fill/guard/count logic. The release build has zero overhead; the debug build catches memory corruption in real time. This is standard practice — every serious engine has allocator instrumentation that compiles out in release.
