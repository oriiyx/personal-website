the game loop & frame timing

fixed vs variable timestep · the accumulator pattern · update/render separation · the spine of every engine

What is a game loop?

Every real-time application — games, simulations, audio engines — has the same core structure: an infinite loop that repeatedly reads input, advances the simulation, and presents a frame. This loop runs 30, 60, 144, or more times per second. It's the heartbeat of the engine, and every subsystem's execution model is a consequence of how you design it.

The simplest possible loop looks like this:

while (running) { poll_input(); update(); render(); }

This works — barely. It runs as fast as the CPU and GPU allow. On a fast machine, the game runs faster. On a slow machine, it runs slower. Physics, animations, AI, movement speeds — everything is tied to how quickly the hardware can churn through the loop. This is how games worked in the DOS era, and it's why old DOS games run at absurd speeds on modern hardware.

The fundamental tension

There are three competing demands, and no loop design satisfies all of them perfectly:

Determinism means a physics simulation produces the same trajectory whether the machine runs at 30fps or 300fps. Without it, gameplay feels different on different hardware, replays don't work, and networked physics diverges.

Smoothness means rendering as many frames as possible (or exactly at the display's refresh rate) to eliminate stutters and judder.

Responsiveness means the delay between pressing a button and seeing the result on screen is as low as possible.

The naive loop above gives you responsiveness (input is polled every frame) and smoothness (renders as fast as it can), but sacrifices determinism entirely. Every timestep model is a different tradeoff between these three.

Why frame time varies

Even on the same machine, consecutive frames don't take the same amount of time. One frame might process 200 entities; the next, a particle explosion spawns 4000. The GPU might stall waiting for a texture upload. The OS might context-switch to another process. VSync might hold the frame for display alignment.

The entire discipline of game loop design is about deciding: given that frame times vary, how do you advance the simulation?

Model 1: Variable timestep (the naive approach)

Measure how long the last frame took (dt), and multiply all movement/physics by it. If the frame took twice as long, move everything twice as far. In theory this makes the simulation frame-rate independent.

while (running) { double dt = time_since_last_frame(); poll_input(); update(dt); // velocity * dt, gravity * dt, etc. render(); }

This is what most tutorials teach first. It's simple. It's also broken for anything beyond trivial cases.

Why variable timestep breaks physics. Consider a ball launched upward with velocity.y = 100 and gravity = -200. With Euler integration: velocity.y += gravity * dt; position.y += velocity.y * dt. At 60fps (dt=0.0167), the ball reaches a peak height of ~25.0 units. At 30fps (dt=0.0333), the peak is ~24.2 units. At 15fps (dt=0.0667), it's ~22.2 units. The trajectory changes because Euler integration is a first-order approximation — larger time steps accumulate more error. A player on a 30fps laptop experiences different jump arcs than one on a 144Hz monitor. Spring systems overshoot. Collisions tunnel through thin walls because objects teleport past them in a single large step.

You can mitigate this with higher-order integrators (RK4, Verlet), but you can't eliminate it. And you still can't do deterministic replays, since frame times will differ on every run.

Model 2: Fixed timestep (the rigid approach)

Always advance the simulation by the same dt — say, exactly 1/60th of a second. The simulation is perfectly deterministic: same inputs, same results, every time. Replays work. Networked physics syncs.

while (running) { poll_input(); update(1.0/60.0); // always exactly 16.67ms render(); }

The problem: what if update + render takes less than 16.67ms? The game runs too fast — you're cranking through simulation time faster than real time. What if it takes more? The game slows down — you fall behind real time and everything moves in slow motion.

A pure fixed timestep only works if you can guarantee your frame budget is exactly your timestep. In practice, you can't.

Model 3: Semi-fixed timestep (the accumulator pattern)

The industry-standard solution. Measure real elapsed time, but consume it in fixed-size bites. You run as many fixed-step physics updates as needed to "catch up" to real time, then render once.

double accumulator = 0.0; const double FIXED_DT = 1.0 / 60.0; // 16.67ms while (running) { double frame_time = time_since_last_frame(); accumulator += frame_time; poll_input(); while (accumulator >= FIXED_DT) { // consume in fixed bites update(FIXED_DT); accumulator -= FIXED_DT; } render(); }

This gives you fixed-timestep determinism for physics, while running the render loop at whatever rate the hardware allows. The next tab walks through exactly how the accumulator behaves frame by frame.

The interpolation problem

There's a subtle visual issue. After the physics loop drains, the accumulator still has a remainder — leftover time that's less than one FIXED_DT. When you render, you're showing the physics state from the last completed tick, not from "right now." The rendered frame is always slightly behind real time.

For most games this is imperceptible (the remainder is at most 16.67ms behind). But for buttery-smooth high-refresh-rate rendering, you can interpolate between the last two physics states using the remainder as a blend factor:

double alpha = accumulator / FIXED_DT; // 0.0 to 1.0 // Blend between previous and current physics state for rendering Vec3 render_pos = lerp(prev_position, curr_position, alpha); draw(render_pos);
Interpolation between physics states
Physics tick N ────── ◆ render point (alpha=0.6) ──── Physics tick N+1
render_pos = prev_pos + (curr_pos − prev_pos) × 0.6
This means you need to store two copies of every renderable physics state: the previous tick and the current tick. The render phase blends between them. This adds memory overhead and complexity — it's why many games skip interpolation entirely and accept the sub-frame latency.
Extrapolation vs. interpolation. Some engines extrapolate forward from the current state instead of interpolating between two known states. This avoids the one-tick visual delay but introduces prediction errors — objects overshoot and snap back when the next physics tick corrects them. Interpolation is the safer default; extrapolation is used in networked games where latency hiding matters more than visual precision.

The spiral of death

What happens if a frame takes 200ms? The accumulator fills up with 200ms of debt. The inner loop runs 12 physics updates to catch up. But running 12 updates takes even longer than running 1, so the next frame takes even longer, accumulating even more debt. This is the spiral of death — the simulation can never catch up to real time.

The standard fix is a frame time clamp:

double frame_time = time_since_last_frame(); if (frame_time > 0.25) // clamp to 250ms max frame_time = 0.25; accumulator += frame_time;

With a 250ms clamp, the maximum number of physics updates per frame is ceil(250 / 16.67) = 15. Beyond that, the simulation runs in slow motion rather than trying to catch up — a better outcome than an infinite loop. The exact clamp value is a design decision: too low and the game slows down too easily; too high and a single bad frame causes a cascade of updates.

VSync and the display pipeline

Monitors refresh at a fixed rate (e.g. 60Hz). Without synchronization, the GPU might push a new frame to the display mid-refresh, causing a horizontal tear — the top half shows the old frame and the bottom half shows the new one.

VSync prevents this by making the GPU wait for the monitor's vertical blank interval before swapping buffers. The tradeoff: if you miss the deadline by even 1ms, you wait an entire refresh cycle (~16.7ms at 60Hz), effectively halving your framerate to 30fps for that frame. This causes a visible stutter.

Adaptive sync (G-Sync, FreeSync) solves this by letting the monitor match its refresh rate to the GPU's actual output rate. The display waits for the frame instead of the frame waiting for the display. This eliminates both tearing and VSync stutter, but only works on supported hardware.

What to do with leftover time

If your frame finishes in 11ms at a 60fps target, you have ~5.7ms to spare. Three strategies:

Strategy How Pro Con
Busy-wait (spin) Loop checking the clock until the deadline Most precise timing; lowest latency to next frame Burns 100% CPU; wastes power; bad for laptops and thermals
Sleep sleep(remaining - margin) CPU can idle; power-friendly OS sleep granularity is 1–15ms; you might oversleep and miss the deadline
Hybrid Sleep for most of the gap, busy-wait the last 1–2ms Low power + precise timing More code; need to tune the margin per platform

The hybrid approach is what most shipping engines use. On Windows, you also need timeBeginPeriod(1) to get 1ms sleep resolution instead of the default 15.6ms — without it, Sleep(1) actually sleeps for 15ms.

Input and the physics loop

A common design question: should input be polled inside or outside the physics inner loop?

Poll outside (standard)

Input is captured once per frame. All physics ticks in that frame see the same input snapshot. Simpler, lower latency to screen.

poll_input(); while (acc >= FIXED_DT) { update(FIXED_DT, input); acc -= FIXED_DT; } render();

Poll inside (for fighting games)

Input is re-polled before every physics tick. Lets you capture mid-frame input changes. Used when frame-perfect input matters (fighting games, rhythm games).

while (acc >= FIXED_DT) { poll_input(); update(FIXED_DT, input); acc -= FIXED_DT; } render();

For most games, polling outside is the right default. If you need frame-perfect input, polling inside gives you sub-frame resolution at the cost of slightly more input processing overhead.

Threading implications

Your loop design constrains your threading model. The single-threaded accumulator loop is the baseline. More complex engines thread subsystems off the main loop, but the dependencies create a DAG (directed acyclic graph) of work that must be respected:

Common threading splits
Single-threaded — everything sequential on the main thread. Simplest. Where you should start.
Render on separate thread — main thread does input+physics+gameplay, render thread consumes a snapshot. Requires double-buffering all renderable state. Classic "game thread / render thread" split.
Job system — the loop kicks off parallel jobs (physics broadphase, animation, culling) and synchronizes at barriers. Most complex but scales to many cores. This is where Unity DOTS and Unreal's task graph live.
The rule: don't thread until you've profiled and found a bottleneck. A well-structured single-threaded loop on modern hardware handles an enormous amount of work. Threading introduces synchronization bugs, data races, and dramatically increases debugging difficulty. Start single-threaded, profile, then thread the specific bottleneck.

Timer precision

The entire accumulator pattern depends on accurately measuring elapsed time. A 1ms error in a 16.7ms frame is a 6% timing error — enough to cause visible stutters. Platform-specific high-resolution timers:

Platform API Resolution Notes
Windows QueryPerformanceCounter ~100ns Always use with QueryPerformanceFrequency. Don't use GetTickCount (15ms resolution).
Linux / macOS clock_gettime(CLOCK_MONOTONIC) ~1ns Monotonic means it never jumps backward (unlike wall-clock time during NTP adjustments).
C# / Unity Stopwatch.GetTimestamp() ~100ns Uses QPC on Windows. DateTime.Now is not suitable — it has ~15ms resolution and isn't monotonic.
Cross-platform C SDL_GetPerformanceCounter Platform-dependent Wraps the best available timer per OS.
Never use time() or gettimeofday() for frame timing. Both return wall-clock time, which jumps during NTP corrections, daylight saving transitions, or manual clock changes. A monotonic clock only ever moves forward, which is what the accumulator requires.

Use doubles, not floats

A 32-bit float has ~7 decimal digits of precision. After the game runs for 4+ hours, a float storing "seconds since start" loses sub-millisecond precision — 14400.0 + 0.0001 rounds to 14400.0. Your delta time calculations become noisy, causing physics jitter.

Always store time values as double (15–16 significant digits). Or store time as a 64-bit integer in nanoseconds or microseconds and convert to a float dt only at the last moment. Unity's Time.deltaTime is a float, which is one reason long Unity play sessions can develop subtle timing artifacts.

First frame handling

The very first call to time_since_last_frame() will return garbage — either zero (no previous frame) or a huge value (time since process start, including asset loading). Always initialize the "previous time" timestamp immediately before entering the loop, and clamp the first dt:

double prev_time = get_time(); // ← capture RIGHT before the loop double accumulator = 0.0; while (running) { double now = get_time(); double frame_time = now - prev_time; prev_time = now; // Clamp: prevents spiral of death AND handles first-frame spike if (frame_time > 0.25) frame_time = 0.25; accumulator += frame_time; // ... rest of loop }

Choosing your physics rate

The physics tick rate (1 / FIXED_DT) doesn't have to match the display refresh rate. Common choices:

Rate FIXED_DT Use case
30 Hz 33.3ms Slow simulations, RTS games, turn-based with physics garnish
60 Hz 16.67ms Standard for most games. Good balance of accuracy and performance.
120 Hz 8.33ms Fast action games, VR, where collision precision matters.
240 Hz 4.17ms Fighting games, competitive shooters. Expensive — 4× the physics work of 60Hz.

Higher rates = more stable physics (smaller steps = less integration error) but more CPU cost. If your rendering runs at 144fps but physics at 60Hz, you'll get ~2.4 physics ticks per rendered frame on average, with interpolation smoothing the visual result.

Complete reference loop

const double FIXED_DT = 1.0 / 60.0; const double MAX_FRAME_TIME = 0.25; double prev_time = get_time(); double accumulator = 0.0; State previous_state, current_state; while (running) { double now = get_time(); double frame_time = min(now - prev_time, MAX_FRAME_TIME); prev_time = now; accumulator += frame_time; poll_input(); while (accumulator >= FIXED_DT) { previous_state = current_state; physics_update(¤t_state, FIXED_DT); accumulator -= FIXED_DT; } double alpha = accumulator / FIXED_DT; State render_state = lerp(previous_state, current_state, alpha); render(render_state); swap_buffers(); }
This is the "Fix Your Timestep" loop from Glenn Fiedler's canonical article. All four pieces are present: frame time measurement with clamp, fixed-step physics with accumulator drain, interpolation for smooth rendering, and decoupled render submission.
