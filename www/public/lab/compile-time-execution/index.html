compile-time code execution

the problem space · jai · zig · c/c++ · why metaprogramming matters for engines

What "compile-time execution" means

The idea is simple: run arbitrary code during compilation, using the same language you're already writing in, and fold the results into the final binary. Not a separate macro language. Not a template system. Not an external code generator. The same functions you call at runtime, but executed by the compiler.

This is different from what most compilers already do (constant folding, dead code elimination). Those are optimizations the compiler performs silently. Compile-time execution is programmer-directed: you explicitly say "evaluate this now, at compile time" and use the result as if it were a literal.

Why game engines need this

Game engines hit a specific class of problems that are all variations of the same underlying issue: code that needs to know about your types. C and C++ give you almost no clean way to do this, so engines accumulate layers of external tooling to compensate.

The four recurring problems
Serialization Every struct needs a save/load function. Change a field, forget to update the serializer → silent data corruption. In C you write these by hand. In C++ you abuse templates or use an external IDL (protobuf, FlatBuffers). Editor UI Click an entity in your editor, see its fields with sliders and color pickers. Requires knowing "struct Transform has a field position of type Vec3" — runtime reflection. C has none. C++ has RTTI but it's useless for this. Engines build their own with macros (UPROPERTY()) or external header parsers. SOA transforms You write struct Particle { pos, vel, life } in AOS because it's natural. For cache performance you want SOA: separate arrays per field. Maintaining both layouts by hand is duplication. C++ template approaches are unreadable. Build config C/C++ projects use CMake, Make, Ninja, or Python scripts. The build system is a separate language from the code it builds. Every project has its own dialect of build-system pain.

All four problems share the same shape: you need a program that inspects your code's structure (types, fields, dependencies) and generates more code based on what it finds. The question is where and how that program runs.

The spectrum of solutions

Where does the "type-aware code" run?
External tool
Separate language, separate build step. Protobuf, Qt MOC, Unreal Header Tool. Maximum friction. Different language, different debugger, generated code you can't easily read.
Preprocessor
C macros. Text substitution before the compiler sees the code. No type awareness, no debugging, no recursion. Error messages reference expanded code you never wrote.
Template system
C++ templates. Technically Turing-complete. In practice: a functional language accidentally embedded in a type system. Catastrophic error messages, compile time explosion, unreadable code.
Same language
Jai, Zig, (partially) D. Write normal imperative code, mark it to run at compile time. Same syntax, same debuggability, same mental model. Minimum friction.
The further down this list, the less friction between "writing code" and "writing code that generates code." Jai's bet is that the bottom row should be the default, not the exception.

The Blow thesis

"There are many features I can get rid of and it would still be the same programming language. If I got rid of full arbitrary compile-time execution, it wouldn't be the same programming language. What I mean by 'full' here is — many compilers have some limited set of expressions they'll evaluate at compile time. There's constexpr in C++ and languages like D or Rust try to formalize in order to give you more versatility. My approach is: why are you doing that? Let's do everything at compile time. And by everything, I mean everything."
— Jonathan Blow

The key distinction: Jai doesn't add compile-time execution as a feature on top of an existing language. The entire compiler is structured around it. The build system, the code generation, the type introspection — they're all the same mechanism.

C preprocessor macros

The C preprocessor operates on text before the compiler sees it. It has no concept of types, scopes, or expressions — it's string substitution with conditionals.

// "Reflection" via X-macros — the C workaround #define ENTITY_FIELDS(X) \ X(float, x) \ X(float, y) \ X(int, health) \ X(int, armor) typedef struct { #define FIELD(type, name) type name; ENTITY_FIELDS(FIELD) #undef FIELD } Entity; void serialize_entity(Entity* e, FILE* f) { #define FIELD(type, name) fwrite(&e->name, sizeof(type), 1, f); ENTITY_FIELDS(FIELD) #undef FIELD }

This works, barely. The field list is defined once and expanded into both the struct definition and the serializer. But the cost is severe: the code is unreadable, undebugable (breakpoints land on the macro expansion, not the definition), and the error messages when you get it wrong reference code you never wrote. You also can't do anything conditional — if you want to skip a field during serialization based on its type, you're out of luck without nesting more macros.

C++ template metaprogramming

C++ templates are Turing-complete by accident. They were designed for generic containers (vector<int>), but people discovered you could compute with them at compile time by exploiting template specialization as a form of pattern matching.

// Computing factorial at compile time via templates // This is "programming" in C++'s type system template<int N> struct Factorial { static constexpr int value = N * Factorial<N-1>::value; }; template<> struct Factorial<0> { static constexpr int value = 1; }; // Usage: Factorial<7>::value == 5040 // Try getting a useful error message when this goes wrong.

This is functional programming encoded in a type system that was never designed for it. The "variables" are template parameters, the "functions" are template specializations, and "recursion" is nested template instantiation. It works, but the practical consequences are devastating:

Compile times

Each template instantiation creates a new type. Heavy template use (Boost, Eigen, ranges) can push compile times to minutes or hours. One blogger reported a 19,000-line C++ project taking 45 minutes to compile.

Error messages

A misuse of std::ranges or Boost.Spirit can produce error messages that are thousands of lines long, referencing deeply nested template instantiations that have no connection to your source code.

Debuggability

You can't step through template metaprogramming. It happens in the compiler's type-checker, not in any executable. There's no printf, no breakpoints, no way to inspect intermediate results.

Readability

Template metaprograms use a completely different programming paradigm (functional, pattern-matched) from the imperative C++ they generate. Two mental models in one file.

C++11 introduced constexpr to improve the situation — allowing normal-looking functions to run at compile time. But constexpr is heavily restricted: no allocations (until C++20), no I/O, no complex control flow in early versions. It's a whitelist approach — you can do X, Y, Z at compile time but nothing else. Jai's position is: why whitelist? Just run everything.

External code generators

When macros and templates aren't enough, large C++ projects add external tools that parse header files and generate code:

Tool Engine / Project What it does Cost
UHT Unreal Engine Parses UPROPERTY() / UCLASS() macros, generates reflection data, serialization, editor bindings Separate C++ tool (~50k LOC), must run before compilation, adds minutes to build
MOC Qt Parses Q_OBJECT, signals:, slots:, generates signal/slot dispatch code Separate preprocessor, custom build integration, generated files in build dir
protoc gRPC / many Compiles .proto schema files into C++/Go/etc serialization code Separate language (protobuf IDL), separate compiler, generated code is verbose and hard to debug

Each of these is a shadow compiler — a separate program that understands some subset of your code's structure and emits more code. The generated code is typically committed to version control or placed in a build directory. It's not part of your source; it's an artifact that exists between your code and the real compiler.

The meta-problem: Every external code generator introduces a new language, a new build dependency, a new failure mode, and a class of bugs that exist only in the gap between "what you wrote" and "what the generator produced." The motivation for compile-time execution in Jai and Zig is to eliminate this entire category.

The core primitives

Jai's compile-time system is built from a small set of composable directives. Each one does one thing. Together they cover the full range from "compute a constant" to "rewrite the program's AST during compilation."

Directive What it does Scope
#run Execute a function or block at compile time. The result is folded into the binary. Any expression position. Can return values, modify globals (with #no_reset), or call the compiler API.
#insert Insert a string or Code value into the program as source code. Any statement position. The inserted code is compiled normally after insertion.
#code Capture a block of code as a first-class value of type Code. Can be passed to functions, inspected at compile time via compiler_get_nodes, or inserted with #insert.
#no_reset Preserve a global's compile-time modifications into the runtime binary. Global variables only. Without this, globals are reset to their initial values after all #run directives finish.
#compile_time Boolean that's true during compile-time execution, false at runtime. Conditional blocks. Lets the same function behave differently depending on when it runs.
#placeholder Declare that an identifier will be defined by a metaprogram, not by source code. Module level. Tells the compiler "don't error on this undefined name — it's coming."

#run — the foundation

Any function can be executed at compile time. No special annotation on the function itself — you mark the call site, not the definition. The compiler builds the function to an internal bytecode and runs it in an interpreter.

// Compute PI at compile time using the Leibniz formula. // compute_pi is a normal function — nothing special about its definition. PI :: #run compute_pi(); compute_pi :: () -> float { n := 1.0; s := 1.0; pi := 0.0; for 0..10000 { pi += 1.0 / (s * n); n += 2.0; s = -s; } return pi * 4.0; }

After compilation, PI is a compile-time constant — the float literal 3.14159... is embedded directly in the binary. No runtime computation, no initialization function. The same code could be called at runtime too; it's just a function.

Values can depend on each other. The compiler resolves dependencies automatically:

a := #run f1(); // a = 1000 b := #run f2(a); // b = 1001, depends on a f1 :: () => 1000; f2 :: (a) => a + 1;

#insert — code generation via strings

#insert takes a string and splices it into the program as source code. Combined with #run, this gives you arbitrary code generation: run a function at compile time that returns a string, then insert that string as code.

// Unrolling a loop at compile time unroll_for_loop :: (a: int, b: int, body: Code) #expand { #insert -> string { builder: String_Builder; print_to_builder(*builder, "{\n"); print_to_builder(*builder, "`it: int;\n"); for i: a..b { print_to_builder(*builder, "it = %;\n", i); print_to_builder(*builder, "#insert body;\n"); } print_to_builder(*builder, "}\n"); return builder_to_string(*builder); } } // Usage: unroll_for_loop(0, 3, #code { print("%\n", it); }); // The compiler sees, after expansion: // { it: int; it = 0; print(...); it = 1; print(...); it = 2; ... }

The #insert -> string { ... } syntax is shorthand for #insert #run (function that returns a string). The function runs at compile time, produces a string, and that string becomes source code.

#insert,scope — zero-cost callbacks

A common pattern: you want a sort function that takes a comparison, but you don't want the overhead of a function pointer. #insert,scope splices code into the caller's scope, so the inserted code can access local variables directly:

bubble_sort :: (arr: [] $T, compare_code: Code) #expand { for 0..arr.count-1 { for i: 1..arr.count-1 { a := arr[i-1]; b := arr[i]; if !(#insert,scope() compare_code) { arr[i-1], arr[i] = arr[i], arr[i-1]; } } } } // Usage — the comparison is inlined, no function pointer: bubble_sort(my_array, #code (a < b));
Key insight: #code (a < b) captures the expression as a compile-time value. #insert,scope() splices it into the sort's inner loop where a and b are local variables. The result is a fully inlined sort with a custom comparator — zero overhead compared to a hand-written loop.

#no_reset — baking data into the binary

By default, global variables modified during #run are reset to their initial values before the final binary is emitted. This prevents accidental state leaks from compile-time execution. But sometimes you want to bake computed data into the binary:

#no_reset lookup_table: [256] u8; #run { // Populate the table at compile time for i: 0..255 { lookup_table[i] = cast(u8) (i * i % 256); } } // At runtime, lookup_table is pre-populated — no initialization needed. main :: () { print("%\n", lookup_table[42]); // just a table read }

Blow has demonstrated this taken to the extreme: running an entire game during compilation and baking the resulting game state into the binary. The compiler's bytecode interpreter can execute anything the language can express — including graphics, sound, and network I/O.

Type introspection

Jai retains full static type information for every struct at both compile time and runtime. You can iterate over a struct's fields, inspect their types, offsets, and names — and use this information to generate code.

// Jai stores a table of all type information in the data segment. // You can iterate over it: for info: type_table { if info.type == .STRUCT { s := cast(*Type_Info_Struct) info; print("Struct: %\n", s.name); for member: s.members { print(" field: % (type: %)\n", member.name, member.type); } } }

This is the foundation for automatic serialization, editor GUIs, and network replication — all the problems listed in Tab 1. Instead of an external tool that parses your headers, the compiler itself knows your types and makes that knowledge available to your code.

Serialization in practice: The Jai community has built serialization modules that use type info to generate bespoke serialization code for each struct at compile time. A single serialize(*data) call handles any struct — the generated code is specialized per type, so it's as fast as hand-written serialization but requires zero manual field-by-field code.

The build system is Jai

In most languages, the build system is a separate program written in a separate language: Make, CMake, Ninja, Cargo, go build. Jai eliminates this by making the build system a compile-time Jai program. You write a build() function, mark it with #run, and it drives the compiler directly.

#import "Basic"; #import "Compiler"; build :: () { w := compiler_create_workspace(); if !w { print("Workspace creation failed.\n"); return; } options := get_build_options(w); options.output_executable_name = "my_game"; options.backend = .LLVM; set_optimization(*options, .OPTIMIZED); set_build_options(options, w); add_build_file("main.jai", w); } #run build();

No CMakeLists.txt. No Makefile. No build.gradle. It's just Jai code. The same language you write your game in is the language you configure your build with. And since it's real code, you can use conditionals, loops, read files, compute values — anything.

Workspaces

A workspace is an isolated compilation environment. You can create multiple workspaces from a single build script — each compiles independently with its own options, source files, and output. Workspaces don't share namespaces or modules.

build :: () { // Build the game game_ws := compiler_create_workspace("Game"); opts := get_build_options(game_ws); opts.output_executable_name = "game"; set_build_options(opts, game_ws); add_build_file("game_main.jai", game_ws); // Build the asset pipeline tool (different executable, different options) tool_ws := compiler_create_workspace("Asset Tool"); tool_opts := get_build_options(tool_ws); tool_opts.output_executable_name = "asset_tool"; tool_opts.backend = .X64; // fast backend, debug only set_build_options(tool_opts, tool_ws); add_build_file("tool_main.jai", tool_ws); } #run build();

One build script, two executables, different compiler settings for each. No external build system needed.

The compiler message loop

The most powerful feature of Jai's build system: you can intercept compiler messages during compilation and react to them. The compiler tells you when files are loaded, when types are checked, when phases complete — and you can inject code in response.

→

→

Message kind When it fires What you can do with it
.FILE Each source file is loaded Track which files are in the build, implement custom preprocessing
.IMPORT A module is imported (once per unique import) Log dependencies, enforce import policies
.TYPECHECKED Declarations pass type-checking Inspect type info, enforce coding rules, generate code based on types
.PHASE Compiler advances to a new phase Trigger code generation at the right moment (e.g., after all types are known)
.ERROR Compilation error occurs Custom error handling, reporting
.COMPLETE Compilation finishes Break the message loop

Real example: auto-generating main() from notes

Jai has notes — string tags on functions and structs (similar to Java annotations or C# attributes, but simpler). A metaprogram can find all functions with a given note and generate code that calls them:

// main.jai — the "user" code dog :: () { print("dog\n"); } @note banana :: () { print("banana\n"); } @note apple :: () { print("apple\n"); } @note // The metaprogram will generate main() automatically.
// build.jai — the metaprogram // 1. Intercept .TYPECHECKED messages // 2. Find all functions tagged @note // 3. Sort them alphabetically // 4. Generate a main() that calls them in order while true { message := compiler_wait_for_message(); if message.kind == .TYPECHECKED { typechecked := cast(*Message_Typechecked) message; for decl: typechecked.declarations { for note: decl.expression.notes { if equal(note.text, "note") { array_add(*functions, decl.expression.name); } } } } if message.kind == .PHASE { phase := cast(*Message_Phase) message; if phase.phase == .TYPECHECKED_ALL_WE_CAN { // All types known — generate code now sort(functions); code := "main :: () {\n"; for f: functions code = tprint("% %();\n", code, f); code = tprint("%}\n", code); add_build_string(code, workspace); } } if message.kind == .COMPLETE break; }
Output: main calls apple(), banana(), dog() — sorted. The user never wrote main. The metaprogram generated it from the @note tags.

This pattern scales to much more complex use cases: auto-registering test functions, generating dispatch tables from tagged handlers, building plugin systems where modules self-register at compile time.

Enforcing house rules

The message loop gives you programmatic access to every declaration in the program after type-checking. You can enforce arbitrary coding standards by inspecting the AST and calling compiler_report to emit custom errors:

// MISRA rule: no more than 2 levels of pointer indirection check_pointer_level :: (decl: *Code_Declaration) { type := decl.type; pointer_level := 0; while type.type == .POINTER { pointer_level += 1; p := cast(*Type_Info_Pointer) type; type = p.pointer_to; } if pointer_level > 2 { loc := make_location(decl); compiler_report("Too many levels of pointer indirection.\n", loc); } }
Not a linter — a compiler extension. This runs inside the compiler, with full type info, and produces errors indistinguishable from built-in compiler errors. It points to the exact line in your source. You can enforce any rule that can be expressed as "inspect a declaration and decide if it's valid."

Zig's philosophy: same power, deliberate restrictions

Zig has compile-time execution too — via the comptime keyword. But where Jai says "do everything, I trust you," Zig says "do everything that's safe and reproducible." The restrictions are intentional design choices, not limitations:

Capability Jai Zig
Run arbitrary code at compile time Yes — any function, any code Yes — any pure function
Types as first-class values Yes Yes — type is a value
Compile-time reflection Yes — full type info at compile and runtime Yes — @typeInfo, @TypeOf
I/O during compilation Yes — files, network, graphics, anything No — comptime has no I/O facilities
Modify the AST Yes — read/write access to program AST No — can generate types, not rewrite code
Code as data (quasiquoting) Yes — #code, #insert No — no string-to-code insertion
Compiler message loop Yes — intercept and react to compiler events No — build.zig is separate from compilation
Cross-compilation safety Not a design goal Enforced — comptime can't leak host details

What Zig comptime looks like

The core mechanic: tag a function parameter or variable as comptime, and the compiler evaluates it during compilation. Types can be returned from functions, type info can be destructured in switch statements, and generic code falls out naturally:

// Zig: generic max function — types are compile-time values fn max(comptime T: type, a: T, b: T) T { return if (a > b) a else b; } // Usage — T is inferred at compile time: const result = max(f32, 3.14, 2.71); // compiles to: return 3.14
// Zig: compile-time type construction fn Matrix(comptime T: type, comptime W: comptime_int, comptime H: comptime_int) type { return [H][W]T; } // Matrix(f32, 4, 4) evaluates to [4][4]f32 at compile time
// Zig: compile-time reflection via @typeInfo fn addSmallInts(comptime T: type, a: T, b: T) T { return switch (@typeInfo(T)) { .int => |info| if (info.bits <= 16) a + b else @compileError("ints too large"), else => @compileError("only ints accepted"), }; }

The deliberate restrictions

Zig's comptime restrictions aren't bugs — they're load-bearing design decisions:

No I/O

Comptime code can't read files, talk to databases, or make network requests. This means compilation is hermetic and reproducible: the same source code always produces the same binary, regardless of what's on disk or what server is running.

If you need I/O at build time, Zig's build.zig (a general-purpose build system) can run a separate Zig program to generate code, which is then @import-ed.

No host leakage

Comptime code can't observe properties of the machine it's compiling on. This matters for cross-compilation: if your Linux build machine is compiling for ARM, comptime can't accidentally bake in x86 assumptions.

Jai doesn't enforce this boundary. Its #run can do anything the host machine supports, which is powerful but makes cross-compilation harder to reason about.

The philosophical split: Zig's position is that the restrictions buy you cacheability (the compiler can skip re-evaluating comptime blocks if inputs haven't changed), reproducibility, and safety. Jai's position is that the programmer should have the full power and accept the responsibility. Both are coherent positions — they just optimize for different things.

What Zig can't do that Jai can

The practical difference comes down to three things Jai has that Zig deliberately doesn't:

1. String-to-code insertion. Jai's #insert takes a string and compiles it as code. This is the most flexible form of code generation — any string manipulation technique produces code. Zig has no equivalent. To generate code in Zig, you either return a type from a comptime function (type-level metaprogramming only) or use the build system to generate a .zig file and import it.

2. Compiler as API. Jai's compiler message loop lets the metaprogram intercept every declaration, inspect its AST, and inject code in response. Zig has no compiler API accessible from user code — the build system and the compiler are separate concerns.

3. Unrestricted side effects. A Jai #run can open files, make network requests, render graphics, play sound — Blow's demo of running a game during compilation is the canonical example. Zig's comptime is pure computation with no side effects.

What you gain

Problems eliminated by compile-time execution
No shadow compilers No protoc, no MOC, no UHT. Code generation happens in the same language, in the same build step, with the same debugger. No build system language No CMake, no Make, no Ninja. Build configuration is code in the language you already know. No reflection gap Type information is available at compile time (for code generation) and at runtime (for debugging, serialization). No macros, no RTTI hacks. Zero-cost abstractions Compile-time evaluated code produces the same output as hand-written specialized code. No vtables, no function pointers, no runtime dispatch for things known at compile time. One mental model The same syntax, the same semantics, the same debugging tools whether code runs at compile time or runtime. No template-metaprogramming-brain-mode required.

What you pay

The costs and risks
Compile time cost Running code during compilation takes time. Heavy metaprograms push compilation from "instant" to "noticeable." Jai mitigates this with fast compile times overall (80k lines in ~1 second), but the cost isn't zero. Debugging generated code When #insert splices a string into the program, the resulting code doesn't exist in any source file you wrote. Breakpoints, stack traces, and error messages reference generated code. Jai dumps generated strings to .build/.added_strings_w2.jai for inspection, but it's still indirection. Readability of metaprograms A program that generates programs is inherently harder to read than a program that just runs. You have to simulate the metaprogram in your head to understand what code it produces. This is true of all code generation, but making it easy and integrated means people use it more — including in places where a simpler solution would suffice. Tooling difficulty IDEs, linters, and refactoring tools need to understand not just the code you wrote but the code your metaprogram generates. An identifier might not exist in source — it's #placeholder'd and generated by a message loop. This makes static analysis substantially harder. Reproducibility (Jai-specific) If #run can read files, query databases, or depend on host environment, the same source code can produce different binaries depending on where/when it's compiled. Zig avoids this by design; Jai leaves it to programmer discipline.

The Handmade Network perspective

An important counterpoint from the Handmade community: you don't always need metaprogramming, even when it's available. The Handmade fishbowl discussion on this topic surfaced a useful heuristic from Ryan Fleury: the less you use metaprogramming, the better the solution tends to be.

The categories of "metaprogramming" worth distinguishing:

Category Example Necessity
Runtime type info Serialization, editor UI High — you genuinely need type knowledge at runtime
Parametric polymorphism Generic containers, Array(T) High — solves real code duplication
Compile-time execution Lookup tables, constant computation Medium — useful but often achievable other ways
Code introspection Custom lint rules, enforcing conventions Medium — valuable in large teams
AST manipulation / code generation Auto-generating dispatchers, ORM layers Use with caution — powerful but hard to debug

The risk is that compile-time execution becomes a hammer for every nail. A loop that iterates over struct fields at compile time and generates specialized code is elegant — but a hand-written function that does the same thing at runtime with a table might be simpler to read, debug, and maintain.

Where things stand (2026)

Language Status Compile-time model
Jai Closed beta; engine going open-source after Order of the Sinking Star ships Most powerful: #run + #insert + compiler message loop + full AST access. Unrestricted.
Zig Public, active development Powerful but restricted: comptime with no I/O, no AST manipulation, no string insertion. Hermetic by design.
Odin Public, stable Lighter touch: built-in SOA/AOS, parametric polymorphism, some introspection. Minimal metaprogramming by design.
C++ constexpr/consteval expanding each standard Whitelisted: each C++ version allows slightly more in constexpr. Still no code generation, no type iteration, no string insertion.
Rust Stable, proc macros available Procedural macros: separate crate that receives and emits token streams. Powerful but complex, separate compilation model.
The convergence: Every modern systems language is moving toward more compile-time capability. The debate isn't whether compile-time execution is useful — it's how much power to give the programmer and how many guardrails to enforce. Jai sits at the maximum-power end; Zig at the maximum-safety end; everyone else is somewhere in between.
