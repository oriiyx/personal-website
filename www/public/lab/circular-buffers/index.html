circular buffers from first principles

the ring buffer · FIFO in fixed memory · O(1) push and pop · for developers who want the full picture

Why does the ring buffer exist?

You have a producer that hands you data piece by piece — bytes from a UART, samples from a sound card, log lines from worker threads, network packets, key events. You have a consumer that processes them in order, but at its own pace. Between the two, you need a FIFO queue: first in, first out.

Now add the real-world constraint: this runs on a microcontroller with 8 KB of RAM, or inside an audio callback that must not allocate, or in a kernel module where malloc isn't an option. You need a queue in a fixed amount of memory, with no allocation, no copying, and O(1) push and pop.

The naive alternatives fall apart:

Approach Push Pop Why it fails
Growable array (push back, pop front) O(1) amortised O(n) — shift everything left Pop is linear. Audio callback dies.
Linked list O(1) O(1) Every push allocates. Terrible cache behaviour. No fixed memory bound.
Fixed array, shift on pop O(1) O(n) — memmove Same memmove problem as the first row.
Fixed array, never shift O(1) O(1) Runs out of space after N pushes, even if you've popped everything.

The last row is almost right. The missing insight: once the write cursor walks off the end of the array, wrap it back to the start. The cells we popped from are free again — reuse them. The array is no longer a line; it's a ring.

The core idea in one sentence

A circular buffer is a fixed-size array with two indices — a write position and a read position — that both wrap around modulo the array size. Pushing advances the write index; popping advances the read index. The "queue" is the segment between them.

No allocation. No copying. No shifting. Every operation touches exactly one slot and bumps one counter. Memory usage is a fixed, predictable sizeof(slot) × N.

The trade you accept: capacity is bounded. Once the ring is full, you must either refuse new writes, overwrite the oldest, or block the producer. That's a decision to make per use-case — and the interesting one.

Where you find them in real systems

Context Producer → Consumer Full-buffer policy
UART / serial I/O ISR writes, main loop reads Drop newest, raise overflow flag
Audio playback App thread writes, DMA reads Refuse; producer must wait
Network RX rings (NIC drivers) NIC DMA writes, kernel reads Drop packet, increment counter
Logging / tracing (dmesg, perf) Many writers, one reader Overwrite oldest (newest wins)
Game input events OS writes, game loop reads each frame Drop oldest or newest depending on taste
SPSC queues between threads One producer thread, one consumer Varies; lock-free variant common

The pattern shows up anywhere a fast producer hands bounded work to a slower (or differently-paced) consumer without the two needing to rendezvous on every item.

What "circular" actually means

The memory itself is still a flat, contiguous array — there's nothing circular about the bytes in RAM. What's circular is the index arithmetic: when the write pointer reaches the end, it goes to 0 instead of running off. The ring lives in your head (and in the modulo operator), not in the hardware.

This is worth internalising because it explains both the efficiency (still linear memory, still prefetchable, still cache-friendly) and the gotchas (bulk reads that cross the wrap point need two memcpy calls, not one).

The anatomy of a ring

Five pieces of state. That's the entire data structure:

Field Type Meaning
buf T * Backing storage. Fixed allocation. Static array, stack, or malloc'd once.
cap size_t Total slots in buf. Never changes after init.
head size_t Index where the next push will land. Advances on push.
tail size_t Index where the next pop will read from. Advances on pop.
count (optional) size_t Number of filled slots right now. Optional; some variants derive it from head and tail instead.

Some authors name head/tail differently — write/read, in/out, producer/consumer, front/back. The naming is a swamp; the semantics are what matters: one index names "where to put new data", the other names "where to take old data".

Watch the naming conventions in any given codebase. Linux's kfifo calls them in and out. Some textbooks swap head and tail relative to what you'll see here. When reading code, look at which index the push operation modifies — that's the "write" side, whatever it's called.

The ring, drawn as a ring

Capacity 8 ring · mid-life state
Four items stored (A, B, C, D) in slots 6, 7, 0, 1 — the queue wrapped past the end of the array. tail = 6 points at the next pop target (A). head = 2 points at the next push destination (empty). The data lives in the half-open arc from tail up to (but not including) head, going forward.

That last detail matters. The queue is [tail, head) — tail-inclusive, head-exclusive. Head always points at an empty slot (or at tail when full, as you'll see). If you've written queue code that got off-by-one, this convention is usually the one that was inconsistent.

Wrap-around, which is just modulo

The operation that makes the ring a ring:

head = (head + 1) % cap; // after a push tail = (tail + 1) % cap; // after a pop

Modulo wraps the index. If cap = 8 and head = 7, then after increment it becomes 8 % 8 = 0. That's the entire circular trick. Everything else in this page is a consequence of this one line.

When cap is a power of two, you can replace % cap with & (cap - 1) — same result, one instruction instead of a division. The Performance tab walks through why.

The state space

Given head, tail, and cap, every ring sits in exactly one of these states:

State Condition Push allowed? Pop allowed?
Empty head == tail AND count == 0 yes no
Partially full head != tail yes yes
Full head == tail AND count == cap no (or overwrite) yes

Notice that empty and full both satisfy head == tail. That collision is the single most important design problem for the whole data structure, and the next-but-one tab is entirely about resolving it.

Operation summary

push(x): if count == cap: return ERROR_FULL buf[head] = x head = (head + 1) % cap count += 1 pop(): if count == 0: return ERROR_EMPTY x = buf[tail] tail = (tail + 1) % cap count -= 1 return x peek(): // non-destructive read if count == 0: return ERROR_EMPTY return buf[tail]

Three to five instructions plus the bounds check, no loops, no allocation. This is what "amortised O(1)" looks like when it's actually just O(1).

The collision

Start with an empty ring: head = tail = 0. Now push until the ring would be full — for cap = 8, that's 8 pushes. Each push bumps head by 1, so head takes values 1, 2, 3, …, 7, and then (7 + 1) % 8 = 0. Head is back at 0. Tail never moved. head == tail.

But we said head == tail means "empty" too. The ring is now holding 8 bytes, but the two fields we defined can't tell us that; they look identical to the empty state.

The fundamental ambiguity: with just head and tail modulo cap, there are cap + 1 possible fill levels (0, 1, 2, …, cap) but only cap distinct (head, tail) pairs where head == tail (there's just one — the collision). You can't inject an extra bit of information into a structure that has none to spare.

Three classical fixes. Each one buys that extra bit somewhere.

Fix 1: sacrifice a slot

Declare the ring "full" one short of cap. Effective capacity becomes cap - 1. Full is now defined as "incrementing head would collide with tail", which is (head + 1) % cap == tail. Empty stays head == tail. The two conditions no longer overlap.

is_empty: head == tail is_full: (head + 1) % cap == tail push: if is_full: fail buf[head] = x head = (head + 1) % cap pop: if is_empty: fail x = buf[tail] tail = (tail + 1) % cap

With cap = 8, you store at most 7 items. Slot between head and tail is always "wasted" — but you've eliminated the need for a count field and kept the implementation to just two indices.

Pros: minimal state (2 fields), no count to update, classic and widely recognised in textbook code.
Cons: one slot permanently unused. Annoying when cap is small or when you wanted a power-of-2 usable size.

Fix 2: explicit count field

Keep a count variable. Empty is count == 0; full is count == cap. Head and tail are free to collide — the count tells you which collision it is.

is_empty: count == 0 is_full: count == cap push: if count == cap: fail buf[head] = x head = (head + 1) % cap count += 1 pop: if count == 0: fail x = buf[tail] tail = (tail + 1) % cap count -= 1

Pros: uses all N slots. size() is O(1) (just read count). Easy to reason about.
Cons: extra field to maintain. Concurrently, updating count atomically with head/tail is the hard part — three fields can't be written in one instruction on most ISAs.

For single-threaded code or code with explicit locking, this is the cleanest option and the one used in the Implementation tab.

Fix 3: mirror-bit / extended indices

The clever one. Let head and tail count up to 2 × cap (or just let them run in a full size_t without any modulo). When you actually dereference buf, take head % cap or head & mask to get the real slot.

Now:

is_empty: head == tail is_full: head - tail == cap // not head == tail push: if head - tail == cap: fail buf[head & mask] = x // masked only when dereferencing head += 1 // raw increment, no wrap pop: if head == tail: fail x = buf[tail & mask] tail += 1

For cap = 8, valid head values are 0 through 16 (well, any value — the difference from tail is what matters). When head gets as large as tail + 8, the ring is full. When head == tail, it's empty. No collision, because head doesn't reset.

But wait — doesn't head overflow? Yes, eventually, after 2^64 operations on a 64-bit machine. At 1 billion ops/second, that's 584 years. The overflow is harmless anyway, because subtraction in two's complement wraps cleanly: head - tail gives the correct unsigned difference even across an overflow boundary, as long as the difference itself fits in the type (which it trivially does, bounded by cap).

Pros: all N slots usable. No count field. Producer touches only head; consumer touches only tail — which is exactly what lock-free SPSC needs.
Cons: requires careful reasoning about unsigned overflow. Less familiar idiom.

This is the approach Linux's kfifo uses. It's also what most production SPSC lock-free queues do, because the head/tail separation maps cleanly onto atomic loads and stores with acquire/release semantics.

The modulo is the bottleneck

On most CPUs, integer division and modulo are among the slowest arithmetic operations. On modern x86 a 64-bit DIV is on the order of 20–40 cycles and is typically not pipelined. On small microcontrollers (Cortex-M0, AVR) there may be no hardware divider at all — modulo compiles to a software routine of dozens of instructions.

Meanwhile, the push/pop body itself is three or four instructions. The modulo dominates the cost of every single operation. Fixing it matters.

Power-of-2 capacity turns modulo into AND

If cap = 2^k, then x % cap equals x & (cap - 1). Same result, one AND instruction, always 1 cycle.

Why it works: cap = 2^k means cap in binary is a single 1-bit followed by k zeros. And cap - 1 is k ones in a row — the bit pattern that exactly masks "the lowest k bits". Those lowest k bits are the remainder modulo 2^k.

Non-power-of-2 capacities

If you genuinely need cap = 1000, you have options:

Option Tradeoff
Round up to next power of 2 (1024) +2.4% memory. Eliminates modulo entirely. Almost always the right choice.
Keep exact cap, use modulo Predictable memory. Slower increments.
Keep exact cap, use branchless conditional next = head + 1; if (next == cap) next = 0; — compiles to a conditional move on modern CPUs, often faster than DIV.

For anything performance-sensitive: pick power-of-2 and use the mask. The 2.4% memory isn't going to matter; the cycles will.

Cache behaviour

The ring is a single contiguous allocation. Sequential pushes and pops walk through it linearly. Modern CPU prefetchers recognise the pattern and pull the next cache line before you ask for it. A linked-list queue of the same length has terrible cache behaviour by comparison — each node is a cache miss.

For small element types (bytes, words, packet descriptors), expect ring operations to be bound by memory bandwidth rather than compute. This is as fast as FIFO gets.

False sharing in concurrent rings

When a ring is shared between a producer thread and a consumer thread (the SPSC case), head is written only by the producer and tail only by the consumer. That's the whole reason lock-free SPSC works.

But if head and tail sit next to each other in memory, they end up on the same 64-byte cache line. Now every write by the producer invalidates the consumer's cached copy of tail (and vice versa), forcing a cache-coherence round-trip on every single operation. This is false sharing, and it can slow a "lock-free" queue to below what a plain mutex would give you.

The fix is mechanical: pad the struct so head and tail live on different cache lines.

typedef struct { uint8_t *buf; size_t cap; size_t mask; alignas(64) size_t head; // producer owns this cache line char pad1[64 - sizeof(size_t)]; alignas(64) size_t tail; // consumer owns this cache line char pad2[64 - sizeof(size_t)]; } spsc_ring_t;

Measure before padding — for single-threaded rings this is pure bloat. But once multiple cores are involved, padding is the difference between "sharing nothing" and "fighting on every operation".

Bulk transfers and the wrap

Pushing one byte is easy. Pushing a 500-byte chunk into a ring that has 300 bytes free before the end of the array and another 200 after — that's two memcpy calls, not one. The split is at the wrap.

// bulk push of `n` bytes starting at `src`, returns bytes actually pushed size_t ring_push_bulk(ring_t *r, const uint8_t *src, size_t n) { size_t free = r->cap - r->count; if (n > free) n = free; // truncate to available space size_t first = r->cap - r->head; // bytes until end of array if (first > n) first = n; memcpy(r->buf + r->head, src, first); // segment 1 memcpy(r->buf, src + first, n - first); // segment 2 (0 bytes if no wrap) r->head = (r->head + n) & r->mask; r->count += n; return n; }

Note the second memcpy is safe even when n - first == 0 — the C library handles zero-length copies. The split is where all the complexity of bulk I/O on a ring lives.

A complete, usable C implementation

A byte-oriented ring using the explicit count variant with power-of-2 capacity for the mask trick. Single-threaded. No allocation inside the ring itself — caller provides the backing storage. Self-contained header, suitable to drop into an embedded or kernel-adjacent project.

// ring.h #ifndef RING_H #define RING_H #include <stddef.h> #include <stdint.h> #include <stdbool.h> typedef struct { uint8_t *buf; // backing storage, caller-owned size_t cap; // must be a power of 2 size_t mask; // cap - 1, cached for speed size_t head; // next write position, always < cap size_t tail; // next read position, always < cap size_t count; // filled slots, 0..cap } ring_t; typedef enum { RING_OK = 0, RING_EMPTY = -1, RING_FULL = -2, RING_BAD_CAP = -3, // cap was 0 or not power of 2 } ring_err_t; // Initialise. `storage` must point to at least `cap` bytes. // `cap` MUST be a power of 2 (1, 2, 4, 8, 16, ...). ring_err_t ring_init(ring_t *r, uint8_t *storage, size_t cap); // Queries. static inline bool ring_is_empty(const ring_t *r) { return r->count == 0; } static inline bool ring_is_full (const ring_t *r) { return r->count == r->cap; } static inline size_t ring_size (const ring_t *r) { return r->count; } static inline size_t ring_free (const ring_t *r) { return r->cap - r->count; } // Single-byte operations. ring_err_t ring_push(ring_t *r, uint8_t value); ring_err_t ring_pop (ring_t *r, uint8_t *out); ring_err_t ring_peek(const ring_t *r, uint8_t *out); // Bulk operations. Return number of bytes actually transferred. size_t ring_push_bulk(ring_t *r, const uint8_t *src, size_t n); size_t ring_pop_bulk (ring_t *r, uint8_t *dst, size_t n); #endif
// ring.c #include "ring.h" #include <string.h> static bool is_power_of_2(size_t x) { return x != 0 && (x & (x - 1)) == 0; } ring_err_t ring_init(ring_t *r, uint8_t *storage, size_t cap) { if (!is_power_of_2(cap)) return RING_BAD_CAP; r->buf = storage; r->cap = cap; r->mask = cap - 1; r->head = 0; r->tail = 0; r->count = 0; return RING_OK; } ring_err_t ring_push(ring_t *r, uint8_t value) { if (r->count == r->cap) return RING_FULL; r->buf[r->head] = value; r->head = (r->head + 1) & r->mask; r->count++; return RING_OK; } ring_err_t ring_pop(ring_t *r, uint8_t *out) { if (r->count == 0) return RING_EMPTY; *out = r->buf[r->tail]; r->tail = (r->tail + 1) & r->mask; r->count--; return RING_OK; } ring_err_t ring_peek(const ring_t *r, uint8_t *out) { if (r->count == 0) return RING_EMPTY; *out = r->buf[r->tail]; return RING_OK; } size_t ring_push_bulk(ring_t *r, const uint8_t *src, size_t n) { size_t free = r->cap - r->count; if (n > free) n = free; if (n == 0) return 0; size_t first = r->cap - r->head; // bytes until array end if (first > n) first = n; memcpy(r->buf + r->head, src, first); if (n - first > 0) { memcpy(r->buf, src + first, n - first); } r->head = (r->head + n) & r->mask; r->count += n; return n; } size_t ring_pop_bulk(ring_t *r, uint8_t *dst, size_t n) { if (n > r->count) n = r->count; if (n == 0) return 0; size_t first = r->cap - r->tail; if (first > n) first = n; memcpy(dst, r->buf + r->tail, first); if (n - first > 0) { memcpy(dst + first, r->buf, n - first); } r->tail = (r->tail + n) & r->mask; r->count -= n; return n; }

Using it

// Caller owns the backing storage. Nothing is malloc'd inside the ring. static uint8_t storage[256]; ring_t ring; int main(void) { if (ring_init(&ring, storage, sizeof storage) != RING_OK) { return 1; } // push three bytes ring_push(&ring, 'A'); ring_push(&ring, 'B'); ring_push(&ring, 'C'); // pop one uint8_t x; ring_pop(&ring, &x); // x == 'A' // bulk push const uint8_t chunk[] = { 'D', 'E', 'F', 'G' }; ring_push_bulk(&ring, chunk, sizeof chunk); return 0; }

Design notes — why the code looks the way it does

Caller-owned storage. ring_init takes a pointer to pre-allocated memory rather than calling malloc. This is the difference between a library you can use in an ISR / kernel / embedded context and one you can't. If you want heap allocation, do it in a thin wrapper; the core stays allocator-agnostic.

Power-of-2 enforced at init, not per-operation. The is_power_of_2 check runs once in ring_init, so every subsequent push/pop can trust mask = cap - 1 without re-validating. The trick (x & (x - 1)) == 0 detects powers of two: a power of two has exactly one bit set, and subtracting 1 flips that bit to 0 and every bit below it to 1 — ANDing them yields 0. For any non-power-of-two, at least one bit overlaps.

Cached mask. We store mask = cap - 1 once and use it in every push/pop, rather than computing it each time. One memory load instead of a subtract-immediate — probably measurable on a tight inner loop on an MCU.

Return codes, not asserts. RING_FULL and RING_EMPTY are normal, expected conditions for a bounded queue. The caller must handle them; an assert would be inappropriate. Returning an enum keeps the interface explicit about what can go wrong.

Bounded bulk transfer. ring_push_bulk returns the number of bytes actually pushed rather than failing if the input doesn't fit. This matches the write(2) convention and lets callers do partial-write retry loops naturally.

No erase on pop. We leave the popped byte in the buffer. Zeroing it costs a write and buys nothing functional. If you're storing secrets (keys, session tokens), add an explicit ring_pop_zero variant rather than making everyone pay the cost.

Common traps

Trap What goes wrong Fix
Forgetting to update count size() returns garbage; full/empty checks lie Keep count update directly next to the head/tail increment — never split them across branches.
Using signed int for indices Can go negative, corrupt mask arithmetic, crash on array deref Always size_t. No exceptions.
Bulk copy without split Overruns the end of the array, corrupts memory after the buffer Two memcpys. Always. See ring_push_bulk.
Using modulo with non-power-of-2 cap and the mask trick head & (cap - 1) produces nonsense indices Either enforce power-of-2 at init, or use real % everywhere.
Accessing count from two threads without sync Races; head/tail get out of sync with count For concurrent use, switch to the extended-index variant (no count field), use atomics for head/tail.
Treating cap == 0 as "empty ring" Mask becomes -1, reads anywhere in address space ring_init rejects cap == 0 via the power-of-2 check.

Variants worth knowing about

Overwrite-on-full. For logging, tracing, and "latest N events" use cases, a full push should drop the oldest item rather than refuse. Modify ring_push: if full, advance tail as well as head, keep count at cap. The ring always holds the most recent cap items.

SPSC lock-free. One producer thread, one consumer, no locks. Uses the extended-index variant (head and tail grow monotonically, mask applied at deref). Producer writes head with release semantics; consumer reads head with acquire. This is what production systems like LMAX Disruptor and DPDK rings use. The devil is in the memory-ordering details — don't hand-roll it unless you're sure you need to.

Typed ring. The byte-oriented API above generalises to arbitrary types with a few macros or a typed wrapper. C11 _Generic or plain macro-per-type both work. Prefer a single typed file over a void-pointer "generic" implementation — it compiles better and the errors are legible.

Iovec / zero-copy variant. For scenarios where the consumer wants to process data in place (e.g. parsing bytes from a UART ring without copying them out), expose a "peek region" API that returns a {ptr, len} pair — possibly two of them if the data straddles the wrap. Consumer processes in place, then calls consume(n) to advance tail. Avoids a copy for bulk-read paths.
