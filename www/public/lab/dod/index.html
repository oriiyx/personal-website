data-oriented design: memory & caches

why memory layout determines performance · AoS vs SoA · cache lines · hot/cold splitting

The performance cliff you don't see in a profiler

You've written a game loop that updates 100,000 entities. You profile it. The hotspot is your update function — but the CPU isn't doing complex math. It's waiting for memory. The instructions are fast; the data they need isn't here yet.

This is the fundamental problem Data-Oriented Design addresses. Modern CPUs can execute billions of operations per second, but fetching a single cache line from main memory costs ~100 nanoseconds — enough time for the CPU to have executed 200–400 instructions. Every time your code touches memory that isn't already in cache, the CPU stalls.

Object-Oriented Design organizes code around types and inheritance hierarchies. Data-Oriented Design organizes code around how data flows through memory — because that's what actually determines performance at scale.

The cost of a cache miss — real numbers

On a typical modern x86 CPU (Zen 4, Raptor Lake era):

Memory hierarchy latency
L1 cache hit~1 ns (4 cycles)
L2 cache hit~4 ns (12 cycles)
L3 cache hit~12 ns (40 cycles)
Main memory (DRAM)~100 ns (300+ cycles)
~100× slower than L1
These are approximate for modern desktop CPUs. Server CPUs with more cores and larger caches have even higher DRAM latency (120–150 ns). The key insight: each level is roughly 3–10× slower than the previous.

The CPU doesn't fetch individual bytes — it fetches in cache lines, typically 64 bytes at a time. If you need 4 bytes from memory, you pay for all 64. If the other 60 bytes are data you'll never use, you've wasted 93% of the memory bandwidth.

This is the entire argument for data-oriented design: arrange your data so that when the CPU fetches a cache line, every byte in it is something you actually need for the operation you're performing right now.

Why OOP makes this worse

In a classic OOP game engine, each entity is a heap-allocated object. An Entity* points somewhere in the heap. Iterating over entities means chasing pointers — each one potentially a cache miss. Even if you allocate objects sequentially, their internal layout packs all fields together: position next to name next to texture handle next to AI state. Your physics update only needs position and velocity, but it's paying to load everything else into cache.

DOD asks: what if we stop thinking about "objects" and start thinking about the data the CPU actually needs for each operation?

How CPU caches actually work

The cache sits between the CPU and main memory. When the CPU issues a load instruction, the cache controller checks whether that address is already cached. If yes (hit): data arrives in 1–12 cycles. If no (miss): the controller fetches the entire cache line (64 bytes) from the next level down, stalling the CPU until it arrives.

Caches are organized as a hierarchy. Each core has its own private L1 (split into instruction and data caches, typically 32–48 KB each) and L2 (256 KB–1 MB). L3 is shared across all cores (16–64 MB on modern desktop CPUs). When data is evicted from L1 it may still be in L2 or L3 — misses at each level cost progressively more.

The cache line — the unit of transfer

The CPU never fetches a single float. It fetches the 64-byte aligned block that contains it. This is the cache line — the fundamental unit of data movement between memory and cache.

64 bytes = 16 floats, or 8 doubles, or 16 ints, or 2 typical Vec3+padding structs. Everything you store in here that you don't need is wasted bandwidth.

Two critical consequences: (1) if you access byte 0, bytes 1–63 are "free" — they're already in cache. Sequential access is fast because each cache line serves multiple consecutive reads. (2) If you access byte 0, then byte 4096, then byte 128, then byte 8000 — each access is a separate cache miss. Random access is slow.

The hardware prefetcher — your silent ally

Modern CPUs have hardware prefetchers that detect sequential access patterns and start fetching the next cache lines before you ask for them. If you iterate through an array linearly, the prefetcher will be 2–4 cache lines ahead of you — effectively hiding the latency entirely.

But it only works for predictable patterns. Chasing pointers to random heap locations? The prefetcher can't predict that. It gives up, and you pay full DRAM latency on every access. This is why a linked list of 100,000 nodes is dramatically slower than an array of 100,000 elements, even though the "algorithmic complexity" is the same O(n).

The prefetcher is the reason DOD works. If you lay out your data so that each operation is a linear sweep through a contiguous array, the prefetcher makes cache misses almost disappear. This is the single biggest performance win in DOD — not clever algorithms, just feeding the prefetcher a pattern it can predict.

Measuring it: bandwidth vs. latency

DRAM on a modern system can deliver 40–60 GB/s of bandwidth. But that bandwidth is only available for sequential, prefetchable access. Random pointer chasing gets you closer to 1–3 GB/s effective throughput — the bandwidth is there, but you can't use it because each load depends on the result of the previous one (pointer → data → next pointer → …).

This 15-30× gap is the concrete, measurable reason DOD matters. It's not theoretical — it's the difference between your game loop taking 2ms and 30ms.

It gets worse with heap allocation

The diagram above assumes contiguous allocation (Entity entities[N]). If entities are heap-allocated individually (new Entity()), they're scattered across the heap. Now each entity access is potentially a full cache miss — you can't even benefit from the 33% of useful data being adjacent. The prefetcher sees random addresses and gives up.

This is the typical OOP game engine pattern: a vector<Entity*> of pointers. Iterating means: load pointer → follow pointer (miss) → load entity data (another miss) → next pointer. Two misses per entity instead of ~one miss per cache line.

Struct-of-Arrays: the core transformation

Instead of one array where each element contains all fields, we create one array per field. Same data, different layout. The struct describes columns, not rows.

Array of Structs (AoS)
struct Entity { float pos_x, pos_y, pos_z; float vel_x, vel_y, vel_z; int32_t health, max_health; char name[32]; uint32_t tex_id, mesh_id; }; Entity entities[N];
Struct of Arrays (SoA)
struct Entities { float *pos_x, *pos_y, *pos_z; float *vel_x, *vel_y, *vel_z; int32_t *health, *max_health; char (*name)[32]; uint32_t *tex_id, *mesh_id; }; // each pointer → array of N

The loop itself changes shape

AoS loop
for (int i = 0; i < n; i++) { entities[i].pos_x += entities[i].vel_x * dt; entities[i].pos_y += entities[i].vel_y * dt; entities[i].pos_z += entities[i].vel_z * dt; }
SoA loop
for (int i = 0; i < n; i++) { e->pos_x[i] += e->vel_x[i] * dt; e->pos_y[i] += e->vel_y[i] * dt; e->pos_z[i] += e->vel_z[i] * dt; }

The code is nearly identical. The semantic meaning is the same. But the memory access pattern is fundamentally different: AoS strides by 72 bytes between each entity's pos_x, while SoA strides by 4 bytes. That stride difference is the entire game.

SIMD: the bonus you get for free

When data is laid out as contiguous arrays of the same type, the compiler can auto-vectorize the loop. SIMD instructions (SSE, AVX, NEON) process 4–8 floats per instruction from consecutive memory. SoA feeds them perfectly — the data is already packed the way SIMD wants it.

With AoS, auto-vectorization requires the compiler to "gather" scattered fields from different struct offsets — which is either impossible or requires expensive gather instructions. Most compilers simply won't vectorize AoS loops.

SoA gives you two wins at once: better cache utilization (less data loaded) and SIMD vectorization (more work per instruction). Together, these commonly yield 5–10× speedups on tight update loops over the AoS equivalent.

When full SoA is too much

Pure SoA — one array per field — is maximally cache-efficient for single-field operations. But some operations need multiple fields together. A rendering pass needs pos + tex_id + mesh_id at the same time. With pure SoA, those live in 5 separate arrays, and the CPU is streaming from 5 memory locations simultaneously. That's 5 separate prefetch streams — doable, but there's a limit (modern CPUs track 8–16 streams).

Hot/cold splitting is the pragmatic middle ground: group fields that are accessed together into a "hot" struct, and put rarely-accessed fields in a "cold" struct. Two arrays instead of twelve.

Splitting by access pattern

Look at which systems touch which fields:

System Fields accessed Frequency
Physics pos, vel every frame
Rendering pos, tex_id, mesh_id every frame
Gameplay health, max_health, pos every frame
UI / Debug name, health on demand

The name[32] field is 32 bytes — nearly half the struct — and it's only used for UI labels and debug logging. It's the definition of cold data. Splitting it out immediately halves the hot struct size.

Hot/cold splitting isn't as optimal as full SoA for a single operation, but it's often the best practical tradeoff: simpler code, fewer arrays to manage, good cache behavior for all systems that touch the hot data, and excellent behavior for systems that can ignore the cold data entirely.

Choosing the split point

The split isn't about "important" vs "unimportant." It's about access frequency. Ask: for each field, which systems read it and how often? Fields touched every frame go in hot. Fields touched occasionally go in cold. If you have three distinct access frequency tiers, use three structs.

Rule of thumb: if a field is accessed by fewer than half of your per-frame systems, it's a candidate for cold storage. If a field is large (strings, arrays, blobs) and accessed infrequently, it should almost certainly be cold.

Game development patterns

The Entity Component System (ECS) architecture is DOD applied to game engines. Components are stored in contiguous arrays (SoA or archetype-based), and systems iterate over only the components they need. Engines like Unity DOTS, Bevy (Rust), and flecs (C) all use this model. The tab on ECS architecture would cover this in depth — here's the key insight:

An ECS world is essentially a database. Entities are row IDs. Components are columns. Systems are queries. "Give me all entities with Position and Velocity" returns two arrays you can iterate in lockstep. The data layout is SoA by construction — you never load Health data when running the physics query.

// Bevy (Rust) — the query IS the data layout guarantee fn physics_system( mut query: Query<(&mut Position, &Velocity)>, time: Res<Time>, ) { for (mut pos, vel) in &mut query { pos.0 += vel.0 * time.delta_secs(); } } // Under the hood: iterates contiguous Position[] and Velocity[] // arrays in lockstep. No Name, no TextureId — just what was queried.

Backend / server patterns

DOD isn't just for games. Any system that batch-processes large collections benefits from the same principles:

Domain AoS pattern DOD equivalent
Database Row store (PostgreSQL, MySQL) Column store (ClickHouse, DuckDB, Parquet)
API server ORM objects with all fields hydrated Select only needed columns; process in batches
Analytics Array of JSON objects Apache Arrow columnar format
Network Per-connection object with all state Hot/cold split: connection state vs. config/metadata

Column stores are SoA for databases. When you run SELECT avg(price) FROM orders, a column store reads only the price column — a contiguous array of numbers. A row store must scan every row, loading all columns (customer_id, date, status, address…) to extract one field. This is literally the AoS vs SoA cache utilization problem at the storage layer, and it's why analytical databases are 10–100× faster than row stores for aggregation queries.

When DOD doesn't help

DOD is about batch operations on homogeneous data. It's not universally better:

Scenario DOD helps? Why
Update all 100K entities per frame yes Classic batch iteration — cache + prefetch + SIMD
Access a single entity by ID no difference One lookup is one lookup regardless of layout
Small collections (<100 elements) no difference Everything fits in L1 either way
Graph traversal (sparse, pointer-heavy) limited Random access pattern defeats prefetcher; reordering nodes by traversal order helps somewhat
All fields needed for every operation no benefit If every byte is useful, AoS and SoA load the same amount

Migration: you don't have to go all-in

DOD is a spectrum, not a binary choice. Practical steps, roughly ordered by effort/impact:

Step Change Effort Impact
1 Replace vector<T*> with vector<T> trivial eliminates pointer-chasing; contiguous allocation
2 Move large/cold fields to a separate struct low hot struct fits in cache; cold data untouched in hot paths
3 Split hot struct into per-field arrays (SoA) for the hottest loop medium 100% utilization for that loop; enables SIMD
4 Adopt an ECS framework / columnar data model high systematic SoA for the entire codebase
Start with step 1. It's free, it's always correct, and it's often the single biggest win. A vector<Entity> with 33% cache utilization is still dramatically faster than a vector<Entity*> with 33% utilization + a pointer chase per element. Contiguous allocation is the prerequisite for everything else.

Measurement cheat sheet

You can verify all of this on your actual hardware with performance counters:

# Linux — count L1 data cache misses for your program perf stat -e L1-dcache-load-misses,L1-dcache-loads ./my_program # macOS — Instruments → Counters (or dtrace) # Quick sanity check: time your hot loop with both layouts # and compare. If AoS→SoA doesn't speed it up by at least 2×, # your bottleneck is elsewhere (ALU, branch misprediction, I/O).

Key metrics: L1-dcache-load-misses (how often you miss L1), LLC-load-misses (how often you miss the last-level cache and hit DRAM), and instructions per cycle (IPC — if it's below 1.0, you're memory-bound; if above 2.0, the CPU is well-fed).
