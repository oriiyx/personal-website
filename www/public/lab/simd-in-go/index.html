SIMD in Go from first principles

Go 1.26 · simd/archsimd · GOEXPERIMENT=simd · AMD64 intrinsics for the rest of us

Why does SIMD exist?

CPUs are fast, but they're designed around scalar execution: one instruction operates on one value. If you need to add 1024 floats together pairwise, the CPU issues 1024 ADD instructions, each one waiting for the previous to retire through the pipeline.

But most of those additions are independent. The result of adding a[0] + b[0] has nothing to do with a[1] + b[1]. They could happen simultaneously if the hardware supported it.

SIMD — Single Instruction, Multiple Data — is exactly this. Instead of one ADD that operates on one pair of values, a single SIMD instruction operates on 4, 8, 16, or even 64 pairs at once. Same instruction, same latency, multiple results. The tradeoff: you need your data laid out contiguously in memory and your operations to be uniform across all lanes.

Where does it matter?

SIMD shines on data-parallel workloads — operations where you apply the same transformation to every element of a large array. Image processing, audio DSP, physics simulations, matrix math, cryptography, text search (finding bytes in a buffer), and vector similarity search all fit this pattern.

It doesn't help with serial/branchy code: tree traversal, graph algorithms, or business logic with complex conditionals. If every element needs different treatment, you can't pack them into one instruction.

Why does Go need this now?

Before Go 1.26, your options for SIMD in Go were painful: write raw Go assembly by hand (using Go's idiosyncratic assembler syntax), or call C via cgo (paying ~100ns per call in overhead). Both approaches break the Go development model — no type safety, no compiler optimizations across the boundary, no debugger support.

Go 1.26 introduces simd/archsimd under GOEXPERIMENT=simd: a package of compiler intrinsics that look like normal Go method calls but compile directly to single SIMD instructions. The compiler sees through them — it can inline, register-allocate, and fuse loads with arithmetic, just like it does with regular Go code.

Experimental status. This is behind a build flag (GOEXPERIMENT=simd), AMD64 only, and not covered by the Go 1 compatibility promise. The API may change in Go 1.27. A portable high-level API is planned but not yet available — archsimd is the low-level foundation it will build on.

Vector registers

A normal CPU register (like RAX on x86-64) holds one 64-bit value. SIMD introduces wider registers that hold multiple values packed side by side. On AMD64, these come in three sizes:

Register Width ISA Extension Float32 lanes Int8 lanes
XMM0–XMM15 128 bits SSE2+ 4 16
YMM0–YMM15 256 bits AVX2 8 32
ZMM0–ZMM31 512 bits AVX-512 16 64

The key insight: a 256-bit YMM register can hold eight 32-bit floats. One VADDPS instruction adds all eight pairs simultaneously — not "eight times faster" (there's overhead), but close to it for large arrays.

The same operation, two ways

Let's add two slices of float32 element-wise: result[i] = a[i] + b[i]. This is the canonical SIMD example because every element is independent.

Scalar

func AddScalar(a, b, dst []float32) { for i := range a { dst[i] = a[i] + b[i] } }

SIMD (archsimd, AVX2)

func AddSIMD(a, b, dst []float32) { n := len(a) i := 0 for ; i+8 <= n; i += 8 { va := archsimd.LoadFloat32x8Slice(a[i:]) vb := archsimd.LoadFloat32x8Slice(b[i:]) va.Add(vb).StoreSlice(dst[i:]) } for ; i < n; i++ { dst[i] = a[i] + b[i] } }

The scalar version compiles to a loop that loads one float, adds it, stores the result, and increments. The SIMD version loads 8 floats at once into a YMM register, adds all 8 in parallel with a single VADDPS, and stores all 8 results back. The scalar tail loop handles any remaining elements that don't fill a full 8-wide vector.

What the compiler generates

Each archsimd method call compiles to exactly one machine instruction. The SIMD loop body becomes roughly:

VMOVUPS (AX), Y0 // LoadFloat32x8Slice(a[i:]) VMOVUPS (BX), Y1 // LoadFloat32x8Slice(b[i:]) VADDPS Y1, Y0, Y0 // va.Add(vb) VMOVUPS Y0, (CX) // .StoreSlice(dst[i:]) ADDQ $32, AX // i += 8 (8 × 4 bytes) ADDQ $32, BX ADDQ $32, CX

That's 3 data instructions + 3 pointer bumps to process 8 floats. The scalar loop needs 8× the data instructions for the same 8 floats. The compiler can also fuse the load and add into a memory-operand form (VADDPS (BX), Y0, Y0), eliminating one instruction entirely.

Performance comparison

From benchmarks on an AMD EPYC 9575F (Go 1.26, GOEXPERIMENT=simd), adding two float32 slices:

Size Scalar throughput SIMD throughput Speedup
1K elements ~14 GB/s ~366 GB/s ~26×
65K elements ~15 GB/s ~246 GB/s ~16×
1M elements ~12 GB/s ~127 GB/s ~10×

The speedup decreases at larger sizes because the bottleneck shifts from compute to memory bandwidth. At 1K elements, everything fits in L1 cache and the ALU is the constraint. At 1M, the data streams from L3 or main memory, and the SIMD ALU spends most of its time waiting for loads.

This is the fundamental SIMD insight: SIMD helps when you're compute-bound (the CPU can do more work per cycle than memory can deliver). Once you're memory-bound, wider vectors don't help — you need faster RAM or better cache utilization. This is why SIMD + data-oriented design (contiguous arrays, struct-of-arrays layout) work together.

Type naming convention

Every archsimd type encodes its shape in the name: {ElementType}x{LaneCount}. The total register width is element_bits × lane_count.

Type Element Lanes Register Total bits
Float32x4 float32 (32b) 4 XMM (128b) 128
Float32x8 float32 (32b) 8 YMM (256b) 256
Int64x4 int64 (64b) 4 YMM (256b) 256
Uint8x64 uint8 (8b) 64 ZMM (512b) 512
Float64x8 float64 (64b) 8 ZMM (512b) 512

These are Go structs, but they're special: the compiler maps them directly to hardware vector registers. You use them as values (not pointers) — the compiler keeps them in registers as long as possible.

Mask types

Comparisons like .Greater(y) don't return bool — they return a mask type (e.g. Mask32x8), which is a per-lane true/false. Masks are used for conditional operations without branching:

mask := va.Greater(vb) // which lanes of a > b? result := va.Merge(vb, mask) // pick from a where true, b where false

This is how SIMD handles conditionals: instead of branching (which would break the pipeline), you compute both paths and blend the results using a mask. It's more work per element but avoids branch mispredictions entirely.

CPU feature detection

Not all machines support AVX2 or AVX-512. The package provides runtime checks via the archsimd.X86 variable:

if archsimd.X86.HasAVX2() { // safe to use Float32x8, Int64x4, etc. } if archsimd.X86.HasAVX512F() { // safe to use Float32x16, Int8x64, etc. }

The compiler treats these as pure functions — they're evaluated once at startup and the check itself is essentially free. Guard your SIMD code paths with these checks and provide a scalar fallback.

Worked example: summing an int64 slice

A realistic example from the archsimd community: summing all elements of an []int64 using 256-bit (AVX2) vectors. This demonstrates the full pattern — vectorized main loop, horizontal reduction, scalar tail.

The scalar baseline

func SumScalar(input []int64) int64 { var sum int64 for _, v := range input { sum += v } return sum }

One accumulator, one addition per element, fully serial. The compiler may unroll this but cannot vectorize it automatically.

The SIMD version

func SumSIMD(input []int64) int64 { i := 0 n := len(input) var sum int64 if n >= 8 { // Load first 4 elements into accumulator y0 := archsimd.LoadInt64x4Slice(input[0:4]) // Main vectorized loop: add 4 elements per iteration loopEnd := n - n%4 for i = 4; i < loopEnd; i += 4 { y1 := archsimd.LoadInt64x4Slice(input[i : i+4]) y0 = y0.Add(y1) } // Horizontal reduction: 4 lanes → 1 scalar x0 := y0.GetLo() // lower 2 lanes (128-bit) x1 := y0.GetHi() // upper 2 lanes (128-bit) x0 = x0.Add(x1) // [lane0+lane2, lane1+lane3] sum = x0.GetElem(0) + x0.GetElem(1) } // Scalar tail for remaining elements for ; i < n; i++ { sum += input[i] } return sum }

The structural pattern

Every SIMD function follows this skeleton:

func VectorOp(data []T) R { i := 0 n := len(data) var result R // 1. Check: do we have enough data for at least one vector? if n >= LANE_COUNT { // 2. Initialize accumulator from first vector load acc := archsimd.Load...Slice(data[0:LANE_COUNT]) // 3. Main loop: process LANE_COUNT elements per iteration loopEnd := n - n%LANE_COUNT for i = LANE_COUNT; i < loopEnd; i += LANE_COUNT { v := archsimd.Load...Slice(data[i : i+LANE_COUNT]) acc = acc.Op(v) } // 4. Horizontal reduction (if accumulating to scalar) result = reduce(acc) } // 5. Scalar tail for remainder for ; i < n; i++ { result = scalarOp(result, data[i]) } return result }

Build setup

GOEXPERIMENT=simd go build ./... GOEXPERIMENT=simd go test -bench . ./...

The build tag //go:build goexperiment.simd && amd64 is required on files using archsimd. Without GOEXPERIMENT=simd, the package doesn't exist — import fails at compile time, not runtime. You'll need Go 1.26+.

Slice loads and bounds checking

LoadFloat32x8Slice(s) will bounds-check that len(s) >= 8. In a tight loop, subslicing like a[i:i+8] generates a bounds check on every iteration. Two ways to eliminate this overhead:

// Option A: subslice once, loop on the subslice chunk := a[i:loopEnd] for j := 0; j+8 <= len(chunk); j += 8 { v := archsimd.LoadFloat32x8Slice(chunk[j:]) // ... } // Option B: use unsafe.Pointer + LoadFloat32x8 (no bounds check) ptr := unsafe.Pointer(&a[0]) for i := 0; i < loopEnd; i += 8 { v := archsimd.LoadFloat32x8((*[8]float32)(unsafe.Add(ptr, i*4))) // ... }

Option A is idiomatic Go. Option B avoids bounds checks entirely but drops safety — the same tradeoff as always with unsafe.

ClearAVXUpperBits

On older hardware, mixing 256-bit AVX instructions with legacy 128-bit SSE can cause a performance penalty. The CPU must save/restore the upper 128 bits of YMM registers when transitioning between modes. archsimd.ClearAVXUpperBits() issues a VZEROUPPER instruction to avoid this stall.

In practice: call it after a block of AVX2 code before returning to non-SIMD code, especially if you're in a library that might be called from SSE-using code. Modern CPUs (Zen 3+, most Intel since Skylake) handle this automatically, but it's cheap insurance.

When SIMD doesn't help

Situation Why SIMD struggles
Small arrays (<32 elements) Setup/tail overhead dominates. Scalar loop is fast enough.
Memory-bound workloads Already waiting on RAM. Wider ALU doesn't help when the bus is saturated.
Branchy per-element logic Masks can simulate conditionals, but if every lane takes a different path, you're computing all paths and discarding most work.
Non-contiguous / scattered data Gather/scatter instructions exist but are much slower than contiguous loads. Restructure data first.
Cross-lane dependencies If element i's result depends on element i-1, you can't parallelize across lanes.

The two-level roadmap

Go's SIMD plan is layered. archsimd is the low-level foundation — architecture-specific, 1:1 with hardware instructions, for power users. On top of it, the Go team plans a portable high-level vector API (inspired by C++ Highway) that abstracts over architectures. The high-level API would let you write SIMD code once and have it work on AMD64, ARM64 (NEON/SVE), and future architectures.

For now, archsimd is AMD64-only. ARM64 support (NEON, SVE) is expected in future releases. If you need portability today, write your SIMD code behind a build-tagged file and provide a scalar fallback for other platforms.
