the game engine toolkit

SDL3 · OpenGL · stb · Lua/LuaJIT — the C ecosystem for building engines from scratch

What SDL is and isn't

SDL (Simple DirectMedia Layer) is a cross-platform C library that gives you a single, portable API for the things an OS does differently everywhere: creating windows, handling input, managing audio devices, accessing the GPU, reading gamepads, and dealing with file I/O. It's the platform abstraction layer your engine sits on top of.

SDL is not a game engine, not a renderer, and not a framework that structures your game for you. It doesn't have a scene graph, an entity system, or a physics pipeline. It opens the door to the OS so you can build those things yourself. Think of it as the portable main() — it replaces a hundred #ifdef _WIN32 blocks with one clean API.

Why SDL3 specifically

SDL3 is the first major version bump since 2013. The API was rethought, not just patched. The biggest changes: the audio subsystem was redesigned from scratch around audio streams instead of callbacks, the GPU API is entirely new (a portable abstraction over Vulkan/Metal/D3D12), properties replaced the old hint system, and the main loop can now use a callback model that works properly on mobile and web targets.

SDL3 was released as v3.2.0 in January 2025. The satellite libraries (SDL_image, SDL_ttf) are stable; SDL_mixer is the last one pending a stable release. Porting from SDL2 is relatively straightforward — the concepts are the same, the function signatures shifted.

The callback main loop. SDL3 introduces SDL_AppInit, SDL_AppIterate, SDL_AppEvent, and SDL_AppQuit as an alternative to the traditional while(running) loop. This model is essential for platforms like iOS, Android, and Emscripten where the OS owns the run loop and your code must yield between frames.

What SDL3 manages for your engine

Window + Context
Create windows, manage GL/Vulkan/Metal contexts, handle Hi-DPI, multi-monitor, fullscreen toggling.
Input
Keyboard, mouse, touch, gamepad, pen/tablet. Unified event queue. Rumble, LED, touchpad on modern controllers.
Audio
Audio streams with automatic resampling and format conversion. Logical devices that survive hardware changes (unplug headphones → auto-migrate).
GPU API
New in SDL3. Portable abstraction over Vulkan, Metal, D3D12. Shader cross-compilation. An alternative to using raw OpenGL or Vulkan directly.
Filesystem + I/O
Platform-correct paths for save data, preferences, app bundles. Async I/O streams. Native file dialogs.
Timers + Threads
High-resolution counters, mutexes, semaphores, condition variables, atomics, TLS. Enough to build a job system on top.

SDL is the bottom of your code. Everything above it is your engine; everything below it is the OS. The renderer (OpenGL) talks to SDL for context creation, stb loads assets from files SDL opens, and Lua scripts drive logic that ultimately produces SDL events and draw calls.

The SDL + OpenGL relationship

If you're using OpenGL for rendering (rather than SDL's own GPU API), SDL's role is to create the window, create the OpenGL context, and swap buffers. Everything between SDL_GL_CreateContext and SDL_GL_SwapWindow is pure OpenGL — SDL steps aside. You still need SDL for input and audio during that frame, but the rendering pipeline is OpenGL's territory.

// Minimal SDL3 + OpenGL setup (pseudocode) SDL_Init(SDL_INIT_VIDEO); SDL_GL_SetAttribute(SDL_GL_CONTEXT_MAJOR_VERSION, 3); SDL_GL_SetAttribute(SDL_GL_CONTEXT_MINOR_VERSION, 3); SDL_GL_SetAttribute(SDL_GL_CONTEXT_PROFILE_MASK, SDL_GL_CONTEXT_PROFILE_CORE); SDL_Window *win = SDL_CreateWindow("engine", 1280, 720, SDL_WINDOW_OPENGL); SDL_GLContext ctx = SDL_GL_CreateContext(win); // --- everything here is OpenGL --- // glClear, glDrawArrays, shader programs, etc. SDL_GL_SwapWindow(win); // present the frame

What OpenGL actually is

OpenGL is a specification, not a library. The Khronos Group defines what each function should do and what the output must be; GPU vendors (NVIDIA, AMD, Intel, Apple) write the actual implementation inside their drivers. Your code calls glDrawArrays(); the driver translates that into hardware-specific commands for the GPU sitting in the machine. You never link against "the OpenGL library" — you link against the driver's implementation of the spec.

This is why OpenGL behaves differently on different hardware, why driver bugs exist, and why you sometimes need a loader library (like GLAD or GLEW) to resolve the actual function pointers at runtime — the OS doesn't expose the modern functions by default.

Core profile vs. immediate mode

OpenGL has two personalities. The legacy/immediate mode (pre-3.0) let you throw vertices at the GPU one by one with glBegin/glEnd. It was simple but horrifyingly slow — each call crossed the CPU→GPU boundary. The core profile (3.3+) removed all of that and requires you to batch geometry into GPU-side buffers, write your own shaders, and manage the pipeline yourself.

Core profile 3.3 is the standard teaching target. All features from 3.3 forward are additive — later versions (up to 4.6) add things like compute shaders, tessellation, and bindless textures, but the foundational model doesn't change. If you learn 3.3, everything above it is incremental.

The "dead API" question. OpenGL is not getting new versions. Vulkan is its successor for new development. Apple deprecated OpenGL on macOS/iOS. But OpenGL remains the most pragmatic choice for learning GPU programming from scratch: it has vastly more educational material, a gentler learning curve, and it runs on essentially every desktop GPU and driver in existence. You can always move to Vulkan or SDL's GPU API later — the concepts (shaders, buffers, pipelines, draw calls) transfer directly.

The rendering pipeline

Every frame, your CPU-side code uploads geometry and uniforms, then tells the GPU to draw. The GPU processes vertices through a fixed sequence of stages. Two of these stages are programmable (you write the code — these are shaders); the rest are fixed-function (the GPU handles them with configurable parameters).

Simplified OpenGL pipeline
CPU
Vertex data + uniforms → glDrawArrays()
↓
Programmable
Vertex Shader — runs per-vertex. Transforms positions from model space → clip space (MVP matrix). Passes data to next stage.
↓
Fixed
Primitive Assembly + Rasterization — connects vertices into triangles, clips to screen, converts to fragments (candidate pixels).
↓
Programmable
Fragment Shader — runs per-fragment. Computes final color using lighting, textures, and material properties.
↓
Fixed
Depth test + Blending + Write — discards occluded fragments, blends transparent ones, writes to framebuffer.
The vertex shader and fragment shader are the two stages you write in GLSL (OpenGL Shading Language). Everything else is configured through state-setting functions like glEnable(GL_DEPTH_TEST), glBlendFunc(), etc.

The core objects

Modern OpenGL is built around a handful of named objects that you create, configure, bind, and draw with. Understanding what each one holds is the entire mental model.

Object What it holds Purpose
VBO Raw vertex data on the GPU Positions, normals, UVs, colors — stored as a flat byte buffer. You define the layout; the GPU doesn't know what's in it until you tell it.
EBO Index data An array of integer indices into the VBO. Lets you reuse vertices (e.g., a cube has 8 unique positions but 36 indices for 12 triangles).
VAO Vertex attribute configuration Records how to interpret the VBO: which bytes are positions, which are normals, their types, strides, offsets. Bind it once, draw many times.
Shader Program Compiled GPU code A linked pair of vertex + fragment shaders (written in GLSL). Uploaded to the GPU, selected before drawing.
Texture Image data on the GPU A 2D (or 3D, cubemap, array) image that shaders sample from. This is where stb_image's output ends up.
FBO Render target A framebuffer you draw into instead of the screen. Used for post-processing, shadow maps, deferred rendering.

The state machine model

OpenGL is a global state machine. When you call glBindTexture(GL_TEXTURE_2D, myTex), you're not passing a texture to a draw call — you're setting the "current 2D texture" slot in a global context. The next draw call will use whatever's currently bound. This is the single biggest source of bugs for newcomers: forgetting to unbind something, or binding in the wrong order. The VAO exists partly to alleviate this — it captures vertex-attribute state so you don't have to re-specify it every frame.

The philosophy

stb is a collection of single-file, public-domain C libraries by Sean Barrett (nothings.org). The core idea: one .h file per library, no build system, no dependencies, no configuration — just drop it into your project and go. Each library solves one specific problem (loading images, rasterizing fonts, decoding audio) with a minimal, C-native API designed for game developers who want to avoid heavyweight dependencies.

The "stb style" has become its own design pattern in the C ecosystem. The libraries are intentionally limited in scope — stb_image doesn't try to support every obscure image format, and stb_truetype doesn't try to be a full text layout engine. They cover the 90% case cleanly and leave edge cases to specialized libraries. For an engine built from scratch, this is exactly the right tradeoff: you get working asset loading in minutes, not days.

The implementation macro pattern

Every stb library uses a two-mode include. By default, including the header gives you only declarations (function prototypes, types). To generate the actual implementation (the compiled code), you define a macro before the include in exactly one source file:

// In every file that needs the API — just declarations #include "stb_image.h" // In exactly ONE .c file — generates the implementation #define STB_IMAGE_IMPLEMENTATION #include "stb_image.h"

If you define the implementation macro in more than one translation unit, you get duplicate symbol errors at link time. The common practice is to create a dedicated file (e.g., stb_impls.c) that defines all your stb implementations in one place — you never edit it, so recompilation is rare.

The libraries you'll actually use

stb_image_write.h Writes PNG, BMP, TGA, JPG from a pixel buffer Screenshots, debug texture export, baking generated textures to disk.
stb_truetype.h Rasterizes TrueType fonts into bitmaps Font atlas generation for your text rendering system. Outputs alpha bitmaps you upload as textures.

Customization hooks

Most stb libraries let you override memory allocation and assertions by defining macros before the implementation include. This is critical for engines that use custom allocators (arena allocators, pool allocators, tracked allocators for leak detection):

#define STBI_MALLOC(sz) my_alloc(sz) #define STBI_REALLOC(p, newsz) my_realloc(p, newsz) #define STBI_FREE(p) my_free(p) #define STBI_ASSERT(x) my_assert(x) #define STB_IMAGE_IMPLEMENTATION #include "stb_image.h"
stb_image + OpenGL in practice. The typical flow: call stbi_load() to get a pixel buffer + dimensions + channel count → call glGenTextures / glBindTexture / glTexImage2D to upload it → call stbi_image_free() to release the CPU-side copy. Five function calls to go from a PNG on disk to a texture on the GPU.

Why engines embed a scripting language

Your engine's core — the renderer, the physics, the memory allocator — is C or C++. It needs to be fast, it needs to be predictable, and it compiles slowly. Now imagine every time a designer wants to tweak an enemy's patrol behavior, they have to recompile the engine. Or every time you want to test a new dialogue tree, you wait 30 seconds for a build.

A scripting language decouples behavior from infrastructure. The C side provides the fast, stable subsystems. The script side describes what happens in the game — and it reloads in milliseconds, without recompiling anything. This is the same split that Unity uses (C++ engine / C# scripts) and that World of Warcraft uses (C++ engine / Lua UI and addon system).

Why Lua specifically

Lua was designed from the ground up to be embedded into C applications. That's not a secondary use case — it's the primary one. The entire interpreter is about 300KB and compiles as part of your project with no external dependencies. The C API is a clean stack-based interface: you push values onto a stack, call Lua functions, and pop results back. No heavy runtime, no GIL, no framework overhead.

Compared to alternatives: Python is enormous and has the GIL; JavaScript requires V8 or a similar heavyweight runtime; C# requires Mono or .NET. Lua's embedding story is unmatched in simplicity and size.

Property Lua 5.4 LuaJIT
Runtime size ~300KB ~500KB
Language version Lua 5.4 (latest) Lua 5.1 (frozen)
Execution Bytecode interpreter JIT compiler → native machine code
Performance Fast for an interpreter 10–50× faster; approaches C speed in hot paths
FFI Stack-based C API only Built-in FFI: call C functions directly from Lua, no bindings needed
Tradeoff Latest language features (integers, generational GC) Raw speed + FFI, but stuck on 5.1 syntax
The LuaJIT FFI is the killer feature for engines. Instead of writing tedious C-side binding functions for every engine API you want to expose to scripts, you declare C function signatures in Lua and call them directly. The JIT compiler inlines these calls — there's no marshalling overhead. This is why engines like Defold, CryEngine, and many custom engines choose LuaJIT over vanilla Lua.

The C API: how embedding works

All communication between your C engine and Lua scripts goes through a virtual stack. You push arguments onto the stack, call a function, and pop the results. It's low-level but explicit — there's no hidden magic, and every operation has a predictable cost.

// Create a Lua state, load standard libraries lua_State *L = luaL_newstate(); luaL_openlibs(L); // Register a C function that Lua scripts can call static int l_get_player_hp(lua_State *L) { lua_pushnumber(L, player.hp); // push result onto stack return 1; // number of return values } lua_register(L, "get_player_hp", l_get_player_hp); // Load and run a script file luaL_dofile(L, "scripts/enemy_ai.lua"); // Call a Lua function from C lua_getglobal(L, "on_enemy_damaged"); // push function lua_pushnumber(L, enemy_id); // push arg 1 lua_pushnumber(L, damage); // push arg 2 lua_pcall(L, 2, 0, 0); // call with 2 args, 0 returns

With LuaJIT's FFI, the Lua side becomes even simpler — scripts can call your engine's C functions directly without you writing any binding code at all:

-- LuaJIT FFI: declare and call C functions from Lua local ffi = require("ffi") ffi.cdef[[ float get_player_hp(void); void apply_damage(int entity_id, float amount); void spawn_particle(float x, float y, int type); ]] -- Call them as if they were Lua functions local hp = ffi.C.get_player_hp() ffi.C.apply_damage(42, 10.5) ffi.C.spawn_particle(player.x, player.y, PARTICLE_SPARK)

What to script vs. what to keep in C

Keep in C/C++ Script in Lua
Render loop, draw calls, shader management Entity behaviors, AI state machines, patrol routes
Physics step, collision detection, broadphase Collision responses (what happens when things collide)
Audio mixing, streaming, DSP Sound triggers, music transitions, ambient rules
Memory allocators, resource loading, file I/O Level scripts, cutscenes, dialogue trees
Core math (matrix ops, spatial hashing) Gameplay formulas, damage calculation, loot tables
Input polling, event dispatch Input mapping, control rebinding, input combos

The rule of thumb: anything that runs thousands of times per frame stays in C. Anything that runs once per frame or per event — and that a designer might want to change — goes in Lua. The boundary between them is your engine's public C API: the set of functions you expose for scripts to call.

Hot reloading

One of the biggest practical advantages of Lua in a game engine is hot reload. Because scripts are loaded and executed at runtime, you can re-run luaL_dofile() on a modified script without restarting the engine. The new behavior takes effect immediately — the designer changes a number, saves the file, and sees the result in the running game within a second. This feedback loop is what makes scripting worth the complexity of embedding.
